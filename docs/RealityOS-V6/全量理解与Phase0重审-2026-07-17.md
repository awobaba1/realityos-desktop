## 0. TL;DR（三句话）

1. **产品理解锁定**：RealityOS V6 = 装在电脑上的「个人数据资产引擎」（北极星、第一理由、~70% 新建），外挂一个「操作电脑」能力作为新增卖点 + 数据捕获场。形态 = 桌面客户端（mac/win 安装包），fork hermes 作身体，自建 PTG 大脑。
2. **5 个 agent 抓到 12 个会致推倒重来的硬伤**（见 §3）。其中最致命的 3 个：① 换大脑的唯一缝我之前写错位置（`plugins/memory/__init__.py` 错，实为 `agent/memory_provider.py`）；② `conversation_loop` 的 Nous 引用我之前查得 = 0（错，实为 77，全部门控不执行）；③ **C6 铁律悬空**——V6 的 12 表 schema 里**根本没有 `llm_call_logs` 表**，而铁律要求每次 LLM 调用必须全量入日志。
3. **但有一个意外的好消息**（第 6 个 agent 刚确认）：数据捕获层**不依赖**桌面端走哪条路径——`PTGProvider.sync_turn` + `post_tool_call` 两个挂钩**路径无关**，已覆盖「流经即捕获」全部需求。我之前悬而未决的「桌面是否触发 pre_gateway_dispatch」**不再是阻塞项**。

## 2. hermes 身体地图（核实，含纠正）

### 2.1 换大脑的唯一缝（**纠正我之前的错误**）

> ❌ 我之前在 ADR-V6-005 写「基类在 `plugins/memory/__init__.py`」——**错**。
> ✅ 实证（Agent 1）：基类 `MemoryProvider` 在 **`agent/memory_provider.py`**（316 行）。`plugins/memory/__init__.py` 只是插件加载/发现器；`hermes_cli/memory_providers.py` 是桌面 UI 的配置 schema（同名陷阱）。

**MemoryProvider ABC 的真实接口面**（Agent 1 核实）：
- **4 个抽象方法**（子类必须实现）：`name` / `is_available` / `initialize` / `get_tool_schemas`
- **15 个可选 hook**（非文档原先说的「8 能力接口」）：`system_prompt_block` / `prefetch` / `queue_prefetch` / `sync_turn` / `handle_tool_call` / `shutdown` / `on_turn_start` / `on_session_end` / `on_session_switch` / `on_pre_compress` / `on_delegation` / `get_config_schema` / `save_config` / `on_memory_write` / `backup_paths`

**PTGProvider 注册 = 三层，只改 2 文件**：
1. **discovery**：新建 `plugins/memory/ptg/__init__.py`，导出 `register(ctx)`
2. **activation**：`config.yaml` 设 `memory.provider: ptg`
3. **runtime 强制**：`agent/memory_manager.py` 的 `add_provider`（384-450 行）强制「最多 1 个外部 provider」——保证 PTGProvider 唯一 active

**`agent/conversation_loop.py`（5601 行脑干）零改动**——这是 fork-inherit 哲学的硬保证。

### 2.2 捕获三挂钩（核实 + **关键去险结论**）

| 挂钩 | 状态 | 触发点 | 路径无关？ | 对 PTG 飞轮 |
|---|---|---|---|---|
| `pre_gateway_dispatch` | ✅ 已存在 | `gateway/run.py:9226-9232`，runner 侧，在 auth 之前、对完整 `MessageEvent` | ❌ 仅 gateway 路径（6 通道消息） | 捕获通道原始 payload（raw_message/media_urls/source），**非必需** |
| `post_tool_call` | ✅ 已存在（黄金挂钩） | `model_tools.py:1200,1300` 工具分发器 | ✅ **路径无关**（所有工具调用都过这里） | **必需**：捕获 agent 执行（文件/终端/browser/computer-use/send_message 等） |
| `sync_turn` | ✅ MemoryProvider hook | memory_manager 每轮 fan-out | ✅ **路径无关** | **必需**：PTGProvider 作 active provider，每轮收到 user+assistant 全文 |
| `post_gateway_send` | ❌ **不存在**（hook 名已预留 `hermes_cli/plugins.py:173`，但全代码库无 `emit` 调用） | 最佳注入 `gateway/platforms/base.py:4151`（`_send_with_retry`，所有非旁路发送都过此）+ 旁路点 `base.py:4739` | ❌ 仅 gateway 路径 | 捕获出站投递元数据（message_id/送达成功/raw_response），**非必需**（sync_turn 已有 assistant 全文） |

> **🔑 关键去险结论（解除 H3 阻塞）**：
> 数据飞轮的核心捕获 = `PTGProvider.sync_turn`（对话）+ `post_tool_call`（执行），**两者都路径无关**。因此我之前悬而未决的「桌面端是否触发 pre_gateway_dispatch」**不再是阻塞项**——桌面端无论走 web_server/gateway 还是 run_conversation，PTGProvider 作 active memory provider 都在脑干里收到每轮全文。pre_gateway_dispatch 只是 6 通道的「原始 payload 加餐」，对 PTG 飞轮非必需。**捕获层架构因此不再卡在任何未知数上。**

### 2.3 电脑操作（继承 + **6 个 P0 数据边界口子必须堵**）

hermes `tools/computer_use/`（CuaDriverBackend 经 MCP stdio 连 cua-driver 二进制，14 动作）全量继承。但**默认配置有 20 个数据边界口子**（Agent 2 核实），其中 6 个 P0（违「流经即捕获但不偷窥屏 + 图像不沉淀」）：

1. `_SAFE_ACTIONS` 把 `capture`（截图）列为免审动作（`tool.py:80`）→ 用户不知情被截屏
2. 无 callback 时默认 **allow**（`tool.py:311-315`）→ 应默认 deny
3. 多模态信封把截图 PNG **base64 上送云端 LLM**（`tool.py:657-667`）→ 截图出境
4. aux-vision 路径把图送**外部 Gemini**（外部依赖）
5. `start_recording` 核弹级——录整屏 mp4（`cua_backend.py:2173`）
6. SKILL.md 教 agent「把截图写到磁盘留用」

**Phase 0 P0-2 必堵这 6 口**：capture 改审批 / 无回调 deny / 主模型本地化或显式云模型白名单 / aux-vision 关 / start_recording 不暴露 / SKILL 删截图沉淀。`tools/approval.py` + `computer_use/permissions.py` 已在底座，边界非空话。

### 2.4 Nous 耦合（**纠正我之前的错误**）

> ❌ 我之前在 ADR-V6-005 写「`grep -nci nous conversation_loop.py` = 0」——**错**。
> ✅ 实证（Agent 4）：在 `agent/conversation_loop.py` 上 grep = **77**。原因是**我之前 grep 的路径错了**（在根 `$RD/conversation_loop.py` 上查，真文件在 `agent/conversation_loop.py`）。
> 77 处全部是 `if agent.provider == "nous"` 门控分支 + 4 个 Nous 专属函数。V6 用智谱 GLM-5.2/DeepSeek（ADR-093），`provider != "nous"`，**这 77 处全部永不执行**。

**结论不变但事实纠正**：脑干对 Nous 是「软/死耦合」——代码在但不激活。**单一 kill switch**：`hermes_cli/nous_account.py:322` 的 `get_nous_portal_account_info` stub 返回 `logged_in=False` → 25+ 调用方全部 fail-closed。

### 2.5 6 通道适配器审计（第 6 个 agent 刚到的报告精华）

6 个中国 messenger 通道（微信/QQ/钉钉/飞书/企微×2/Telegram）逐行审计完成。对 V6 捕获面的关键结论：

- **通道层是 hook 盲区**：6 个适配器文件里 grep 任何 hook 关键词 = **0 匹配**。所有 hook 都在 runner/agent 层触发（`handle_message` 返回后），不在适配器内。
- **入站捕获无需改适配器**：每个适配器在构造 `MessageEvent` 时都填好了 `raw_message` / `source` / `media_urls` / `media_types` / `metadata`——`pre_gateway_dispatch` 直接读这些字段即可，**6 通道零改动**。
- **出站捕获唯一可用信号**：每个适配器 `send()` 返回的 `SendResult`（success/message_id/error/raw_response）。在 `base.py:_send_with_retry:4151` 加一个 `emit("post_gateway_send", ...)` 即可统一覆盖全部 6+ 通道；旁路点（`base.py:4739` 的 `/approve`/`/deny`）需再加一个 emit 才全覆盖。
- **语音入站转录现状**（影响 R8 原子）：只有 **QQ**（Tencent ASR）和 **企微 Smart**（自带 ASR）在入站时产出转录文本；微信/飞书/Telegram 只缓存音频字节（silk/ogg/amr）留待 agent 侧 STT。这决定了 V6 的 R8 捕获策略要分通道处理。

### H7 — `llm_call_logs` 不在 V6 schema（**C6 铁律悬空！**）🔴 最致命

### H9 — computer-use 默认泄漏截图到云 LLM（违数据边界）
- 见 §2.3，6 个 P0 口子
- **修**：Phase 0 P0-2 堵 6 口

### H10 — location_log 不迁违反 C2
- **产品决策**：V6 砍 GPS（桌面端无意义）
- **铁律 C2**：「不可丢失」——创始人 V5 历史 GPS 数据若不迁即丢失
- **张力**：产品决策 vs 铁律
- **修**：① ADR 显式记录「location_log 不迁入 V6 schema，但创始人 PG dump 永久备份」；② 不迁入主库 = 不丢（备份在）；③ user_id 保留映射（见 H15）

### H11 — PG → SQLite 未映射的 4 类构造
- **pgvector `Vector(512)`**：V6 用 sqlite-vec 扩展，或向量单独存 BLOB + 应用层余弦
- **`STDDEV()` SQL**：SQLite 无内置，改 Python 计算后回写
- **GIN 索引**：SQLite 不支持，改 FTS5 或应用层
- **PG advisory lock**：SQLite 无，改 `BEGIN IMMEDIATE` 或应用层互斥（V5 跨时间引擎用过）
- **修**：Phase 0 P0-3 加类型 spike 实测这 4 类（尤其 sqlite-vec 在目标 mac/win 上的可用性）

### H12 — V5 设计智慧 V6 文档未提（继承清单缺失）
V5 踩坑沉淀的 6 项关键设计，V6 文档完全没提，若不主动继承会重踩：
1. **专有名词 5 层防护**（ADR-048~052）：热词 + 纠错字典 + LLM 注入 + R0_Entity + 拼音归一化 → V6 `realityos_atomizer` 必须重做

3. **Prompt 版本树**：V5 v11 作 V6 v1 基线（永不覆盖）
4. **ASR 纠错自学习**（ADR-056）：编辑源文本→diff 学习→重新提取
5. **用户反馈 18 target_type**（ADR-089）
6. **渐进式置信度**（ADR-045）+ **advisory lock 防并发**
