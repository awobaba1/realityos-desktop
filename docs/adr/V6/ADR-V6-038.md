# ADR-V6-038：外部用户接待前置（桶 D 代码侧铺路）——诚实四分

- **状态**：Accepted（Phase 1b 收口后·桶 D 铺路，2026-07-20）
- **日期**：2026-07-20
- **Phase**：1b（对外发布可用的运营侧前置）
- **关联**：ADR-V6-022（母纲，桶 D 生产验证真空）、ADR-V6-027（quality_metrics 本地落点）、ADR-V6-037（发布管线 autopublish）、ADR-V6-025（net-policy 主权接线）
- **铁律**：C1（本决策一 ADR + 每实现项配套测试）、C7（崩溃不静默丢失→兜底落盘；分发物不可达不静默放行→门禁）、C3（生产现实即真理——无外部用户时不伪装运营指标）

## 背景

ADR-V6-037 把批次 1/2/3（ADR-027~036）随 v2026.7.21 推进分发，「100% 生产级对外发布可用」的代码侧收口完成。用户要求「都做」推进三方向：真实端到端冒烟（桶 D）、Phase 2、外部用户接待前置。

为不凭 V5 ADR-202 记忆断言 V6 现状，先开 Explore 子 agent 逐项核实 V6 桌面「外部用户接待 / 生产可观测性」的真实代码落地（114 tool-use，只读）。结论：**接待前置缺 5 件事**，但其中部分「缺失」在主权叙事下是**有意设计**，不可盲目补成假绿。

## 决策（诚实四分：做 / 主权判定不补 / 诚实延迟 / 已达标不动）

### 做（代码可自主、真对齐铁律、非假绿）—— D1~D4

#### D1：接待门禁脚本 `scripts/reception_preflight.py`（创始人分发物 Pre-flight）

创始人**邀请外部用户前**的一揽子 exit-code 门禁（对标 V5 ADR-202 Pre-flight，但聚焦 V6 分发物 + 本地就绪）：

- CHECK 1：最新 Release tag 存在 + 资产（dmg+exe）存在（`gh release view`）。
- CHECK 2：资产 size 非空非畸（对照历史量级 138MB/118MB，反 20B 假文件教训 [[v5-backup-broken-since-0709]]）。
- CHECK 3：`install.sh @ <tag-commit>` GitHub raw 可达（HTTP 200）——E12 首启链的源头可达性。
- CHECK 4：（可选 `--probe-download`）公开下载 HTTP 200（跟 302 到 release-assets CDN 终态）。
- CHECK 5：本地 `hermes` 运行时就绪（state.db 可读 + config 有效 + model 配了，调 `collect_runtime_readiness` 逻辑的本地子集）。
- 全绿 exit 0 + 绿字「✓ 可邀请外部用户」；任一红 exit 1 + 逐项红字。

**边界（诚实）**：本脚本是「创始人/运营者视角的分发物门禁」，**不**是「外部用户客户端就绪」（后者由 onboarding 的 `evaluateRuntimeReadiness` 已覆盖，C8）。V6 当前是纯本地桌面拓扑（每用户自己 hermes + 自己的 LLM key + 自己的 bot token，无中心服务端），故门禁不含「服务端健康」项。

#### D2：桶 D 真实端到端冒烟清单 SOP `docs/external-user-reception.md`

把「装 dmg → 首启拉 install.sh@commit → onboarding 配 provider → 建 atom → 出日报 → 周镜面」整条**真机操作链**写成可勾选验收清单（每步：操作 + 预期 + 失败排查）。这是桶 D 的**操作侧**产出——真机冒烟天然需用户在自己机器跑（CI runner 非 mac 桌面真机、无法验证「真实安装体验」），我提供清单 + 验收点，用户跑、我对照。**不**伪装「自动化冒烟 = 已验证」（E11 真机链不可全自动，诚实标注）。

#### D3：崩溃兜底落盘（electron main.ts，C7 真增强，纯本地不破坏主权）

`apps/desktop/electron/main.ts` 加 `process.on('uncaughtException')` + `process.on('unhandledRejection')`：

- 结构化落盘到 `DESKTOP_LOG_PATH`（timestamp + type + message + stack），复用既有 cascade 轮转 + discard ceiling。
- **纯本地，零外发**——符合「数据不出设备」主权叙事（ADR-V6-025 net-policy 接线已确立）。崩溃堆栈可能含文件路径/用户名等 PII，自动外发违主权。
- 用户体验：崩溃后下次启动提示「检测到上次崩溃记录，可用 `hermes debug share` 上报」——**opt-in、用户主动**，非自动外发。

**对齐 C7**：现状 electron 崩溃只在散点 handler 里 `console.error`/局部 `appendFile`，全局兜底缺失 = 崩溃可能静默丢失无追溯。本增强让「崩溃必落盘可追溯」，堵死崩溃形态的静默失败。

#### D4：D1/D7 本地留存查询 `scripts/retention_local.py`（创始人自看，诚实标注非运营库）

只读本地 sessions DB 的 D1/D7 回访查询（首次活跃日 = 安装日；D1 = 安装+1天有 session；D7 = 安装+7天有 session）。**诚实标注**：单用户本地视角的「我作为用户是否回访」，**非**跨用户运营留存（V6 无中心运营库，B5 quality_metrics 是 per-user 本地 SQLite）。桶 D（B3）的运营侧留存须外部用户上量后才有数据，本脚本只填「创始人本地自看」的空。

### 不做（主权判定——不盲补，避免破坏主权的假绿）

- **自动崩溃外发（Sentry/crashpad）**：崩溃堆栈含 PII（文件路径/用户名），自动外发违「数据不出设备」主权叙事（ADR-V6-025）。Explore 报 A1「缺失」，但结合主权这是**有意设计**。D3 用「本地兜底 + opt-in 主动上报」替代，**不接自动外发**。
- **跨用户运营库 ETL**：V6 纯本地拓扑，无中心库可汇聚；quality_metrics 已是 per-user 本地（ADR-027）。建跨用户 ETL = 空中楼阁（无数据源），是假绿。

### 诚实延迟（桶 D，需真实用户/真机，做了是假绿）

- **真机 dmg 全链自动化冒烟（E11）**：CI runner 非 mac 桌面真机，无法验证「真实安装体验」。D2 提供 SOP 清单让用户真机跑；自动化只到 `test-desktop.mjs` 的 validateBundle+openDmg（已达标，不重做）。
- **桶 D 运营 KR（§8 Day-7 留存>40% 等）**：母纲桶 D 明确需外部用户上量，无外部用户时无法验证。

### 已达标不动（避免重做假绿）

- **首启 install.sh@commit 拉取链（E12）**：完整落地（stamp→dmg→loadInstallStamp→bootstrap-runner），有回归测。不重做。
- **onboarding 流程（C8）**：UI + 状态机 + 首次提示全在。不重做。
- **本地用量自看（B4）**：`hermes insights` + `/api/analytics/usage` 已通。不重做。

## 后果

- V6 桌面「外部用户接待」从「散件可拼但无门禁/SOP」升级为：**创始人邀请前有硬门禁（D1）+ 真机冒烟有 SOP（D2）+ 崩溃不静默丢失（D3）+ 本地留存可自看（D4）**。
- 反假绿：不伪装「自动冒烟 = 已验证」「自动上报 = 已接入」；主权判定让「崩溃外发缺失」从「该补的假绿」纠正为「有意设计」。
- 桶 D 的运营侧（跨用户留存/KR）诚实延迟——代码侧铺到边界，剩余天然需真实用户。

### 实测结果（2026-07-20，commit 917be57c3）

- **CI 主 run 29710424175 = success**（真发布门禁）：headSha `917be57c3` 精确匹配本地 HEAD；step 级非 skip（ruff check .=success / Python tests e2e=success / JS&TS checks=success / Docker build smoke=success / OSV scan=success）；6 个 skipped job 是路径/条件门控（Supply-chain scan、ruff+ty diff 等），非假绿 skip。
- **js-autofix（辅助自动 lint-fix+PR，非门禁）red**：`Generate eslint --fix patch` = success（本 ADR 改动文件 prettier 合规，patch 不含 .md）；`Apply patch` job 的 `Create PR` step exit 4——根因 `secrets.AUTOFIX_BOT_PAT` 未配/失效（脚本 token 分离：push branch 用 GITHUB_TOKEN 成功 commit da4c31e，create PR 用 AUTOFIX_BOT_PAT 失败；repo 仅 5 dependabot PR、0 autofix PR 为证）。PAT 配置是用户侧 GitHub 操作（建 fine-grained PAT：contents:write+pull-requests:write + 加 repo secret `AUTOFIX_BOT_PAT`），非代码、非门禁——诚实延迟，记 [[v6-reception-preflight-adr038]]。
- 本 ADR 的 D1~D4 自身验证全绿（ruff 全净 / tsc 双绿 / eslint 0 / reception 16 测 / retention 11 测 / crash-guard 8 vitest）。
