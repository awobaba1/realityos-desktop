# ADR-V6-075 — C4 审计收口 + desktop-build publish 契约守卫

**状态**: Accepted
**日期**: 2026-07-21
**关联**: 第五轮 C4 维度专项 / [ADR-V6-037](ADR-V6-037.md)（autopublish「做了没发」）/ [ADR-V6-072](ADR-V6-072.md)（PyPI token 守卫）/ [C4](../../../CLAUDE.md)（每个缺陷成为用例）/ [ADR-403](../../adr/ADR-403.md)（密钥不入库）

## 背景（C4 维度：五轮审计的唯一盲区）

前四轮 + ADR-073/074 覆盖 C1/C2/C5/C6/C7/ADR-037/R6，但 **C4「每个缺陷成为用例」从未专项审计**。C4 是 P1 铁律：bug fix PR 无对应回归测试须拒收。本 ADR 收口这个维度。

### 审计方法

扫 fork main 上近 60 个 `fix(...)` 提交，逐个核验：每个修复有没有 co-commit 的回归测试，或被既有不变量测试覆盖。

### 审计结果（代码级 bug-fix 全覆盖）

绝大多数 `fix(...)` 提交**同提交带测试**（test_files≥1），例如：
- `42e62f66e` sanitize ASCII 兜底 → `test_unicode_ascii_codec.py`
- `3ec8bbffe` theory PC/FR DLQ → `test_adr_v6_050_theory.py`
- `01208dc5b` atomize 外层 DLQ → `test_ptg_provider.py`
- `f0f4ffa69` 精度门禁 → `test_insights_web_api.py` 等

少数 `test_files=0` 的 CLI 接线修复（`be1df0372` 补 "k"、`149617e3f` 补 "calibrate" 进 `_BUILTIN_SUBCOMMANDS`）**被既有双向不变量测试覆盖**：

`tests/hermes_cli/test_startup_plugin_gating.py`：
- `test_builtin_set_covers_every_registered_subcommand` —— 锁「所有 live 注册子命令必须在 `_BUILTIN_SUBCOMMANDS`」（正是这两 commit 修的缺口；提交前 CI 红即此测试）。
- `test_builtin_set_has_no_phantom_entries` —— 锁反向（声明但未注册）。

**双向不变量是 CLI 接线的安全网 —— 任何漏登记都会被 CI 即时抓到。C4 对这两项成立。**

### 唯一真缺口：`f9dad30ac`（desktop-build publish job 加 GH_REPO）

该 commit 修了 publish job 的 `not a git repository` 错误（publish job 无 checkout，`gh` 无法从 git remote 推断仓库 → 显式 `GH_REPO: ${{ github.repository }}`）。但这是 **CI 基建修复，无应用代码，也无任何回归测试**。

这正是 ADR-037「做了没发」的咽喉：publish job 静默失败 → 构建绿、CI 绿、但安装包**永远到不了 Release 下载页**。若将来有人重构 workflow 丢了 `GH_REPO` 或 upload 步骤，会无声回归最致命假绿类，且无测试拦。

## 决策：加 publish 契约静态守卫（C4 收口）

`tests/ci/test_adr_v6_075_publish_job_contract.py`（6 测试）锁 ADR-037 autopublish 契约全要素，任一被静默删除即红：

| 守卫 | 锁的契约 |
|---|---|
| `publish:` + `startsWith(github.ref,'refs/tags/v')` | publish job 存在且仅 tag 触发 |
| `contents: write` | release 创建/上传所需权限（丢了 → 403 静默失败） |
| `GH_REPO:` + `${{ github.repository }}` | **f9dad30ac 原始修复** —— 无 checkout 时 gh 的显式仓库指针 |
| `gh release upload` + `.dmg` + `.exe` | 真正的上传步骤（删了 = 资产只进 CI artifacts 不进 Release） |
| `gh release create` + `gh release view` | 幂等创建（重跑 attach 而非崩） |
| **所有 `${{ secrets.X }}` == GITHUB_TOKEN** | 防「修权限」塞长寿命 PAT（违 ADR-403，同 ADR-072 PyPI 守卫风险类） |

### 教训：守卫须精确锚定，勿裸子串黑名单

初版用 `forbidden not in normalized.upper()` 查 "PAT" → **假阳性**：「PAT」作为子串出现在 PATCH/PATH/COMPAT 等正常注释词里。这是 wrap-lucky-green 的反例（裸子串黑名单太天真）。改为正则抽取所有 `${{ secrets.X }}` 引用，断言每个 == GITHUB_TOKEN —— 精确锁真实风险（注入非默认 secret），不被正常词干扰。

## 合规性

- **C4**：代码级 bug-fix 全覆盖（co-commit 测试 或 双向不变量）；CI 基建缺口（publish job）由本守卫补齐。C4 维度闭环。
- **C1**：守卫本身是决策，记 ADR。
- **ADR-037**：publish 契约现在有测试护栏，「做了没发」回归会被 CI 抓。

## 不发版

纯 docs（本 ADR）+ guard（6 测试）无 distributable 变更，dmg/exe 字节相同 → 发版=dishonest churn。v2026.7.24 仍 Latest。
