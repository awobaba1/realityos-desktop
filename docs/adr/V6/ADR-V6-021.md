# ADR-V6-021：Phase 1b-6 — PTG 记忆浏览器（读取 API + 桌面 UI，de-black-box）

- **状态**：Accepted（Phase 1b 记忆可见性，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（广度优先，闭合「记下来 → 看得到」）
- **关联**：ADR-V6-011（Atomizer 心脏 + 图物化）、ADR-V6-012（结构化数据 recall 面）、ADR-V6-016（R8/R9/R12 + Stage2）、ADR-V6-020（insight 读取 API，本 ADR 复用其 store/founder helper）、ADR-V6-008（PTGStore 共享单例）
- **铁律**：C1（本 ADR）、C2（纯读，走 `deleted_at IS NULL`，无任何 DELETE）、C5/C6（不涉及——本 ADR 是只读，不抽原子、不跑 LLM）、C7（读取 API 全路径 fail-open，no_data/error 都是 200 + 结构化载荷，绝不向 UI 抛）

---

## 背景

到 ADR-V6-020 为止，桌面已经能**捕获**原子（ADR-V6-011 心脏：飞书/文字 → Atomizer → 8 原子落 4 事件表 + 图物化）并**生成报告**（ADR-V6-017/018/019/020 周镜面/日报/调度/读取 UI）。但桌面端**没有任何入口让用户直接看到捕获到的原子本身**——探查确认：

- `apps/desktop` 的 `starmap`（星图）调的是 `/api/learning/graph`，是 Hermes **技能学习图**（触及技能目录 + usage ledger + memory files），**不是** RealityOS PTG 原子图。
- 后端对 `/api/atoms` / `/api/people` / `/api/entities` / `/api/timeline` / `/api/graph`（PTG 口径）**零命中**——结构化数据是「只写」的（ADR-V6-012 注释原话：「the structured data was WRITE-ONLY」）。

结果：产品对用户是**黑盒**——它在记，但用户看不到它记了什么，无法验证「记住人/事/状态」（Phase 1b 交付承诺）是否兑现。移动端有 Memos/Tasks/People/Timeline 屏幕正是为此；桌面没有。本 ADR 闭合这条「记下来 → 看得到」的可见性缺口。

### 为什么不复用移动端 / 不另写图渲染

1. **移动端（`danao13/web`）**：有一整套 Memos/Tasks/People/Insights 屏，但那是 Capacitor 移动壳 + React Native 风格组件 + 后端把原子解析成结构化字段渲染。移植量大、与桌面取向相悖。
2. **星图（starmap）**：是 d3-force 径向时间轴渲染器，喂的是技能学习图数据结构（`StarmapGraph`/`SimNode`/`SimLink`），与 PTG 原子模型（R3_Person/R2_Task/...）不兼容；复用要重写数据契约。

决策：**新写一个轻量只读浏览器页**，复用 PTGStore **已有的丰富读取面**（`recent_atoms`/`list_top_entities`/`relations_for_user`/`memo_count`/`user_created_at`），零新表、零新 store 方法、零 LLM。走桌面既有「贡献式全页 + 侧栏入口」机制（ADR-V6-020 验证过），零侵入核心路由。

---

## 决策

### 1. 后端：单个只读端点 `GET /api/memory/browse?limit=N`（`hermes_cli/web_server.py`）

一个端点一次返回浏览器所需的全部数据（原子时间线 + 实体目录 + 关系边 + 计数），避免前端三 Tab 各打一次。流程（`_get_memory_browse` → `_read_memory_browse`）：

1. 开共享 store（refcounted，ADR-V6-008）——**复用** ADR-V6-020 的 `_open_ptg_store_for_insights`（名字历史，实为通用「开共享 PTGStore」helper，browse 也用它）；
2. 解析 founder（**复用** `_resolve_insight_founder`）；无 founder → 温暖 `no_data`（首启竞态，永不报错）；
3. 一次性读 `recent_atoms` + `list_top_entities` + `relations_for_user` + `memo_count` + `user_created_at`；
4. **fail-open（C7）**：任何异常 → `{"status": "error", ...}`，HTTP 200，绝不 5xx、绝不向 UI 抛。

**`limit`** 默认 200、上限 500（`limit` 超大或非数字 → 回落默认，不 4xx）。原子按 `_ts` 倒序（`recent_atoms` 已排）。

### 2. 响应形状：atom 卡片模型（type-dispatch + fields 兜底）

```json
{ "status" ∈ {ok, no_data, error},
  "atoms":   [{ "type" ∈ {R3_Person,R2_Task,R7_Expression,R8_Cognition,
                          R12_Outcome,R1_SelfState,R9_Emotion,R0_Entity},
                "confidence", "timestamp",
                "fields": { ...type-specific salient columns... } }],
  "entities":   [{ entity_name, entity_type, mention_count, aliases }],
  "relations":  [{ relation_type, value, confidence, evidence_count,
                   subject_name, subject_type, object_name, object_type }],
  "memo_count", "created_at", "message" }
```

`type`/`confidence`/`timestamp` 提为顶层（渲染器按 type 派发），其余列下放 `fields`——这样**前端不硬编码列模型**，新增原子字段不用改序列化层。`_serialize_atom` 做这个拆分。

### 3. 桌面 UI：贡献式全页 + 三 Tab（`apps/desktop/src/app/memory/MemoryPage.tsx`）

- **`/memory` 全页**（ROUTES_AREA）+ 侧栏入口（SIDEBAR_NAV_AREA，codicon `book`，label「记忆」，`order: 30`——排在「洞察」order 35 之上，先看原始记忆再看报告）。
- **三 Tab**，共享一次 fetch（不切 Tab 重打）：
  - **时间线**：原子按时间倒序，每条一个卡片 = 类型徽（按类别配色：人物/任务/意图/结果→default；状态/情绪→warn；认知/地点机构→muted）+ 主字段（按 type 抽取：R3→person_name、R2→task_description、R1→state·direction·intensity…）+ 次要明细 + 置信% + 北京时间。
  - **人物与实体**：实体目录（`list_top_entities`，已过滤 self 节点），名字 + 类型徽 + 别名 + 提及次数。
  - **关系**：边列表（`relations_for_user`，已 JOIN 名字），`主体 —关系类型[:值]→ 客体` + 置信/证据数。
- **冷启动**：`status==no_data`（无 founder）→ 温暖空状态；`ok` 但某 Tab 空 → 该 Tab 自己的空状态文案。`status==error` → 错误空状态。刷新按钮重打一次（纯读，无缓存可失效）。

### 4. i18n 跨 4 locale 平价

`Translations` 类型强约束：加 `memory` 段必须同时改 `types.ts` + `en/zh/zh-hant/ja`（TS 编译即门禁）。侧栏 label 是贡献 `data.label`（静态串），取中文「记忆」。

## 关键实现点

### 复用而非新造（store 读取面早已就位）

ADR-V6-012 当初就把结构化 recall 面（`recent_atoms` 等）建好了——那是评测 harness（要测**过 C5 闸后**真落了什么，非原始 LLM 输出）+ 预取渲染器（把图折进上下文）的产物。本 ADR 只是**把这些已有的读取面第一次暴露给用户 UI**。零新 store 方法、零新表、零迁移。SCHEMA_VERSION 不变。

### entity_type vs entity_category（两表两枚举，踩坑点）

`entity_events.entity_category`（R0 原子）CHECK ∈ {place, organization, term}；而图节点表 `entities.entity_type` CHECK ∈ {person, task, topic, context}——**机构在图里是 `context` 节点**，不是 `organization`。测试先撞了这堵墙（`organization` 被 entities 表 CHECK 拒），改成 `context`。两类语义不同：原子层记「这是个机构」，图层把机构归为「context 节点」。

### 纯读 MVP，纠正延后

`recent_atoms` 重构的原子 dict **不带行级 provenance（table, row_id）**，所以「从浏览器删/纠正某条原子」本批不做——`soft_delete(table, row_id)`（C2）store 侧已就位，但要先给原子 dict 打上 `(table, id)` 来源才能从 UI 发起单条软删。纠正流（移动端有）是独立的后续工作，不在本批。本批就是**只读 de-black-box**。

## 不在本批次做的事（诚实标注，后续）

- **单条纠正/软删**：需先给 `recent_atoms` 输出打行级 provenance（`table`+`id`），再做 UI 纠正流。store 侧 `soft_delete` 已就位。
- **搜索**：store 已有 `search_entities` / `search_memos_fts`，本批未接 UI（浏览器先做时间线/目录/关系三视图；搜索是自然后续）。
- **日历/多周期历史浏览**：当前只看最近 N 条原子；多周期翻页 / 日历热力图后续。
- **真实 DeepSeek 端到端冒烟**：本批纯读不涉 LLM，无需；但用户活 session 的真实数据呈现待发布时在真实桌面验。

## 决策小结

| 维度 | 选择 | 理由 |
|------|------|------|
| 端点形状 | 单端点 `/api/memory/browse` 一次返全 | 三 Tab 共享，避免切 Tab 重打 |
| 数据来源 | 复用 PTGStore 已有读取面 | ADR-V6-012 早建好；零新表/零新方法/零迁移 |
| 原子模型 | type-dispatch + `fields` 兜底 | 前端不硬编码列模型；新字段免改序列化 |
| UI 注册 | 贡献式 ROUTES_AREA + SIDEBAR_NAV_AREA | 走既有机制（ADR-V6-020 验过），零侵入核心路由 |
| 失败语义 | 全路径 fail-open，200 + 结构化载荷 | C7：读取是观察者面，绝不向 UI 抛 |
| 复用 helper | `_open_ptg_store_for_insights` + `_resolve_insight_founder` | 通用 helper，名字历史；零 churn 到通过的 insight 测试 |
| 纠正 | 本批只读，纠正延后 | 需先给原子打行级 provenance |
