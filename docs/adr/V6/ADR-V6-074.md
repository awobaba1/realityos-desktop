# ADR-V6-074 — C6 作用域边界：auxiliary/export LLM 调用不在 llm_call_logs 作用域

**状态**: Accepted
**日期**: 2026-07-21
**关联**: 第五轮 P2 backlog 复核 / [ADR-V6-071](ADR-V6-071.md)（quark llm_call_id 断链）/ [C6](../../../CLAUDE.md)（现实必须可重放）

## 背景（P2 backlog 项复核：过宽误判）

第四轮审计 P2 backlog 列：「`query_rewrite`/`teams_pipeline` LLM 调用无 `llm_call_log`」。第五轮复核**亲核代码**后发现这是过宽误判 —— 二者不构成真 C6 违例。

### 事实核查

- `plugins/memory/query_rewrite.py::rewrite_memory_query` 调 `agent.auxiliary_client.call_llm`，输出是一个**检索 query 字符串**（喂向量库做 memory 检索），**不写任何 PTG 现实行**（atom/event/quark/insight）。
- `plugins/teams_pipeline/pipeline.py::_generate_summary_payload` 调 `async_call_llm`，输出 `TeamsMeetingSummaryPayload` → 写**外部 sinks**（Notion/Linear/Teams）。`teams_pipeline/store.py` 建自己的表，全仓**零 import `PTGStore`/`plugins.memory`** —— 完全隔离的导出管线，**不写 PTG 现实表**。

二者都用 `auxiliary_client.call_llm`（上游 hermes-agent 的传输层），该层**设计上不写** RealityOS 的 `llm_call_logs`（那是 PTG-store 概念，上游不知道）。

### C6 的真实契约

C6「现实必须可重放」要求：**LLM 输出落成现实行的调用**必须记 `llm_call_logs`（全 I/O）+ 事件须带 `llm_call_id`。四个 reality 引擎全部满足（统一模式）：

| 引擎 | `_llm()` | `insert_llm_call_log` | 返回/透传 `llm_call_id` |
|---|---|---|---|
| atomizer | ✓ | ✓ (L691) | ✓ |
| quark extractor | ✓ | ✓ (L224) | ✓（ADR-071 刚补） |
| theory engine | ✓ | ✓ (L260) | ✓ |
| insights `_base` | ✓ | ✓ (L321) | ✓ |

模式：`call_llm`（传输）→ `insert_llm_call_log`（WORM 记录）→ 返回 `(content, llm_call_id, schema_valid)`。ADR-071 已补完 quark 这条链的最后一环。

## 决策：划定 C6 作用域边界（非代码改动）

**C6 `llm_call_logs` 作用域 = 写现实行的 LLM 调用（四个 reality 引擎）。** auxiliary/export 路径（query_rewrite / teams_pipeline / compression / vision / title_generation 等）属观察性日志（stdlib `logger.info`）范畴，不在 WORM `llm_call_logs` 作用域内。

这不是放松铁律 —— 是把 C6 的「reality」精确化到 PTG 现实表，避免两种假绿：
1. **瞎加日志装忙碌**：给 auxiliary 路径塞 `llm_call_logs`，既不动它们的传输层（`call_llm` 是上游共享，plumbing PTG 进去是大改），又制造「我修了 C6」的假象。
2. **边界模糊致漏网**：若不钉边界，将来有人把某个 auxiliary 路径接进 PTG（写现实行）却忘了加 `llm_call_id`，C6 就真违例了。

### D1：静态守卫钉边界（C4）

`tests/plugins/test_adr_v6_074_c6_reality_scope.py` 双向钉：
- **正向**：4 个 reality 引擎模块必须含 `insert_llm_call_log(` + 引用 `llm_call_id`（C6 契约）。
- **反向**：query_rewrite / teams_pipeline 必须不含 `PTGStore`/`insert_llm_call_log`（仍属 auxiliary）—— **若有人把它们接进 PTG，此测试会红，强制加日志**（不能静默通过）。

## 取舍：为何不「全量记日志」

严格字面读 C6「every LLM call MUST be logged」似应记所有调用。但：
- auxiliary `call_llm` 是上游 hermes-agent 共享传输层，被 ~7 类任务复用（compression/vision/title_generation/session_search/skills_hub/mcp/query_rewrite）。把 PTG `llm_call_logs`（RealityOS 专属 WORM 表）plumbing 进去是侵入上游核心的大改，回归面广。
- 这些调用不产出现实行，无「重放现实」需求 —— 它们的可观测性已由 stdlib logging（`logger.info` 记 task/provider/model）覆盖。
- 收益（多记几行非现实的调用日志）远不抵风险（动上游传输层）。YAGNI。

若将来 product 决定把 teams 摘要 / memory 检索结果**沉淀进 PTG 现实**（如 action_items → tasks），那一刻 C6 立即生效：必须经引擎 `_llm` 路径记日志 + 带 `llm_call_id`。D1 守卫会在那时变红强制执行。

## 合规性

- **C6**：所有 reality 引擎 100% 记日志 + 透传 `llm_call_id`（ADR-071 补完 quark 后无缺口）。auxiliary 边界由本 ADR 显式化 + 守卫钉死。
- **C1**：本「不做」决策本身记 ADR（一切皆决策，含「不做」）。
- **C3（生产现实即真理）**：基于亲核代码的事实（teams store 零 PTG import），非假设。
