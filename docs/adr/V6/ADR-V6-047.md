# ADR-V6-047 — 源文本纠正 + 重提取闭环（A4）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-040](ADR-V6-040.md) Phase 2-A A4 / [ADR-V6-045](ADR-V6-045.md) deletion_log /
danao13 [ADR-056](../../../danao13/docs/adr/ADR-056-源文本纠正与自学习闭环.md) + ADR-079

## 背景（纠错只降权，不重学）

V6 的 `feedback` 表（`schema.py:328`）能记录「这个原子不准」，`adjust_atom_confidence`
（`store.py:522`）把错原子的 `relation_confidence` 降到 0.3。但**到此为止**——它不重提取、
不改源文本、不算 diff。下一个含同一 ASR 错误的 memo 还会产出同样的错原子。danao13
ADR-056 的「源文本纠正与自学习闭环」在 V6 完全缺席：`memos.corrected_text` 列已存在却无写入
通路，无 `correct_memo_source_text`、无重提取。

## 决策

### D1：四件套，全部复用现有表，零迁移

| 组件 | 复用 | 新增 |
|---|---|---|
| memo 纠正 | `memos.corrected_text`（已存在）+ `version` | `correct_memo_source_text()` |
| 原子快照 | `_EVENT_SPINE.memo_id`（4 表已有）| `snapshot_memo_atom_ids()` |
| 原子退役 | `deletion_log`（ADR-V6-045）+ `transaction()` | `soft_delete_atom_ids()` |
| 洞察失效 | `insight_aggregation.expires_at`（已存在）| `invalidate_insights()` |

**无新表、无新列。** feedback.rating CHECK 保持 binary（不新增 'correction' 类型——ADR-V6-008
明示 rating 不动）；纠正走独立流，不复用 feedback 裁决语义。

### D2：`correct_memo_source_text()` —— 纯 CRUD，LLM 在锁外

`UPDATE memos SET corrected_text=?, version=version+1`（**绝不**改 `source_text`——C2 原文不可
变，danao13 ADR-056）。乐观并发 `expected_version` 不符→`version_conflict`。文本未变→`unchanged`
no-op。`memos` 无 `updated_at`（V6 约定：version bump 即变更标记；纠正事件时间戳由配对的
`deletion_log` 行 `created_at` 承载）。**绝不**在 store 内调 LLM——锁内 ~秒级网络调用会阻塞所有
其它线程。编排 живет 在 `correction.py` 自由函数。

### D3：`re_extract_memo()` —— 写后删（C2 关键不变式）

V6 的 `atomize()` 是「提取+写入合一」（不像 danao13 可拆成 in-memory 校验后写），且不能跨 LLM
调用持锁。故 V6-honest 写后删序列：

1. `correct_memo_source_text`（更新 corrected_text + version）。
2. `snapshot_memo_atom_ids` —— 快照 OLD 活原子 id（4 表，快 SELECT）。
3. `atomizer.atomize(corrected_text)` —— 写 NEW 原子（同 memo_id）。
4. **仅当 `ok=True`**：`soft_delete_atom_ids(OLD ids)`（按精确 id 退役，NEW 原子不动）+
   `invalidate_insights`。

**失败不变式**：重提取失败时 OLD 原子**保留**（corrected_text 已记，无伤；用户可重试）。这是
danao13 ADR-079「写后删」的 V6 翻译——绝不让一次失败的纠正把 memo 变成零原子。按 id 退役
（而非按 memo_id）消除 OLD/NEW 歧义：快照先于重写，新原子 id 不在快照里。

### D4：`hermes memo correct` CLI

`hermes memo correct <memo_id> --text "纠正后文本"`。新 `hermes_cli/subcommands/memo.py` +
`hermes_cli/memo_cmd.py`，按 `calibrate_cmd.py`/`task_cmd.py` 范式接线进 `main.py`。

## 不做（诚实边界）

- **不**做 ASR 热词注入 / LLM 词表注入 / 后处理纠正（danao13 ADR-056 三层「自学习」）——那是
  Phase 2.5 声学项（synthesis roadmap 明示推迟）。A4 诚实最小集 = 记录纠正→重提取→退役错原子→
  刷新洞察。**不**假装「自学习 ASR」。
- **不**在 store 内调 LLM（锁阻塞）。
- **不**在重提取成功前软删 OLD 原子（写后删铁律）。
- **不**忘记 `entity_events`（R0）——danao13 原本漏了，V6 同 4 表面同风险；`_MEMO_ATOM_TABLES`
  含全 4 表。

## 验证

- `tests/plugins/memory/test_adr_v6_047_correction.py`：correct（version 冲突/no-op/源文不变/
  corrected 落地）、snapshot（4 表/隔离/软删排除）、soft_delete_atom_ids（事务原子/审计/
  按 id 精确/NEW 不动）、invalidate_insights、re_extract（happy 写后删 OLD 退役 NEW 存活/
  failure OLD 保留/unchanged/not_found/version_conflict/C7）。
- `tests/hermes_cli/test_memo_cmd.py`：真实 on-disk SQLite 经二进制跑通 `memo correct`。
- 本地全量 + ruff 净；CI 三连随提交核验。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
|---|---|---|
| 纠正通路是否真落地 | ✅ | store 4 方法 + correction.py + CLI；smoke 实测 corrected/v+1/源文不变 |
| 写后删是否真守住 | ✅ | smoke 实测 failure 路径 OLD 原子存活（live=1, retired=0）|
| 是否假装自学习 ASR | ❌ 不假装 | 三层注入明示 defer Phase 2.5 |
| 是否在锁内调 LLM | ❌ 不做 | 编排 в correction.py，store 纯 CRUD |
