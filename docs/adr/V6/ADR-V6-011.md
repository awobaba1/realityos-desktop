# ADR-V6-011：Phase 1a 心脏——Atomizer（HL-12 抽取引擎）+ 置信门 + sync_turn 语义 sink

- **状态**：Accepted（Phase 1a 心脏实现定案；代码已落地，测试/核验见文末）
- **日期**：2026-07-19
- **Phase**：Phase 1a（数据大脑心脏——让 fork 从「只捕 memo」升级为「捕 memo + 自动抽原子入事件表」）
- **关联**：ADR-V6-008（PTG 数据层）、ADR-V6-009（迁移）、ADR-V6-010（分发）；V5：ADR-412（v10 防拆分）、extract-quality-baseline（v11 基线 R1 84.8/R3 85.2/R2 89.2）、`extraction_service.py` / `validation_service.py` / `extraction_schemas.py`
- **铁律**：C1（本 ADR）、C2（事件表软删+version，已具备）、C5（置信门+schema 门）、C6（llm_call_logs 全程 + prompt 版本化 v11 不覆盖）、C7（DLQ 全失败路径 + fail-open）

## 背景

V6 架构设计 §4.3B 规定：Atomizer 走 `background_review` 增 `_ATOM_REVIEW` 路径，直连 PTG SQL，把每 turn 的 memo 抽成 HL-12 原子写入事件表。但全量审计（2026-07-19）证实：**fork 只有 PTG 数据层骨架（memo 捕获 + 召回），抽取引擎 = 0 行**——`background_review._ATOM_REVIEW` 路径在 fork 中根本不存在（planned 未实现），15 个 realityos_* 模块 13 个不存在。fork 完成度 ~5-7%，是「空数据层 + 包装」，不是产品。

V5（danao13）则有一个**经 11 版迭代、生产验证**的抽取管线（v9 数据驱动把 R1 recall 62→84.8；v10 防拆分；v11 声学融合休眠）。其智慧凝结在三处资产：`prompts/hl12_extract_v11.md`（331 行）、`extraction_schemas.py`（5 原子 pydantic 契约）、`validation_service.py`（C5 置信门）。

本 ADR 决策：**不做 background_review 重路径**（它会在守护线程里 fork 一整个 AIAgent——为抽一段 JSON 大材小用，且继承父 agent 的工具注册表/内存管理），而是**忠实移植 V5 抽取管线的智慧**到 fork 的 PTG 数据层之上，用 hermes 已有的轻量 `call_llm` 接缝，让 fork 真正能自己产生原子。

## 决策

### 决策 1：抽取接缝 = `agent.auxiliary_client.call_llm`（非 background_review，非 ctx.llm）

三选一：

| 选项 | 做法 | 代价 | 结论 |
|---|---|---|---|
| **A（采纳）** | 直接调 `call_llm(task="extraction", messages=[system,user], extra_body=json_mode)`，镜像 `title_generator.py` 模式 | 需在 config.yaml 加 `auxiliary.extraction` 任务键（空=继承主模型） | 最轻、同步可用、与 fork 既有 plugin 调用一致 |
| B | 走架构设计原定的 `background_review._ATOM_REVIEW` 重路径 | 需 fork 整个 AIAgent 实例于守护线程，继承工具注册表/内存管理，过度工程 | 否（架构设想未实现，且过重） |
| C | 走 `ctx.llm`（PluginLlm facade） | 每 plugin 信任门 `allow_*_override` 配置摩擦；PTG 代码非 ctx-bound | 否（信任门多余摩擦） |

`call_llm` 同步原语（`auxiliary_client.py:6767`），返回 OpenAI 形状（`choices[0].message.content` / `.model` / `.usage`）。provider 的 `sync_turn` 是同步上下文 → 直接用同步 `call_llm`。`main_runtime` 暂传 None（`call_llm` 从 config.yaml 解析 provider）；后续 `sync_turn` 拿到 agent runtime 凭据再透传是改进项。

### 决策 2：忠实复用 V5 三资产 + 一处诚实改进

- **Prompt**：`hl12_extract_v11.md` **逐字**搬入 `plugins/memory/ptg/prompts/`（C6：版本化、绝不覆盖；V6 若迭代则出 v12，v11 不动）。v11 的声学融合章节在桌面端长期休眠（user prompt 无 `## 声学上下文` 段落即不生效，零回归）。
- **Schema 门**：`extraction_schemas.py` 5 原子 pydantic 类（R0/R1/R2/R3/R7）+ `ExtractionResult` **逐字**移植为 `atom_schemas.py`（`confidence: Field(ge=0,le=1)` 是承重约束，不得放宽）。
- **置信门**：`validation_service.validate_and_filter` 移植为 `ConfidenceEngine`，阈值复用 V5（R3=0.8 / R2=0.7 / R1=0.5 / R7=0.5 / R0=0.7，`>=`，单 `atom.confidence` 字段），**R1 neutral-mood 豁免逐字保留**（`state_type=='mood' and direction=='stable' and intensity=='low'` 绕过 0.5 门——丢它会静默丢中性情绪）。
- **🆕 诚实改进（偏离 V5，更忠 C2）**：V5 用 pydantic 鉴别 Union 解析整条 `ExtractionResult`——**一个坏原子拒绝整条 memo**（兄弟原子全丢）。V6 改为**逐原子校验**（按 `type` 分派到各自 model）：顶层结构错（非 dict / 无 summary / atoms 非列表）仍拒绝整条（同 V5）；单个原子的 schema 错只把它自己送 DLQ，兄弟存活。三桶：`valid_atoms`（写事件表）/ `filtered_atoms`（低置信→DLQ）/ `invalid_atoms`（schema 坏→DLQ）。

### 决策 3：原子→事件表路由（复用 V5 `_write_*_atom` 映射）

5 原子 → 4 事件表（与 PTG schema CHECK 1:1）：

| 原子 | 事件表 | 关键列 |
|---|---|---|
| R0_Entity | `entity_events` | entity_name, entity_category∈{place,organization,term}, mention_context |
| R3_Person | `identity_events` | person_name, sentiment, interaction_type |
| R2_Task | `meaning_events` | intent_class='Need_To_Do', task_description, urgency, deadline, task_status='pending' |
| R7_Expression | `meaning_events` | intent_class, task_description=content_summary |
| R1_SelfState | `feeling_events` | state_type, direction, intensity, ser_source='llm_text' |

全部经 `store._insert_event`（_EVENT_SPINE 共列 + 各表 extra），带 `confidence_base=relation_confidence=atom.confidence`、`memo_id`、`llm_call_id`。

### 决策 4：C6/C7 全覆盖（复用 V5 DLQ 分类法）

- **C6**：每次抽取调 `insert_llm_call_log`（成功+失败都记），字段与 V5 1:1（model/provider/prompt_input/full_prompt/response/input_tokens/output_tokens/latency_ms/success/schema_valid/cost_cny/error_type/error_msg）。**V5 把 `schema_valid` 留 NULL——V6 校验后回填**（C6 诚实）。
- **C7 DLQ 分类法**（逐字复用 V5）：`llm_extract/llm_error`、`llm_extract/json_parse_error`（V6 细化，V5 折进 llm_error）、`schema_validate/schema_invalid`、`atom_write/write_error`、`confidence_filter/below_confidence_threshold`。
- **cost 估算**：复用 V5 `_estimate_cost`（zhipu ¥2/¥8、deepseek ¥1/¥2、qwen ¥0.8/¥2 每 M token；未知 provider 按 zhipu 高估）。

### 决策 5：sync_turn 后台抽取（非阻塞 + 有界并发 + fail-open）

- memo 落库（同步）后，spawn 守护线程跑 `atomizer.atomize`——**不阻塞对话**。
- 模块级 `Semaphore(2)` 限并发（单 founder 桌面节奏足够；turn 本身立即返回）。
- 全链 fail-open（C7）：任何异常 → DLQ + log，**绝不逃出 sync_turn**（捕获面是 observation-only）。
- **默认 ON**：`plugins.ptg.atomize` 缺省 True（数据大脑每次启动必须跳动——同 `memory.provider="ptg"` 默认激活守卫的精神）；捕获测试用 `atomize: False` 显式 opt-out 保持确定性。

## 后果

**正向**：fork 从「空数据层」升级为「能自己产生原子」——每条 memo 自动产出 R0/R1/R2/R3/R7 原子入事件表，可检索/可重放/软删合规。V5 11 版迭代智慧无损迁移（基线可期持平）。C5/C6/C7 全覆盖。

**代价/风险**：
- 每 turn 一次 LLM 调用（后台），token 成本 = system prompt（~3k）+ user text。单 founder 内测可接受；未来需 debounce/批处理降本（留治）。
- `main_runtime=None`：抽取模型从 config.yaml 解析，若父 agent 用运行时 override 的凭据则取不到——首版可接受（config.yaml 有主模型即工作），透传 runtime 是改进项。
- `call_llm` 在测试环境无 provider → 立即抛 → DLQ + log（无真实调用、无网络），现有 25 个 provider 测试全绿验证不破。

**明确不做（Phase 1a 范围之外，勿当已完成）**：
- ❌ 实体/关系节点+边物化（`entities`/`relations` 表的 upsert）——事件即抽取产物，已可查；图谱聚合是后续 Phase。
- ❌ 声学引擎（SER/韵律/diarization）——Phase 2.5，数月工程；v11 prompt 的声学章节在桌面端长期休眠。
- ❌ `dedup_filter`（同 memo R2/R7 去重）——V5 可选安全网，V6 v1 跳过。
- ❌ 实体词汇表注入（ADR-049）——PTG 暂无 entities，跳过。
- ❌ 全量 V6 其余 12 个模块（quark/theory/insights/rag/...）。
