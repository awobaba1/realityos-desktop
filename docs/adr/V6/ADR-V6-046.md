# ADR-V6-046 — R12 显式用户通路（A3 / F8）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-040](ADR-V6-040.md) Phase 2-A A3 / F8 / [ADR-V6-028](ADR-V6-028.md) §11.5

## 背景（F8：R12 只有 LLM 一条路）

V6 的 R12（任务结果）今天**只能**由 LLM 在 atomizer 第二 pass（v12 prompt）从聊天文本里
抽取产出（`atomizer.py:552-566`）。用户**没有**任何显式手段标记「这件事我做完了 / 没成 /
延期」——即 ADR-V6-040 点名的 F8「R12 仅靠 LLM 从 memo 识别，无用户主权显式通路」。

`task_suggestions` 表虽存在（`schema.py:301-325`，带 5 态 CHECK），但**零写入逻辑**（F5 假
状态机）——它是 V 域「主动提醒」的预留概念，跟「用户显式标记任务结果」不是一回事，**不能**
借它假装 A3 已实现（synthesis roadmap 明示）。

## 决策

### D1：在 `meaning_events` 上原地晋升，不引入 `task_queue`

danao14 走 `/完成` → `transition_task`（写 `task_queue`）→ `record_outcome`（写 `meaning_events`
新 R12 行）。V6 **不**移植 `task_queue`（那是一整套派生快照 + nightly 聚合 + 推送调度子系统，
bolt-on 半成品 = 又一个 F5）。V6 的 atomizer 通常已为同一任务写过 R2 行，所以**原地 UPDATE
晋升**（R2→R12）比 danao14 的「新插一行」更正确（C2：version+1，不留第二行）。

`meaning_events` 已具备所需列（`schema.py:192-205`），**无需迁移**：
`task_status` / `completed_at` / `completion_note` / `atom_kind` / `version` / `updated_at`。

### D2：`PTGStore.mark_task_outcome()` —— 显式标记的唯一 sanctioned 通路

```python
mark_task_outcome(*, user_id, ref, outcome, actor="user", resolution_note=None) -> dict
```

- `outcome ∈ {completed, failed, delayed}` → `(task_status, is_overdue)` 映射与 atomizer 的 R12
  写入**逐字一致**（`atomizer.py:557-559`）：completed→completed/0，failed→dismissed/0，
  delayed→pending/1（'failed' 无枚举值→'dismissed' 闭但不达成；精确 outcome 存 completion_note）。
- `ref` 三路解析（首中即停）：
  1. `#N`/`N` → `list_open_tasks` 的 1-based 序号（与 danao14 `_INDEX_SQL` 同序契约）；
  2. 精确 `meaning_events.id`（uuid）→ 该行（允许对已关闭行改判）；
  3. `task_description` 子串 → **唯一**命中（0 或 >1 → 诊断消息，不臆断）。
- 单一 `transaction()`：`UPDATE meaning_events SET task_status/is_overdue/atom_kind='R12'/
  completed_at/completion_note/version+1/updated_at`。
- **审计**：`completion_note` JSON 携 `{outcome, actor, resolution_note, changed_at}` + `version`
  bump + `updated_at` —— 这是 V6 既有的版本化变更范式（与 `upsert_relation`/`upsert_entity` 同）。
  **不**写 `deletion_log`（那是软删退休专用，UPDATE 非删除）；**不**写 `feedback` 行
  （`feedback.rating` CHECK 限 thumbs_up/down，outcome 非裁决，语义不符）。
- 返回 `{ok, resolved, atom_id, task_ref, outcome, task_status, message}`，**永不抛**（C7）。

### D3：`list_open_tasks()` —— 序号基准 + 读取面

```python
list_open_tasks(user_id, *, limit=50) -> List[Dict]
```

`meaning_events WHERE atom_kind IN ('R2','R12') AND task_status IN ('pending','in_progress')
AND deleted_at IS NULL ORDER BY timestamp DESC, id`。`hermes task done #N` 的 N 即此序（1-based）。
纯 SQL，never raises（C7）。

### D4：`hermes task` CLI —— V6 的 `/完成` 对应物

`hermes task list` / `hermes task done <#N|ref> [--note ...]` /
`hermes task failed <...>` / `hermes task delayed <...>`。新 `hermes_cli/subcommands/task.py` +
`hermes_cli/task_cmd.py`（I/O 适配，开共享 PTGStore，resolve_founder），按 `calibrate_cmd.py`
范式接线进 `main.py`（`_BUILTIN_SUBCOMMANDS` + `build_task_parser` + `cmd_task` 委托）。

## 不做（诚实边界）

- **不**移植 danao14 `task_queue` + nightly `aggregate_open_tasks` + 推送调度。那是 V 域主动
  提醒子系统，超出 A3「显式通路」范围；半成品 = F5。
- **不**做 auto-promote（atomizer R2 → 聚合检测 done → record_outcome）。依赖未移植的
  `task_queue` 聚合器，跳过。显式-only 是诚实最小集。
- **不**用 `task_suggestions.status='completed'` 冒充 A3——该表零写入、是另一概念。
- **不**为已存在 R2 行新插 R12 行——原地 UPDATE 晋升（danao14 新插是因其 meaning_events 无
  R2 可改；V6 有）。

## 验证

- `tests/plugins/memory/test_adr_v6_046_r12_explicit.py`：list_open_tasks 排序、#N/子串/uuid 三路
  解析、R2→R12 晋升、version bump、completed_at、is_overdue(delayed)、改判已关闭行、outcome
  映射一致、歧义/未中/bad-outcome 诊断、C7 永不抛、跨用户隔离。
- `tests/hermes_cli/test_task_cmd.py`：真实 on-disk SQLite 经二进制跑通 `task list/done/failed`。
- 本地全量 + ruff 净；CI 三连随提交核验。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
|---|---|---|
| 显式通路是否真落地 | ✅ | store 方法 + CLI + smoke 实测 R2→R12 晋升/version+1 |
| 是否假装补了 task_queue | ❌ 不假装 | 明示不移植，显式-only |
| 是否假装 auto-promote | ❌ 不假装 | 依赖未移植聚合器，跳过 |
| 审计是否真可观测 | ✅ | completion_note JSON + version bump（实跑见 note.actor） |
