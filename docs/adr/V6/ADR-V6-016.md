## 背景

Phase 1a（ADR-V6-011）逐字搬运 V5 v11 基线，落地 **5 类原子**：R0/R1/R2/R3/R7（`atom_schemas.py`、`hl12_extract_v11.md`）。架构稿 §8 Phase 1b（L753）明确要「补 R1/R8/R9(共现)/R12」——其中 R1 已在 Phase 1a 落地（V5 v11 的 5 原子契约已含），**真实缺口是 R8 认知 / R9 共现情绪 / R12 结果**。

ConfidenceEngine 当前仅 **Stage1**（单 LLM 证据 + 5 类静态阈值，逐字复刻 V5）。架构稿 §4.3D（L351-365）定义完整三阶段公式 + 12 类 type_adjustment 全表，Phase 1b 要落到 Stage2。

Explore 实证（2026-07-19）：
- meaning_events 表已有 `topic_tags / completed_at / completion_note / task_status` 字段位（H8 fix），`intent_class` CHECK 8 值
- feeling_events 表已有 `emotion_vad / ser_source / trigger_source` 列（R9 字段位预留），`state_type` CHECK 4 值（R1 结构）
- 迁移框架 `_RECONCILE_COLUMNS` 是 **additive-only**（PRAGMA table_info + ALTER ADD COLUMN，`schema.py:599-613`），**不支持 CHECK 约束变更**（需表重建）

## 决策

### 1. R8/R9/R12 atom schema（C5 gate）

**R8 认知（Cognition）** — `R8CognitionAtom`：
- `type="R8_Cognition"` / `topic: str`（学习/思考主题）/ `knowledge_tags: list[str]`（知识点）/ `engagement: str(high|medium|low)` / `is_question: bool` / `confidence`
- delta（架构 L309）：knowledge_density / question_count / learning_engagement / weakness_flag。Phase 1b 文字子集——knowledge_density 连续值无来源（架构标「Phase 0 定代理定义」），用 `knowledge_tags` 离散标签 + `engagement` 近似，knowledge_density 留 Phase 2 统计。

**R9 共现情绪（Emotion）** — `R9EmotionAtom`：
- `type="R9_Emotion"` / `emotion_label: str`（开心/焦虑/愤怒…）/ `valence: str(positive|negative|neutral)` / `arousal: str(high|low)` / `trigger: str|None`（触发因素）/ `intensity: str(high|medium|low)` / `confidence`
- delta（架构 L310）：emotion_score / shift_vs_baseline / valence / arousal / causal_confidence。Phase 1b 共现版——**causal_confidence 固定 0.65 + co_occurrence_only=true**（架构 L361），valence/arousal 文字推断（声学连续值 Phase 2.5 NULL）。

**R12 结果（Outcome）** — `R12OutcomeAtom`：
- `type="R12_Outcome"` / `task_ref: str`（关联任务描述）/ `outcome: str(completed|failed|delayed)` / `resolution_note: str|None` / `confidence`
- delta（架构 L313）：completion_flag / failure_flag / delay_flag / resolution_status，编码进 `outcome` + `resolution_note`。
- 来源：文字完成语气（「搞定了/没成/拖到下周」）+ 工具执行结果（post_tool_call 闭环，Phase 1b 后续子项接，本批先支持文字抽取路径）。

### 2. 存储映射：纯 additive，零 CHECK 变更（C2）

**核心约束**：meaning_events.intent_class / feeling_events.state_type 都有 CHECK，扩展需表重建（C2 高风险）。决策**不改 CHECK**，复用现有枚举值 + 新增 **`atom_kind TEXT` additive 列**标记真实原子类型：

| 原子 | 表 | 复用枚举值 | atom_kind | 专属字段（全保真 JSON 落 completion_note / emotion_vad / trigger_source） |
|------|----|-----------|-----------|---------|
| R8 | meaning_events | intent_class='Other'（无 learning 枚举值；atom_kind='R8' 是真相源） | 'R8' | task_description=topic / topic_tags=knowledge_tags JSON / completion_note={engagement,is_question} JSON |
| R9 | feeling_events | state_type='mood'（情绪本就是 mood 的子类） | 'R9' | direction={positive:up,negative:down,neutral:stable} / intensity / emotion_vad={valence,arousal,label} JSON / trigger_source={trigger,atom:'R9_Emotion'} JSON / ser_source='llm_text' |
| R12 | meaning_events | intent_class='Need_To_Do' | 'R12' | task_description=task_ref / task_status={completed:completed, failed:dismissed, delayed:pending} + is_overdue=(outcome==delayed) / completion_note={outcome,resolution_note} JSON |

- `atom_kind` 列加入 `_RECONCILE_COLUMNS`（meaning_events + feeling_events）——框架原生 ADD COLUMN，零表重建。
- 顺手补 feeling_events 的 `trigger_source` 到 reconcile（CREATE TABLE 已有，但 `_RECONCILE_COLUMNS.feeling_events` 漏列，老库可能缺列致 R9 写入失败——防患于前）。
- SCHEMA_VERSION 5→6。

### 3. ConfidenceEngine Stage2（架构 §4.3D L351-365）

**完整公式**：`final_confidence = confidence_base × relation_confidence × type_adjustment × asr_quality_factor`

- **Stage1**（已实现）：单 LLM 证据 = `atom.confidence`。
- **Stage2**（本批新增）：**type_adjustment 12 类全表**（架构 L356-364）：
  - R3 identity ≥0.5 / R7·R8 meaning ≥0.5 / R9 feeling ≥0.3（causal_confidence 固定 0.65）/ R3社交·R5·R6 ≥0.4（R5/R6 Phase1 confidence=0 不入库）/ **R1 intensity_weight**（ADR-024：high=1.0/med=0.8/low=0.5）/ R2·R4·R10·R11·R12 ≥0.4（R10/R11 Phase1 confidence=0）
- **asr_quality_factor**：接口就位（默认 1.0），语音入站后由 STT 质量分填（Phase 1b STT 子项接）。
- **Stage3 跨信号交叉验证**：Phase 2+，本批不做。

**type_adjustment 实现方式**：每个原子的入库阈值 = `max(基础阈值, type_adjustment 加权后阈值)`，保持 Stage1 的 `>=` 比较。R1 的 intensity_weight 作为 type_adjustment 乘子（high/med/low → 1.0/0.8/0.5，即低 intensity 的 R1 需要更高 base confidence 才入库）。

### 4. D5 阈值下调整体延迟（诚实，不盲目执行）

架构稿 D5（ADR-V6-007，L366）规定 V6 入库阈值较 V5 整体下调（R3≥0.5 / R7·R8≥0.5 / R2≥0.4，换召回率），门禁 precision<0.85 回调。**但 precision⑦ 实证（commit 7758542e0，ADR-V6-012 收口）已证死阈值路径**：TP/FP 的 confidence 分布重叠不可分，下调阈值会同时砍掉 TP（recall 门崩），precision 天花板 ~73.5%。

**决策**：Phase 1b-1 **不下调 V5 阈值**（R3=0.8/R2=0.7/R1=R7=0.5/R0=0.7 保持），**只补 R8/R9/R12 新原子阈值**（R8=0.5/R9=0.3/R12=0.4）+ Stage2 type_adjustment。D5 下调整体延迟到「重标 samples + dedup（ADR-049）」治法落地后、评测验证 precision 真实改善时单独决策。这是 C3（生产现实即真理）——不盲目执行架构假设，据实证调整。

### 5. 双 pass 抽取架构（C6：v11 逐字不动）——经 200 样本基准实证的必要转向

**初版（single-pass v12，已废弃）**：本想新建 `hl12_extract_v12.md` 在 v11 五类基础上 ADD R8/R9/R12 规则，单次 LLM 调用抽 8 类。**200 样本基准实证此路不通**（C3 生产现实即真理）：

| 指标 | v11 基线 | single-pass v12（首版） | single-pass v12（收紧优先级后） |
|------|---------|----------------------|------------------------------|
| Precision | 65.6% | 59.4% | 57.7% |
| Recall | 90.2% | 83.4% | 85.2% |
| F1 | 76.0% | 69.4% | 68.8% |
| **R3 recall（Phase Gate ≥85%）** | ≥85% ✅ | **81.1% ❌** | **82.0% ❌** |

**根因（注意力稀释）**：单次抽 8 类原子，模型注意力被新类型分散，R3_Person 召回从 ≥85% 跌破 Phase Gate；prompt 加「提取优先级」指令 + 收紧 R8/R12 触发词仅恢复 ~2pt，**无法达标**（结构性问题，非 prompt 可解）。同时 v0（0 期望 R8/R9/R12）上模型凭空造出 48 个新原子——dump 实证**大半是 v0 标签缺口**（v0 早于 R8/R9/R12，未标，如「项目上线了」确系 R12 completed 真阳性），真·过抽取是少数（「孩子学会骑车」→他人成果、「打算看完书」→计划非学会、「电梯聊技术」→闲聊）。

**转向决策（双 pass）**：
- **pass 1（primary，v11 逐字）**：抽 R0/R1/R2/R3/R7。v11 prompt 文件 + system 字符串**逐字不动**，其 85%+ R3 召回基线**逐字节保留**——无注意力竞争可能。
- **pass 2（supplement，v12 聚焦）**：`hl12_extract_v12.md` 重写为**只抽 R8/R9/R12 的补充提取器**（剥离 R0-R7 规则，避免与 pass 1 重复抽取）。聚焦无注意力稀释。
- 每个 memo = 2 次 LLM 调用（桌面个人引擎，memo 量低，成本 ~¥0.01-0.10/memo 可忽略）；2 条 `llm_call_logs`（C6 每调用必记），pass 各自 schema_valid + fail-isolated（一 pass 错不阻断另一 pass）。
- 写入的原子携带**所在 pass 的 llm_call_id**（C6 traceability per call）。

**实现**（`atomizer.py`）：`atomize()` 调两次 `_extract_and_write_pass(version, ...)`（单 pass 全流程：LLM call → JSON parse → C5 gate → 写 → 图物化），pass1=v11、pass2=v12，计数累加。`PROMPT_VERSION="v11"`（primary pass 记录的 prompt_template_version，保基线标识）。

**为什么是双 pass 而非继续调 prompt**：precision⑦（commit 7758542e0）已证 prompt 微调是收益递减兔子洞；双 pass 一次性**确定性**消除基线回归风险（pass1 IS v11），符合「生产级对外发布」对零回归的要求。回到单 pass 是 Phase 1b-2+ 的可选优化（待 prompt 工程稳定后评估）。

### 6. 图谱物化（ADR-V6-011 决策 6 延续）

- **R8 认知** → topic 节点（entity_type='topic' 已支持）+ self→topic `learns` 边（value=engagement）。学习主题是可物化实体。
- **R9 共现情绪** → **无节点**（状态类，同 R1/R7，非实体）。情绪不构成图谱节点。
- **R12 结果** → task 节点（entity_type='task'）+ self→task `has_task` 边（value=outcome）。结果关联任务，物化为任务节点的状态。
