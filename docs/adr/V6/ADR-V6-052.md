# ADR-V6-052 — `_spawn_atomize` 外层 wrapper 补 DLQ（C7 闭环 / ADR-V6-040 P0-1 核验收口）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-040](ADR-V6-040.md) P0-1「atomize DLQ 兜底核验」/ [ADR-V6-011](ADR-V6-011.md) 决策 5「任何异常 → DLQ + log」/ [ADR-V6-007](ADR-V6-007.md) C5/C7 / 铁律 C7（无静默失败）

## 背景（P0-1 核验发现的真实缺口）

ADR-V6-040 P0-1 要求「核验 `_spawn_atomize` 失败路径是否落 DLQ（C7），若 fail-open 静默 = 补 DLQ」。
Phase 2-B 收口审计做了**代码级核验**（C3：生产现实即真理，非记忆），结论分两层：

| 失败层 | 既有处置 | 是否达标 |
| --- | --- | --- |
| `atomize()` **内部**预期失败（LLM 错 / JSON 解析 / schema 拒 / 原子写 / 图谱物化 / 置信过滤） | 6 类各自 `insert_dlq` + WARN + return/continue（ADR-V6-011 决策 6） | ✅ 已闭环 |
| `_spawn_atomize` **外层** wrapper（`atomize()` 逃逸的**未捕获**异常） | 仅 `logger.warning(...)`，**不写 DLQ** | ❌ 缺口 |

外层 wrapper 是线程入口的最后一道 `except Exception`。`atomize()` 内部虽有 6 处 try/except，但它们之间**存在未覆盖路径**——例如 LLM 调用成功后、`response.choices[0]` 取值时若响应畸形（空 choices）会抛 `IndexError`，此异常不在任何内部 try 内，逃逸到外层 wrapper，当前只 WARN-log。

ADR-V6-011 决策 5 自述契约是「**任何异常 → DLQ + log**，绝不逃出 `sync_turn`」。外层 wrapper 只 log 不 DLQ = 违反该自述契约 = C7 静默失败的边界变体（虽 WARN 可观测，但缺 DLQ 审计行）。ADR-V6-040 P0-1 的核验触发条件「若 fail-open 静默」在此从严解读为「缺 DLQ 审计」即触发补 DLQ。

## 决策

### D1：外层 wrapper 补 fail-safe DLQ

`_spawn_atomize._run` 的 `except Exception` 分支，在既有 WARN 之后追加一次 `self._store.insert_dlq(...)`：

- `source="atomize_thread"`、`error_type="uncaught_exception"`（与内部 6 类区分，明确这是「逃逸出 atomize 内部处理的 bug」）。
- `error_msg = f"{type(exc).__name__}: {exc}"`（异常类型+消息，便于诊断）。
- `original_data={"memo_id", "source_text"}`（与内部 DLQ 同范式；DLQ 是设计的 PII 汇，非 stdout/log，符合 ADR-403）。

### D2：DLQ 写入本身 fail-safe（最后一道不能崩捕获面）

外层 wrapper 是捕获面的**最后一道防线**。若 `insert_dlq` 自身抛错（DB 锁 / shutdown 中 store 已关），绝不能让守护线程崩溃——那是比「缺 DLQ 行」更糟的静默失败。故 `insert_dlq` 包在**嵌套 try/except** 内：内层抛错只 WARN-log，线程照常结束。即：先尽力留 DLQ 审计行；留不下也必存活 + WARN。

### D3：分层与单向流（不变）

本改动**仅在** provider 内部（捕获层）加一行 DLQ 写；不引入新依赖、不改 atomize 内部 6 类逻辑、不回写原子层。`insert_dlq` 是 store 既有的公共方法（`plugins/memory/ptg/store.py:479`），atomizer 已用 7 次，provider 直接复用，零新增 API 面。

## 不做（诚实边界）

- **不**改 `atomize()` 内部 6 类 DLQ 逻辑（它们已闭环，仅在收口审计复验确认）。
- **不**把外层 wrapper 的「未捕获异常」当成正常失败路径去重试/重抽取——那是 bug，DLQ 记录 + WARN 即可，人工/后续诊断。
- **不**在外层 DLQ 里塞 LLM 原始响应等大字段——`memo_id` + `source_text` 足够回溯复现（内部失败路径已按需留更详 original_data）。
- **不**为 `atomize()` 内部 6 处 try/except 之间的未覆盖路径（如 `choices[0]` IndexError）逐个补 try——外层 wrapper 的 fail-safe DLQ 已是这类 bug 的统一兜底；逐个补 try 是治标且会掩盖 bug，DLQ 让 bug 可观测才是治本。

## 验证

- `tests/plugins/memory/test_ptg_provider.py` 新增 2 例（C4）：
  - `test_spawn_atomize_uncaught_exception_writes_dlq`：注入「atomize 抛未捕获 `IndexError`」的 stub atomizer，断言 DLQ 恰好 +1 行、`source=atomize_thread` / `error_type=uncaught_exception` / `error_msg` 含异常类名与消息；线程有界 join 不 hang。
  - `test_spawn_atomize_dlq_write_failure_does_not_crash`：`monkeypatch insert_dlq` 抛 `OperationalError("database is locked")`，断言线程仍存活收尾、不逃逸（fail-safe DLQ 的 D2）。
- 本地 `bash scripts/run_tests.sh tests/plugins/memory/test_ptg_provider.py`：31 passed（29→31）；ruff 净；CI 三连随提交核验。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
