# ADR-V6-055 — eval-nightly dormancy 升级黄牌 + 7 天 liveness 检查（反假绿）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-030](ADR-V6-030.md)（eval-nightly 引入）/ [ADR-V6-026](ADR-V6-026.md)（precision deferred_structural）/ [ADR-V6-053](ADR-V6-053.md) P1-T2 / 五份审计 A5 P1 / 铁律 C7（无静默失败）

## 背景（A5 审计发现 recall gate 理论盾）

`eval-nightly.yml` 是 ADR-V6-030 落地的 recall 回归门禁（真实 DeepSeek 跑 200 sample 抽取评测，红则 exit 1）。A5 审计（2026-07-20）发现：

- **从未实跑**：`gh run list --workflow=eval-nightly.yml` → 空。原因：`DEEPSEEK_API_KEY` Actions secret 未配置。
- **dormancy 用 `::notice::`**（GitHub UI 最弱提示档），dormant 时 `exit 0 + ::notice::`——**诚实但事实上 recall 回归零拦截**。
- 评测本身可信：`run_eval.py` 契约（[ADR-V6-026](ADR-V6-026.md)）+ 契约测试钉死，本机手跑产 `v6_eval_report.json`（precision 0.656 / recall 0.902，全类型过门）。**问题不在门禁本身，在门禁从没被 armed**。

一个「存在但从没跑过」的门禁 = 反假绿语境下的理论盾——它给人「recall 有护栏」的错觉，实际零拦截。本 ADR 把它从「静默 dormant」升级为「可见黄牌 + liveness 探测」，让 armed 缺失无法隐藏。

## 决策

### D1：dormancy `::notice::` → `::warning::`

`::warning::` 在 GitHub Actions UI / PR check / run summary 显示为黄牌（比 `::notice::` 蓝条更显眼），且会被一些通知集成捕获。dormant 仍 `exit 0`（不阻断——评测不是 PR gate，是观测性，ADR-V6-030），但**可见性升级**，让「recall gate 沉睡」无法被忽视。warning 文案带 armed 命令（`gh secret set DEEPSEEK_API_KEY`），降低 armed 门槛。

### D2：加 7 天 liveness 检查 step（schedule 触发）

新增 `Recall-gate liveness check (ADR-V6-055)` step，**仅 schedule 触发**跑：

1. 查最近一次 completed 的 eval-nightly run；
2. 若在 7 天前 → `::warning::`（长期沉睡）；
3. 若近 7 天内但无 `eval-report` artifact（dormant run 不产 artifact，`if-no-files-found: ignore`）→ `::warning::`（DORMANT 理论盾）；
4. 有 artifact → `✓ armed`。

逻辑：dormant run 不产 artifact = armed run 从没成功跑过 = recall gate 形同虚设。本 step 把这个事实变成**跨 run 的可见黄牌**，而非单次 notice。

### D3：armed 路径文档化（不擅自配 key）

workflow 注释 + dormancy warning + liveness warning 三处都写明 armed 命令：

```
gh secret set DEEPSEEK_API_KEY
gh secret set DEEPSEEK_BASE_URL
```

**本 ADR 不擅自配 key**。armed 涉及：(a) 真实 DeepSeek key 值（本会话无此值）；(b) nightly 跑 30 sample × 真实 LLM 调用的持续成本；(c) key 进第三方 CI（GitHub encrypted secrets）的攻击面扩张。这是需用户拍板的敏感决策。本 ADR 的价值是**让 armed 缺失无法静默**，armed 本身交用户。

## 不做（诚实边界）

- **不**配真实 key——见 D3，敏感决策交用户。
- **不**把 dormancy 改成 `exit 1`（强制 armed）——那会让 nightly lane 在未 armed 时变红，与「dormant = exit 0 NOT green-faking」的 ADR-V6-030 契约冲突；观测性门禁不应伪装成阻断门。
- **不**改 `run_eval.py` 门禁逻辑（precision deferred_structural 等不变）——本 ADR 只治「门禁从不跑」，不治「门禁判什么」。

## 验证

- YAML 合法性：`python -c "import yaml; yaml.safe_load(open(...))"` ✓（7 steps，`permissions: actions: read`）。
- liveness step 仅 schedule 触发（`if: github.event_name == 'schedule'`），不影响 manual dispatch / 不影响 PR gate。
- CI 随提交三连核验。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
| --- | --- | --- |
| 是否声称「armed 了 recall gate」 | ❌ 不声称 | D3 明示不配 key；只升级可见性，armed 交用户 |
| dormancy 是否仍透明 | ✅ | `::warning::` + liveness 双重可见，比原 `::notice::` 更难忽视 |
| 是否伪装成阻断门 | ❌ 不伪装 | 仍 exit 0，观测性定位不变（ADR-V6-030） |
| liveness 逻辑是否真确 | ✅ | dormant run 不产 artifact（`if-no-files-found: ignore`）→ artifact 存在 = armed 跑过 |
