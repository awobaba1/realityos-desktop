# ADR-V6-028：创始人每日校准 CLI（§11.4/§11.5 闭环质量通道）

- **状态**：Accepted（批次 2，2026-07-20）
- **日期**：2026-07-20
- **Phase**：1b（反假绿收口 → Phase 2 Gate KR 地基）
- **关联**：ADR-V6-022（母纲，镜头 4 揭露 §11.5 未建）、ADR-V6-012（recent_atoms 读回路）、ADR-V6-016（atom_kind）、ADR-V6-027（quality_metrics 漂移治愈——本 ADR 的 correction_rate 落点）、ADR-083 F6（feedback revive）
- **铁律**：C1（本 ADR）、C2（降权非删除，原子 demote 不 erase；feedback 软删+version）、C4（3 文件 17 回归 + 785 广谱零回归）、C5（人改 confidence 的唯一受批通道，非 LLM 旁路）、C7（store 三方法 + run_calibration 全路径 fail-open，永不抛）

---

## 背景

ADR-V6-022 镜头 4 揭露：**§11.4/§11.5 的创始人每日校准闭环是「承诺膨胀」——宪法点名它是「唯一允许人改原子置信的通道」，但代码里根本没建**。Explore 亲核确认四个缺口：

1. **`recent_atoms` 不带行 id**：返回的原子 dict 只有 `type`/`confidence`/`_ts`/类型字段，**无 row id、无源表标识** → 无法定位要降权的具体事件表行。
2. **`insert_feedback` 不存在**：feedback 表 DDL 合规（rating CHECK + 软删 + version），但 store 无写入方法。
3. **`adjust_atom_confidence` 不存在**：ConfidenceEngine 是**无状态 GATE**（抽取时跑），不是 adjuster；「不准→降 confidence」的 §11.5 契约**无落地代码**。
4. **founder 解析散落**：`web_server._resolve_insight_founder` 是 3 层解析（config > ptg_meta），但埋在 web 层；新校准 CLI 若各自重写就会与 insight API 漂移（读不同 tenant 的原子）。

后果：创始人无法对「不准」的原子做任何事——错的原子永远顶着抽取时的高 confidence，周镜面/洞察继续把它当事实。**质量闭环断在「人反馈」这一环**。

## 决策

### 1. `recent_atoms` 加 `atom_id`（additive，向后兼容）

给 4 个事件表 SELECT 各加 `id` 列，返回 dict 加 `"atom_id": r["id"]`（8 个原子类型分支全覆盖：R3/R2/R7/R8/R12/R0/R1/R9）。**纯 additive**——消费方（match_atom 评测、周镜面聚合、记忆浏览器）读具体 key 忽略多余键，无波及。grep 亲核既有用法无整 dict 等价断言（唯一 `== []` 是空列表），785 回归零破坏证实。

### 2. `insert_feedback`（upsert/revive，verdict 编码进 target_type）

```
insert_feedback(*, user_id, target_type, target_id, rating, comment) -> str
```

feedback 表有**普通 UNIQUE 索引**（非 partial）on (user_id, target_type, target_id)——ADR-083 F6：对同一 target 重提 verdict 必须**复活软删行**（un-delete + version++），不能 INSERT（会 UNIQUE 冲突）。本方法实现该 upsert：存在行→更新 rating/comment/revive；新→插入。

**§11.4 三态 verdict 编码进 `target_type`**（`calibration_correct`/`calibration_wrong`/`calibration_surprise`），**不动 rating CHECK**——correct/surprise→`thumbs_up`，wrong→`thumbs_down`。**零迁移**（target_type 是 free-form 列）。`comment` 带 pre-adjust confidence 快照（`wrong;was=0.95;→floor=0.3`），是降权操作的审计轨迹。

### 3. `adjust_atom_confidence`（§11.5 唯一受批降权通道）

```
adjust_atom_confidence(*, user_id, atom_type, atom_id, new_confidence, reason) -> int
```

- **type→table 派发**（`_ATOM_TYPE_EVENT_TABLE`）：R3→identity_events、R2/R7/R8/R12→meaning_events、R0→entity_events、R1/R9→feeling_events。四表共享 `_EVENT_SPINE`（含 `relation_confidence`+`deleted_at`），故一行 `UPDATE <table> SET relation_confidence=? WHERE id=? AND user_id=? AND deleted_at IS NULL`。
- **降权非删除（C2）**：原子被 demote，**不 erase**；row 留存，deleted_at 仍 NULL。未来重抽取或重校准可再调高。
- **clamp [0,1]**（relation_confidence 有 CHECK BETWEEN）；越界自动夹。
- **返回 rowcount**（0 = 原子已软删/类型未知/id 不存在——log warning，**永不抛** C7）。
- **CALIBRATION_FLOOR=0.3**：不准的原子降到此 floor（非零、非乘 0.5——一个自信地错的 R2 med 0.95 必须跌破读端门槛，而非 lingering 在 0.48）。常量非用户配置，保证 correction_rate 跨次可比。

### 4. `plugins/memory/ptg/founder.py`（3 层解析单一来源）

把 `web_server._resolve_insight_founder` 的 3 层（config `founder_user_id` > ptg_meta）抽成共享 `resolve_founder(store, *, config=None)`。web_server 改为薄委托；**校准 CLI 用同一解析**，两者永不漂移。

### 5. `plugins/realityos_calibration`（纯逻辑，可测 rater 注入）

`run_calibration(*, store, user_id, atoms, rater, metric_date) -> CalibrationResult`：对每条原子调 `rater(display, idx, total)` 取 verdict → 写 feedback（+不准降权）→ 累计 correct/wrong/surprise/skipped → 写 `correction_rate` quality_metric（§11.2 KR 证据）。**rater 是注入式 callable**：CLI 给 `input()` 版，测试给确定性版。**逻辑与 I/O 分离**是 §11.5 契约可单测（无 tty）的关键。rater 抛 EOF/Ctrl-C → 提前收尾，返回部分结果（已判的仍落 correction_rate）。

### 6. `hermes calibrate` CLI（I/O 适配）

`hermes calibrate [--date YYYY-MM-DD] [--limit N]`（默认 limit 50）。开共享 PTGStore → resolve_founder → 算「今天 Beijing」[00:00,次日)→UTC 时间窗 → `recent_atoms(since,until,limit)` → 交互 rater → run_calibration → 印总结。无 founder / 空原子 / 早退出三短路均友好提示。

### 7. correction_rate quality_metric（§11.2 KR 证据落点）

会话末 `insert_quality_metric(metric_type='correction_rate', value=wrong/judged, sample_size=judged)`。这是 §8 Phase-Gate KR「创始人纠错率」的**唯一证据来源**（ADR-V6-027 治愈 quality_metrics 漂移后才写得起）。全 judged=0 时不写（避免 div-by-zero + 噪声行）。

## 后果

- **§11.5 闭环打通**：创始人可对「不准」原子降权；错原子不再永远顶高 confidence 污染周镜面/洞察。
- **C5 唯一受批通道**：人改 confidence 只此一路（adjust_atom_confidence），非 LLM 旁路；每次调整留 feedback 审计行。
- **§8 Gate KR 有证据落点**：correction_rate 时序在 quality_metrics 累积（待 ADR-V6-030 进 CI 后跨日可比）。
- **founder 解析去重**：web + CLI 同源，未来 founder-scoped 面不再各自重写。
- **recent_atoms 读回路增强**：atom_id 让记忆浏览器等未来面可定位单原子（纯 additive 收益）。

## 延迟项（诚实标注，非假绿）

- **「惊喜」案例库 few-shot 注入**：本 ADR 让惊喜原子的 **DATA** 进 feedback 累积；把案例库注入抽取 prompt 以改善精度是 **Phase 2**（ADR-V6-022 既定延迟；precision 天花板收窄需重标 samples/dedup，见 [[v6-phase1a-atomizer-heart]] precision⑦实证）。
- **真实 DeepSeek→原子→calibrate 全链路冒烟**：**本特性无 LLM 调用**（纯人机闭环），atom_id 改动经真实 Atomizer 回归（test_ptg_atomizer/phase1b 全绿）验证；真实 DeepSeek 端到端 blocked on key（同 ADR-030 桶D，待用户提供 key）。本特性的「真实」= 真实 on-disk SQLite + 真实 CLI 二进制（已冒烟），非 mock store。
- **校准调度自动化**：当前需创始人手动 `hermes calibrate`；每日定时提醒/调度属 Phase 2（与 ADR-V6-019 startup-lazy 同哲学，待 UI 渲染后接线）。
