# ADR-V6-044：K 域共现相关（F3+F4+F7）— R9→entity 边 + 事务/失效基建 + compute_k_correlations

- **状态**：Accepted（2026-07-20）
- **日期**：2026-07-20
- **Phase**：2-A（消费者层 K 域 / P1）
- **关联**：ADR-V6-040（母纲 / P0 清单 F3/F4/F7）、ADR-V6-042（F6 样本门，复用）、ADR-V6-016（R9 supplement）、danao14 `realityos_correlation/compute.py` + `realityos_atomizer/writer.py:113-128`
- **铁律**：C1（一 ADR + 测试）、C2（纯 UPDATE 失效，非 DELETE；append-only 守历史）、C5（统计置信门控）、C7（事务+DLQ，无静默失败）

---

## 背景

strategy-02 / Agent 调研核验 F3/F4/F7 三缺口（K 域前置 + K 域本体）：

- **F3**：atomizer.py:506-523 的 R9 写入 `trigger_source={"trigger":..., "atom":"R9_Emotion"}` —— **无 entity 键**。K 域需按 `trigger_source.entity` 分组（"我对谁/什么情绪偏负"），缺此键则 K 域无法启动。注：atomizer.py:586 `return # R1/R7/R9 — no graph node in Phase 1` 是 Phase 1 **有意边界**，非 bug；Phase 2 K 域突破此边界。
- **F4**：relations 表 `delta/completeness/consent_tag/version/deleted_at` 列**已存在**（schema.py:282-288，F4 原「无 delta」判定失真，已纠正）。真缺口：(1) `stale_at` 列不存在；(2) `upsert_relation` 无 `delta` 形参（列在但写不进）；(3) 无 `mark_k_correlation_stale()`；(4) 无 `transaction()` 上下文管理器。
- **F7**：V6 K 域**零实现**（F7）。danao14 `compute_k_correlations` 是纯统计生产实现，可移植。

---

## 决策

### D1：F4 store 基建（3 项）

1. **`relations.stale_at TEXT`**：加进 CREATE TABLE + `_RECONCILE_COLUMNS["relations"]`（additive ALTER，存量 DB 自动补 NULL）。
2. **`upsert_relation(..., delta: Optional[dict]=None)`**：新增形参，`delta` JSON 写入 relations.delta 列（列已存在，首次被写）。重证据时 delta 被覆盖（最新快照，符合 K 域「当前视图」语义；历史在 llm_call_logs + evidence_event_ids）。
3. **`transaction()` 上下文管理器** + **`mark_k_correlation_stale(user_id, subject_id, keep_object_ids, now_ts)`**：
   - `transaction()`：`BEGIN … COMMIT/ROLLBACK`，守 `self._lock`（V6 单连接 + threading.Lock）。K 域「重算 + 复活 + 失效」原子提交，中途失败不留孤儿边。
   - `mark_k_correlation_stale`：`UPDATE relations SET stale_at=? WHERE … K_Correlation AND object_id NOT IN keep` —— **纯 UPDATE，非 DELETE**（C2 append-only，value/delta/evidence 全保留；仅 stale_at 标记退出当前视图）。

### D2：F3 R9→entity（Option B 后置解析）

danao14 用 LLM 直捕 `trigger_entity`（prompt v-bump + schema）。**本 ADR 选 Option B（后置解析）**：在 atomizer R9 写入路径，把 `atom.trigger` 对用户已知实体词做匹配，命中则填 `trigger_source={"trigger":..., "entity": <匹配实体名>, "atom":"R9_Emotion"}`；未命中则 `entity=""`（情景型触发，正确退出 K 分组）。

**理由**：(1) 不动 v12 supplement prompt（C6 版本基线，反回归）；(2) K 域只需实体名分组，字符串匹配足够启动；(3) 0 额外 LLM 成本。**Option A（prompt v13 + schema +trigger_entity）文档化为未来保真升级**，当 K 域「他/她」代词消解等需更高保真时启用。

### D3：F7 compute_k_correlations（移植 + V6 适配）

新模块 `plugins/memory/ptg/k_correlation.py`，移植 danao14 `compute_k_correlations`。**V6 关键适配**：

- **valence 语义差异**：danao14 emotion_vad.valence 是 float（0-1，阈值 0.5）；**V6 R9 valence 是分类字符串**（`positive/negative/neutral`，atom.valence pattern）。故 `_NEG_THRESHOLD` → `valence == "negative"`（categorical，比 float 阈值更干净）。
- **baseline**：`P(neg) = count(negative) / total`；`P(neg|entity) = count(neg & entity)/sample`；`lift = P(neg|entity)/P(neg)`。
- **polarity**：`lift ≥ 1.2 → negative`（该实体偏负）；`lift ≤ 0.83 → positive`（该实体偏非负）；否则不下边。
- **confidence**：复用 F6 `grade_confidence_by_sample`（PRD §8.6.7：<10 不下边，10-29 →0.6，≥30→0.9）。
- **self entity**：V6 用 `_ensure_self_entity()`（真实 entity_id），非 danao14 字面量 `"ent_self"`。K 边 subject = self_id。
- **写边**：`upsert_relation(relation_type="K_Correlation", value=polarity, confidence=conf, delta={dimension, method, sample_size, valence_mean, p_neg_*, lift, polarity, evidence_event_ids, model_version})`。复活清 stale_at；失效 mark_stale。
- **correlation≠causation**（PRD 01:93）：边只声明共现，delta.method="conditional_probability" 明示。
- **calibration 排除**：danao14 用 wrong_atom_ids 排除错标 R9。**本 ADR 暂缓**（F6 样本门 ≥10 已提供基线可信地板；calibration 排除是可信增强，文档化为 Phase 2-A 后续，不假装已实现）。

返回写入边数。全 fail-open（C7）。

### D4：测试（C4）

- store：upsert_relation(delta) 写 delta 列；transaction() 原子（中途异常回滚）；mark_k_correlation_stale 只标未保留边、纯 UPDATE 不 DELETE；stale_at reconcile（存量 DB 补列）。
- atomizer：R9 trigger 命中实体 → trigger_source.entity 填充；未命中 → entity=""；trigger 空仍不崩。
- k_correlation：充足负样本→negative 边；偏正→positive 边；<10 样本→不下边；全无数据→现存 K 边全 stale；重算复活+失效一致性；correlation≠causation 标注。
