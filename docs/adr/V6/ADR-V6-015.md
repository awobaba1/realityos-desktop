# ADR-V6-015：隔离预存在上游 hermes 测试 flake——让 CI 信号诚实可用

- **状态**：Accepted（CI 收口，2026-07-19）
- **日期**：2026-07-19
- **Phase**：工程基建（CI 信号治理）
- **关联**：ADR-V6-014（V6 收口审计批次）、commit ab918cc05（export_v5 gitignore 修复）
- **铁律**：C7（无静默失败——本 ADR 反向适用：让原本静默吞掉的 CI 红显式可见）、C3（生产现实即真理——以真实 CI 运行取证）

---

## 背景

ADR-V6-014 批次（commit ac68d4cb3）push 后审查 CI，发现 **main 连续多次 CI 红**，且 v2026.7.18 / v2026.7.19 两个 Release 都是压着红 CI 发的——这正是用户警惕的「假绿」。

逐 run 取证后确认：**CI 红的根因是预存在上游 hermes 测试套件的 flake，与 RealityOS V6 批次无关**。证据链：

| 失败测试 | 预批量 run 7758542e0 | 预批量 run 6e280e207 | V6 批次 run ab918cc05 | 性质 |
|----------|----------------------|----------------------|------------------------|------|
| `tests/hermes_cli/test_nous_account.py`（14 测） | ❌ | ❌ | ❌ | 上游 Nous Portal 认证缺环境，确定性红 |
| `tests/test_export_v5.py` | ❌ | ❌ | ✅（V6 已修） | gitignore 误伤——commit ab918cc05 已修 |
| `tests/plugins/memory/test_ptg_provider.py`（segfault） | —（偶发） | —（偶发） | ❌（偶发） | sqlite-vec 原生扩展并行竞态 |
| `tests/test_windows_subprocess_no_window_flags.py` | ❌（flake） | — | —（偶发） | 捕获到并发 git 子进程的 flake |

**关键证明（非 V6 回归）**：
1. `test_nous_account.py` 在 V6 批次**之前**的两个 commit 上同形失败 → 预存在。
2. `test_ptg_provider.py` 的 segfault 是 C 层段错误；V6 对 `provider.py` 的改动是**纯 Python**（diff 无 `vec`/`ctypes`/`load_extension` import），不可能致 C 段错；且该文件**单跑全绿（28/28）**，仅在与其它文件同 slice 并行收集时偶发崩。
3. `test_export_v5.py` 是**确定性**红（gitignore 误伤 `export_v5.py`），已由 commit ab918cc05 修复（加 negation `!scripts/migrate_v5/export_v5.py`），CI 该项已转绿。

结论：**V6 批次零新增 CI 失败，且修复了一个（export_v5）**。余下红全为预存在上游噪声。

---

## 决策

经用户拍板（选项 A），对**已铁证预存在**的上游 flake 测试文件加**显式 module-level skip**，让 CI 信号诚实可用，然后发 v2026.7.20。

### 隔离清单（1 个文件——修正后）

1. **`tests/hermes_cli/test_nous_account.py`** — `pytest.skip(allow_module_level=True)`
   - 根因：14/21 测依赖 Nous Portal JWT/account-api 环境，CI 无凭证 → `info.source == 'none'`。
   - 取证：7758542e0 + 6e280e207（均 V6 批次前）同形失败。
   - 范围：整个上游 Nous Portal 模块（零 V6 相关）；7 个纯格式化测过但平凡，不值得保留半红文件。
   - 解除条件：本地带真实 Nous Portal 凭证运行时移除 skip。

### 不隔离（诚实记录 + 修正）

- **`tests/plugins/memory/test_ptg_provider.py` / sqlite-vec segfault**——**初版曾隔离，已撤销**。深查后发现：segfault 是**移动靶**——隔离 provider 后下一次 CI 它打到 `test_ptg_atomizer`（同 slice 1）。逐文件隔离 = 打地鼠，且每隔离一个就少跑一批 V6 PTG 回归测试，正是用户警惕的「假绿」。本地复现失败（store+atomizer=80 绿、store+atomizer+read_recall=93 绿），证明它是 **CI 环境特异**（Linux runner + 特定 sqlite-vec wheel + 8 路 slice 并行负载压力）的偶发原生扩展崩溃，非本地可复现、非逐文件确定性。**处置改为**：不隔离任何 PTG 文件（保留全量 V6 回归测试），segfault 作为**已知偶发 CI-env 原生扩展 flake** 记录，Release CI 若触发用 `gh run rerun --failed` 重跑（偶发，通常即过）。根因（sqlite-vec 多连接/进程内负载竞态）单列跟踪，非本批次 scope。
- **`tests/test_windows_subprocess_no_window_flags.py`**：仅 1 次 flake（捕获到并发 `git rev-parse FETCH_HEAD` 子进程），非确定性。**不预先隔离**——若 CI 再 flake，`gh run rerun --failed`。
- **`tests/test_export_v5.py`**：已由 commit ab918cc05 **修复**（非隔离），正常跑、正常绿。

---

## 为什么这不是「假绿」

假绿 = 隐藏失败、把红说成绿。本决策相反：

1. **显式可见**：每个 skip 在被测文件头部、CI 输出（`N skipped`）、git blame 三处可见，reason 字符串写明根因 + 取证 + ADR 编号。
2. **公开留痕**：本 ADR 逐条列隔离清单 + 根因 + 取证 + 解除条件，可审计。
3. **范围克制**：仅隔离「已铁证预存在 + 上游 + 非 V6 回归 + 确定性」的 flake（仅 nous_account 一文件）；V6 自有测试（security/sovereignty/dual_plugins/store/atomizer/provider/migrate/backup）**全部正常跑、全绿**，绝不靠隔离自己的回归测试来刷绿；确定性红（export_v5）是**修复**而非隔离；偶发 segfault 不隔离（移动靶，隔离=打地鼠+牺牲回归覆盖），改用 rerun-on-failure + 文档披露。
4. **CI 信号恢复意义**：隔离上游确定性噪声 + 修复确定性缺陷后，CI 红能真正反映「V6 代码健康」，而非被无关上游 flake 淹没——这才是 C7「无静默失败」在 CI 层的正向落地。

---

## 验证

- 本地：`pytest tests/hermes_cli/test_nous_account.py -q` → `21 skipped`（隔离生效）；PTG 集群 store+atomizer+read_recall 合跑 93 绿（segfault 本地不复现，证 CI-env 特异）。
- ruff 全量：All checks passed。
- CI（本次 push 后预期）：V6 全绿 + export_v5 绿 + nous_account skipped；偶发 sqlite-vec segfault 若触发，`gh run rerun --failed` 重跑（偶发，通常即过）后整体绿。
