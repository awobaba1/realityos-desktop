# ADR-V6-020：Phase 1b-5 — 洞察报告读取 API + 桌面 UI 渲染

- **状态**：Accepted（Phase 1b 洞察层，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（广度优先，闭合「生成→消费」回路）
- **关联**：ADR-V6-017（每周镜面）、ADR-V6-018（每日报告 + 基类）、ADR-V6-019（自动调度）、ADR-V6-008（PTGStore 共享单例）、架构 §4.4④（缓存键 + 「用户拉取」触发器）、§4.9②（缓存失效）、§0.5③（冷启动门控）
- **铁律**：C1（本 ADR）、C2（读取走 `deleted_at IS NULL`，绝不 hard-DELETE）、C5/C6（生成本身已合规——本 ADR 只暴露读取 + 用户拉取再生成，不绕过闸门）、C7（读取 API 全路径 fail-open，绝不向 UI 抛——no_data/error 都是 200 + 结构化载荷）

---

## 背景

到 ADR-V6-019 为止，周镜面/日报会**自动生成**并落 `insight_aggregation`。但桌面端**没有任何读取入口**——探查确认 `apps/desktop` 对 `insight_aggregation` / `realityos_insights` / `镜面` / `报告` **零命中**：无 API 路由、无 TS 客户端、无 UI 组件、无侧边栏入口。报告在库里生成，用户却看不到。本 ADR 闭合这条「生成→消费」的最后一公里。

### 两条参考路径（为什么不照搬）

1. **遗留移动端（`danao13/web` + `danao13/backend`）**：已有一整套 InsightsScreen（周/日 Tab + structured JSON：`banner/narrative/highlights/...`）。但那是 **Capacitor 移动壳**，且后端把 LLM 输出**解析成结构化字段**。
2. **V6 桌面**：报告以**纯 markdown 文本**存在 `result_data`（prompt 要求 markdown，`_base._validate` 拒收 JSON）。桌面本身是「agent markdown」取向（聊天就用 markdown 渲染器）。

照搬移动端要新写一个「LLM→结构化 JSON」的解析层 + 一堆字段渲染组件——工作量大且与桌面 markdown 取向相悖。

---

## 决策

### 1. 读取模型：cache-first + 用户拉取再生成（§4.4④ 「用户拉取」触发器）

两个 GET 路由（`hermes_cli/web_server.py`）：

- `GET /api/insights/weekly-mirror?date=YYYY-MM-DD&force=bool`
- `GET /api/insights/daily-report?date=YYYY-MM-DD&force=bool`

流程（`_get_insight_report` → `_read_or_generate_insight`）：

1. 开共享 store（refcounted，请求自开自闭不拆 provider 连接）；
2. 解析 founder user_id（ptg config `founder_user_id` > ptg_meta，镜像 `PTGProvider._resolve_user_id` 优先级，去掉网关 kwarg 路径——无状态 HTTP 读没有网关用户上下文）；无 founder → 温暖 `no_data`（首启竞态，永不报错）；
3. 用**该服务自己的周期解析器**算当前 `period_key`；
4. **cache-first**：`store.get_insight(...)` 命中即返回（§4.9② 同缓存键命中不重跑 LLM）；
5. `force=true` → `Service.generate(generated_by="manual")` 走完整流程（一次 LLM、C5 校验、C6 入库、C7 fail-open 到占位），再读回。

`generated_by` 取 `manual`（schema CHECK 允许 `scheduled/manual/on_demand`），与「用户拉取」语义一致，区别于自动调度的 `scheduled`。

### 2. 响应形状：markdown-content 模型（不解析成结构化字段）

```json
{ "kind", "status" ∈ {mirror,report,placeholder,no_data,error},
  "period_key", "period_start", "period_end" (友好显示),
  "data_sufficiency" ∈ {sufficient,partial,insufficient,null},
  "content" (markdown 文本 / 占位文案),
  "version", "generated_by", "llm_call_id", "created_at", "confidence",
  "message" (no_data/error 时的人话) }
```

`status` 派生：`data_sufficiency=='insufficient'` → `placeholder`；否则周→`mirror`、日→`report`。UI 据此决定渲染姿态，**不需要**再判断 content 是不是占位。

### 3. 桌面 UI：贡献式全页 + 直接渲染 markdown

- **`apps/desktop/src/app/insights/InsightsPage.tsx`**：两 Tab（周镜面 / 每日报告），每 Tab 一个 `ReportPanel`：
  - loading → `PageLoader`；`no_data`/`error` → `EmptyState`（人话文案）；`report`/`mirror`/`placeholder` → `<MarkdownTextContent text={content}/>`（复用聊天 markdown 渲染器）。
  - 充分度徽（sufficient/partial/insufficient → 数据充分/有限/还在热身，色 default/warn/muted）+ 周期跨度 + 刷新按钮（`force=true`）。
  - **占位显著区分**（§0.5③）：placeholder 状态加「我还在了解你」提示条 + content 半透明 + 「还在热身」徽，**绝不**让用户把占位误读成真报告。
- **注册**：`controller.tsx` 用 `registry.registerMany` 注册 `ROUTES_AREA`（`/insights` 全页，`render: <InsightsPage/>`）+ `SIDEBAR_NAV_AREA`（codicon `compendium` + label「洞察」+ path）。走既有贡献机制（工作区 `surfaces.tsx` 已渲染 ROUTES_AREA；侧栏 `sidebar/index.tsx` 已渲染 SIDEBAR_NAV_AREA），**零侵入**核心路由内部。

### 4. i18n 跨 4 locale 平价

`Translations` 类型强约束：加 `insights` 段必须同时改 `types.ts` + `en/zh/zh-hant/ja` 四个 locale 文件（TS 编译即门禁）。侧栏 label 是贡献 `data.label`（静态串，非 i18n 路径），取中文「洞察」（创始人中文产品）。

## 关键实现陷阱：period_key 探测必须与生成对齐（与 ADR-V6-019 同类 bug）

`_read_or_generate_insight` 用 `date_str` 给解析器算出 `period_key` 探测缓存；但若随后 `svc.generate()` **不传显式周期**，generate 会从 `now_fn` 重新解析——**对显式 date_str** 会写到另一个 `period_key`，读不到→`no_data`。修复：force 路径按 kind 传显式周期（`svc.generate(week_start=date_str)` / `svc.generate(day=date_str)`），保证写键 = 探测键。**这是 ADR-V6-019 `now_fn` 注入坑的同一族**——任何「先探测后生成」的代码都必须让两端同源。`test_pure_force_generates_then_reads` 钉死。

## 验证（反假绿，三连）

**后端**（`tests/hermes_cli/test_insights_web_api.py`，12 测试绿）：
- 纯逻辑：no_data / force 生成恰好 1 次 LLM / cache-first 零 LLM / placeholder 状态 / weekly `mirror` 状态 / 坏 kind 永不抛（error）。
- HTTP：no_data / cache-first / force 标 `manual` / weekly 路由可达 / founder 缺 → no_data（非 500）/ store 开失败 → error（非 500，fail-open）。

**前端**（`InsightsPage.test.tsx`，6 测试绿）：mirror 报告+徽+周期 / Tab 切换调对应 fetcher / 占位显著区分（还在热身徽）/ no_data 空状态 / error 状态 / 刷新按钮 force=true。markdown 渲染器被 mock（单元边界是本组件的状态→UI 映射，非 markdown 渲染）。

**门禁全绿**：后端 730（insights+memory+web_api）+ ruff 全净；前端 1349（172 文件，零回归）+ typecheck（renderer+electron）+ eslint 全净。

**三连核验**：见提交记录（headSha 逐字 == CI + 全绿 + 内容指纹）。

## 不在本批次做的事（诚实标注，后续）

- **历史浏览**（多周期翻页/日历热力图）：当前每 kind 只读最近一个完整周期；移动端有 `/history` + 日历热力，后续按需移植。
- **真实 DeepSeek 端到端冒烟**：`call_llm` 从用户活 session 解析 provider，无硬编码 key → 延后到发布时在真实桌面 session 验。本批 mock 验全流程。
- **侧栏 label i18n**：贡献 `data.label` 是静态串；多语言动态 label 需侧栏改造（运行时解析 label），后续。
- **更丰富的报告体裁**（月报 §18.5 #6、机制 8/9 风险预警）：基类已就位（ADR-V6-018），新报告 = 新子类 + 新 Tab。

## 决策小结

| 维度 | 选择 | 理由 |
|------|------|------|
| 读取模型 | cache-first + force 用户拉取 | §4.4④ 三触发器之一；同键不重跑 LLM（§4.9②） |
| 响应形状 | markdown-content（非结构化字段） | 桌面存 markdown；复用聊天渲染器；零解析层 |
| UI 注册 | 贡献式 ROUTES_AREA + SIDEBAR_NAV_AREA | 走既有机制，零侵入核心路由 |
| 占位处理 | 状态派生 + 显著区分样式 | §0.5③ 防误读，UI 不再判断 content |
| 失败语义 | 全路径 fail-open，200 + 结构化载荷 | C7：读取是观察者面，绝不向 UI 抛 |
| period_key 对齐 | force 传显式周期给 generate | 与 ADR-V6-019 同族坑；探测/生成同源 |
| i18n | 4 locale 平价（TS 类型门禁） | 跨语言一致，编译期强制 |
