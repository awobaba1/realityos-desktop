# ADR-V6-005: Strip 范围修正——继承优先，惰性化而非删除

- **Status**: Accepted（REV-9 纠正 2 处事实错误，见末尾「REV-9 修订」）
- **Date**: 2026-07-17
- **Supersedes**: V6 架构设计 §7.1 的 strip 清单（基于未核实的「零引用/可删」假设）
- **Refs**: C3 生产现实即真理 / ADR-V6-002 操作能力继承

## Context
V6 §7.1 列出 strip 清单（honcho / trajectory_compressor / batch_runner / 云 memory backend / Nous stub / background_review），依据是「训练噪音、零引用、可删」。**Phase 0 实际 grep 核实推翻了多数前提**：

| §7.1 假设 | grep 实证 | 结论 |
|---|---|---|
| `trajectory_compressor` 训练噪音、conversation_loop 零引用 | 被 `env_loader` + `mini_swe_runner` + `batch_runner` 集成使用 | ❌ 非死代码，是上下文/轨迹压缩（身体功能） |
| `batch_runner` 训练噪音 | 实为 **eval harness**（数据集跑 agent + checkpoint + trajectory 保存 + 工具统计） | ❌ 非噪音，V6 benchmark 复用 |
| `background_review._MEMORY_REVIEW` = Nous 外联 | 实为 **skill-write 审批后台**（approval / write_approval / skill_provenance） | ❌ 是审批门（身体功能），非外联 |
| conversation_loop 含 Nous entitlement 分支 | `grep -nci nous agent/conversation_loop.py` = **77**（全 `provider==nous` 门控） | ❌ 死耦合（77 处全不执行），V6 零运行耦合 |
| honcho 可删 | 10 处插件注册引用（cli/tools/hermes_cli） | ⚠️ 非干净删除 |

## Decision
Phase 0 P0-2 从「破坏性 strip」**降级为「惰性化 + 继承」**：

- **KEEP（身体功能继承，ADR-V6-002）**：`trajectory_compressor` / `batch_runner` / `background_review`（审批）/ honcho 插件代码本体。
- **DORMANT（V6 不配置即惰性，代码保留）**：
  - **Nous entitlement**：`hermes_cli/nous_*.py`（account / subscription / billing）+ cli usage/credits 命令——V6 入口不调用，brain（conversation_loop）零耦合。
  - **7 个云 memory provider**：byterover / hindsight / holographic / honcho / mem0 / openviking / retaindb / supermemory——多为云 API，主权层（ADR §5.1）不配置。
- **ACTIVE 替换（换大脑唯一缝）**：`PTGProvider` 取代 honcho 成为唯一 active `MemoryProvider`——基类在 **`agent/memory_provider.py`**（REV-9 纠正：原误写 `plugins/memory/__init__.py`，该文件只是插件加载器；基类实为 316 行，4 抽象方法 + 15 可选 hook），子类化 4 抽象方法（`name` / `is_available` / `initialize` / `get_tool_schemas`）+ 按需实现 hook（`sync_turn` / `prefetch` / `system_prompt_block` 等）即插即用。

## Rationale
ADR-V6-002 已定「身体全量继承白送」。删除身体代码违背继承哲学，且引入断链风险（10 处 honcho 引用）。`MemoryProvider` ABC 抽象保证 PTGProvider 插入即用，其余惰性——**不激活 ≠ 不存在**。

## Consequences
- ✅ Phase 0 P0-2 风险骤降：从「删 10+ 文件改 N 处引用」→「配置层不激活 + 验证惰性」，零断链。
- ✅ 尊重 fork-inherit；brain-swap 缝清晰（conversation_loop 零 Nous，PTGProvider 唯一 active provider）。
- ✅ V6 benchmark 可复用 `batch_runner`（与 ADR-V6-002 eval 能力一致）。
- ⚠️ 仓库体积含惰性云 provider 代码（可接受，Phase 0 不清理）。
- 后续清理（v2 backlog）：consumer 层稳定后，可正式移除云 provider 注册 + Nous CLI 命令。

## REV-9 修订（2026-07-17，全量精读纠正）

本 ADR 初版有 2 处事实错误，经 6 路 agent 全量精读核实纠正：

1. **ABC 位置**：原文「基类在 `plugins/memory/__init__.py`」**错误**。实证：基类 `MemoryProvider` 在 **`agent/memory_provider.py`**（316 行，4 抽象方法 + 15 可选 hook）。`plugins/memory/__init__.py` 只是插件加载/发现器。已在正文 Decision 段纠正。

2. **conversation_loop Nous 计数**：原文「`grep -nci nous conversation_loop.py` = 0」**错误**。实证：`grep -nci nous agent/conversation_loop.py` = **77**（我之前 grep 的路径错了，在根目录而非 `agent/` 子目录）。77 处全部是 `provider == "nous"` 门控分支 + 4 个 Nous 专属函数，V6 用智谱 GLM-5.2/DeepSeek（ADR-093），**全部永不执行**。结论不变（脑干死耦合、不阻塞），但事实已纠正。

**结论强化**：Nous 的处置维持「单点 stub（`hermes_cli/nous_account.py:322` 返回 `logged_in=False`）」——25+ 调用方全部 fail-closed，无需删 77 处门控代码。终版见 [ADR-V6-006]。
