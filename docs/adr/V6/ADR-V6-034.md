# ADR-V6-034：dingtalk/yuanbao 出站语音诚实可见包（action 19）

- **状态**：Accepted（Phase 1b 收口·批次 3，2026-07-20）
- **日期**：2026-07-20
- **Phase**：1b（诚实化）
- **关联**：ADR-V6-022（母纲，桶 C item C6 / action 19）、ADR-V6-029（6 通道 L1 验收）、ADR-V6-031（不可验证 voice = 假绿，同族反假绿原则）
- **铁律**：C4（语音能力矩阵回归测试）、C7（base fallback 须诚实通知非静默）、C3（实证真相优先于文档假设）

---

## 背景

ADR-V6-022 桶 C item C6 / action 19：「dingtalk/yuanbao 出站语音缺 —— P2」。

母纲审计（镜头 10）+ §2.1 叙事暗示「6 内置生产级（原生语音/图片）」但实际语音覆盖不全。需查明哪些通道真有出站语音、哪些缺、为什么。

## 亲核调查结论（反假绿，纠正审计过度悲观）

实证扫描 6 核心通道（`plugins/platforms/<ch>/adapter.py`）的 `send_voice` 覆写：

| 通道 | `send_voice` 覆写？ | 实现 |
|---|---|---|
| telegram | ✅ | 原生 voice bubble（.ogg/.opus）+ sendAudio（.mp3/.m4a） |
| discord | ✅ | 文件附件（channel lookup + upload） |
| slack | ✅ | `_upload_file` 文件上传 |
| whatsapp | ✅ | bridge 音频（`_send_media_to_bridge` audio） |
| feishu | ✅ | 音频附件（`outbound_message_type="audio"`） |
| **dingtalk** | ❌ | **故意不覆写**——继承 base fallback |

**真相：6 核心通道 5 个有原生 `send_voice`**（telegram/discord/slack/whatsapp/feishu），仅 **dingtalk** 缺。这比审计/§2.1 暗示的「语音覆盖不全」**更乐观**（审计镜头 10 称「仅 telegram/feishu/wecom」——wecom 非 6 核心之一，且漏计 discord/slack/whatsapp）。

- **dingtalk 缺语音的根因 = 平台 API 永久限制**：session-webhook 回复 API 仅支持 text/markdown（见 `adapter.py` send_image/send_image_file 已注），不能发语音/本地文件。**非 bug，非漏做**。
- **yuanbao**：`gateway/platforms/yuanbao.py`（legacy 非 6 核心通道）亦无 `send_voice`，仅有 `send_media`/`send_file`。非 6 通道验收集，低优先。

## 决策

### D1：语音能力矩阵测试（C4，可见+受跟踪）

在 `tests/gateway/test_six_channel_acceptance.py` 加 `TestOutboundVoiceCapability`（4 测）：
- `test_five_of_six_core_channels_have_native_send_voice`：钉死 telegram/discord/slack/whatsapp/feishu 覆写 `send_voice`（实证 5/6）。某通道加/减语音支持 → 此测红，强制 review 矩阵。
- `test_dingtalk_lacks_send_voice_intentionally_webhook_limited`：钉死 dingtalk `send_voice IS BasePlatformAdapter.send_voice`（继承 fallback）——**防未来塞入不经 OpenAPI SDK 的「假覆写」**（不可验证 = 假绿）。
- `test_base_send_voice_fallback_is_honest`：base fallback 须发友好通知（`Couldn't deliver the audio`）且**永不回显 host-local audio_path**（防文件系统布局泄漏）——C7 诚实通知 + 安全。
- `_adapter_cls` helper：按 `__module__` 过滤定位各通道 adapter 类（排除导入的 base/helper）。

### D2：dingtalk 源码就地文档（防误判为 bug）

`plugins/platforms/dingtalk/adapter.py` send 方法区前加 ADR-V6-034 注释块：说明 session-webhook 仅 text/markdown、`send_voice` 故意继承 base fallback、原生语音需 OpenAPI media SDK（独立集成，无 live 凭据=桶 D 不可验证）、指向矩阵测试。防未来 dev 误把「缺 send_voice」当 bug 去塞不可验证覆写。

### D3：不实现不可验证的 voice（反假绿铁律）

- **dingtalk voice**：需 DingTalk OpenAPI media-upload SDK 集成，**无 live 凭据无法验证**（桶 D）。写这种代码 = 潜在假绿（看似功能实则未验）。故**不写**，改为可见化（D1 矩阵 + D2 文档）。
- **yuanbao voice**：同理，legacy 非 6 核心，低优先，仅 ADR 记录为已知缺口。

## 不做（诚实）

- **不实现 dingtalk/yuanbao 原生 voice**（D3：不可验证，反假绿）。
- **不改 §2.1 叙事**（已在 ADR-V6-033 D4 落地出站语音现状注）。
- **不把 yuanbao 纳入 6 通道验收集**（它是 legacy `gateway/platforms/` 非 `plugins/platforms/`，非核心）。

## 后果

- ADR-V6-022 action 19 **诚实收口**：出站语音真相 = **6 核心 5/6 原生 voice**（纠正审计过度悲观），dingtalk 缺语音是平台 API 永久限制（非 bug），矩阵 CI 守门可见+受跟踪。
- 反假绿：语音能力不再是「文档声称 vs 实际」的黑盒——CI 矩阵钉死实证；不可验证的 voice 集成明确拒绝（防假绿），诚实 fallback（base 通知）受测试保护（C7）。
- 未来给 dingtalk 接 OpenAPI voice：改本 ADR + 加真实 live 冒烟（桶 D）+ 更新矩阵测试。
