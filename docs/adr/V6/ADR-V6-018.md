# ADR-V6-018：Phase 1b-3 — 每日报告（Daily Report）+ 抽取 InsightReportService 基类

- **状态**：Accepted（Phase 1b 洞察层，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（广度优先，第二个 INSIGHT 产品）
- **关联**：架构 §4.4（洞察缓存）/ §18.5 #2（prompt 模板）、ADR-V6-017（每周镜面，同框架首产品）、ADR-V6-011（Phase 1a 心脏，原子层基座）、ADR-V6-008（PTGStore 共享单例）
- **铁律**：C1（本 ADR）、C2（复用 insight_aggregation 缓存表，零新表）、C5（报告 C5 文本门槛）、C6（daily_report_v1 版本化 prompt + 每次 LLM 调用入 llm_call_logs）、C7（全路径 fail-open + DLQ + 占位降级）

---

## 背景

Phase 1b-2（ADR-V6-017）交付了**每周镜面**——把一周的原子编织成温暖具体的镜面，并落地了完整的生产流程（解析周期 → 聚合 → 冷启动门 → 一次 LLM → C5 → C6 → 缓存）。架构 §18.5 把**每日报告**定为 PRD #2 prompt 模板（Phase 1b）。

每日报告与每周镜面是**同构**的：同样是一段时间窗口的原子 → 一次 LLM → C5 校验 → C6 入库 → insight_aggregation 缓存。区别只在三处——

1. **窗口**：周（周一~周日，7 天）vs 日（单天）。
2. **门控信号**：周用 §0.5③ 的「注册<14天 OR memo<15」（防新用户 Day3 看到"你这周提了 0 次家人"卸载）；日不该套同一把尺——一天值得不值得报告，看的是**当天聊得够不够**，即当天原子数。
3. **prompt + 语气**：周镜面是"一周的弧线"（≤500 字、照见不说教）；日报是"一天的快速回看"（≤250 字、更轻更即时）。

若照搬 Phase 1b-2 的代码再写一份 `DailyReportService`，会有约 250 行与 `WeeklyMirrorService` 几乎逐字相同的流程代码（init / generate / LLM 调用 / C5 / C6 / DLQ / upsert）。这违反 DRY，且未来月报（§18.5 #6，Phase 2）会是第三份。

## 决策

### 1. 抽取共享基类 `InsightReportService`（`_base.py`）

把"解析周期 → 聚合 → 门控 → LLM → C5 → C6 → 缓存"的完整流程提到基类，子类只配置：

| 配置点 | 基类默认 | Weekly | Daily |
|--------|---------|--------|-------|
| `AGGREGATION_TYPE` | "" | `weekly_mirror` | `daily_report` |
| `PROMPT_FILE` / `PROMPT_VERSION` | "" / v1 | `weekly_mirror_v1.md` | `daily_report_v1.md` |
| `ENGINE`（dlq source + llm engine） | `insight` | `weekly_mirror` | `daily_report` |
| `REPORT_STATUS` | `report` | `mirror` | `report` |
| `MIN_CHARS`（C5 文本门槛） | 80 | 80 | 60（日报更短） |
| `CACHE_TTL_DAYS` | 14 | 14 | 7（日报更易过期） |
| `_resolve_period` | 抽象 | 周一~周日 | 单天 |
| `_gate` | 抽象 | §0.5③ memo/注册 | 当天原子数 |
| `_placeholder` | 抽象 | 周镜面引导/错误文案 | 日报引导/错误文案 |
| `_cache_data_days` | 7 | 7 | 1 |

子类覆盖的钩子：`_resolve_period` / `_gate(memo_total, reg_days, agg)` / `_placeholder(agg, *, error)` / `_validate` / `_cache_data_days` / `_dlq_error_type` / `_dlq_schema_invalid_type`。

**`aggregate_week` → `aggregate_window` 重命名**：聚合函数本就窗口无关（接收任意 `[start, end)`），原名是首个周调用方留下的。重命名为 `aggregate_window` 并保留 `aggregate_week = aggregate_window` 别名（向后兼容既有导入）。

### 2. WeeklyMirrorService 重构为薄子类（零行为变更）

`weekly_mirror.py` 从"完整流程 + 周配置混在一起"重构为"薄子类 + 模块函数"。**17 个周镜面测试零改动全绿**——这是重构的回归护栏。为严格保留已发布契约：

- `generate(week_start=...)` 转译为基类 `generate(period_start=...)`。
- `_resolve_period` 把 `_resolve_week` 的周命名键映射到基类的窗口词汇。
- 覆盖 `_dlq_error_type`/`_dlq_schema_invalid_type` 保留**已发布的** DLQ 标签（`source='weekly_mirror'`、`error_type='llm_mirror_error'` / `'mirror_schema_invalid'`）——测试钉死这两个标签。
- 模块函数 `_resolve_week` / `_validate_mirror` / `_placeholder` 保留（测试直接导入 + 子类委托）。

### 3. DailyReportService：窗口=1天 + 原子数门控

- **窗口**（`_resolve_day`）：默认 = 北京时间**昨天**（最近一个完整天）。半开 `[day 00:00, 次日 00:00)` 北京 → 转 UTC 查询。`period_key` = 那一天的日期；`end_display` = 同一天（单天报告）。
- **门控**（`_gate`，当日原子数）：
  - `atom_total < 3` → `insufficient` / `placeholder`（**不发 LLM**）
  - `3 ≤ atom_total < 8` → `partial` / `report`（结论要软）
  - `atom_total ≥ 8` → `sufficient` / `report`
- **为何不用 §0.5③ 的 memo/注册门**：那是**周**特有的（防新用户看到空周镜面卸载）。日报看的是"今天聊得够不够"——当天原子数。新用户首日自然 ~0 原子 → `insufficient` → 占位，无需额外注册检查。
- **prompt**（`daily_report_v1.md`，C6 独立 v1 永不覆盖）：≤250 字、≤1 个明天小提议、单日样本极小绝不下"你总是"绝对结论、空数据宁可不写。结构 `# 今日报告（{period_start}）` / `## 今天` / `## 值得记住的` / 可选 `## 明天`。

### 4. 复用 insight_aggregation，零新表（C2）

与周镜面完全相同——`aggregation_type='daily_report'`、唯一键 `(user, type, period_key)` ⇒ 每用户每天一行、重算 `ON CONFLICT DO UPDATE` 原地替换（`version+1`）。SCHEMA_VERSION **不变**（6）。

## 验证

- **15 项日报测试全绿**（`tests/plugins/realityos_insights/test_daily_report.py`）：
  - 聚合：窗口边界（窗外原子排除）+ 空天。
  - 四门控分支：insufficient(<3) / boundary(==3→partial) / partial(7) / sufficient(8)；placeholder 路径**零 LLM 调用**。
  - LLM 成功：C6 入库（v1 / success / schema_valid=1）。
  - LLM 失败 + C5 失败：降级占位 + DLQ（source=`daily_report`）+ llm_call_logs success=0。
  - 存储：insight_aggregation（`daily_report`）落库 + 重算原地替换（version→2，仍 1 行）+ 占位也落库。
  - 窗口：默认 = 昨天；显式 day。
  - 端到端：真实聚合数据（张三/述职/React/愤怒）进 prompt。
- **17 项周镜面测试零改动全绿**（重构回归护栏——基类抽取未改变周镜面任何行为）。
- **ruff 全量净**。
- **全量 memory + insights 套无回归**（`aggregate_window` 别名保留，`recent_atoms` 时间窗扩展向后兼容）。

## 后续（Phase 1b-4+）

1. **调度接线**：日报每日（如每晚 22:00 北京）+ 周报每周一 09:00，同 backup 的 startup-lazy 模式（桌面大脑可能没开 → ptg_meta 时间戳判过期）。两个服务共用同一调度框架。
2. **桌面 UI 渲染**：读 insight_aggregation 当日/当周行 → 渲染报告/占位。
3. **真实 LLM 端到端冒烟**（C3）：用真实 DeepSeek 跑一份日报 + 一份周报，人工审读"具体/不说教/不捏造/日报比周报轻"。
4. **创始人校准回流**（§11.4）：报告接受"不对"反馈 → 回流到提取质量基准。
5. **月报 #6**（Phase 2）：同框架第三个子类，验证基类抽象的可扩展性。
