# ADR-V6-025：主权护城河接线 — LLM egress 审计 + 外部内存 provider dormant guard（反假绿桶 C2）

- **状态**：Accepted（批次 1，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（反假绿收口）
- **关联**：ADR-V6-022（母纲，镜头 6 揭露）、ADR-093（智谱主/DeepSeek 备——§6.6 白名单基准）、ADR-V6-024（同属主权/捕获面收口）
- **铁律**：C1（本 ADR）、C3（生产现实——审计暴露「主权原语零接线」）、C7（全路径 fail-open，绝不抛进 agent 主循环）

---

## 背景

ADR-V6-022 镜头 6 揭露「主权护城河是纸老虎」的两条假绿：

1. **net-policy 原语零接线**：`plugins/realityos_security/policy.py` 提供 `fetch_guard` / `classify_url` / `assert_llm_egress`（§6.5 单一咽喉 + §6.6 sole-egress pin），原语真实有测试，但全代码库**零调用点**（排除定义/测试/docstring）。policy.py docstring 自承"wiring into every network call site is the next step"——承诺的「用户文本仅可离开设备到白名单 LLM 主机」从未被任何调用点强制或审计。
2. **外部 SaaS 内存 provider 可激活**：`honcho`（app.honcho.dev）/ `hindsight`（vectorize.io）/ mem0 等外部 SaaS 内存后端仍可通过 `memory.provider` 激活，激活即把用户记忆（PIPL L3）发往第三方主机——**直接违反 V6「零服务器/数据主权」招牌承诺**。

## 决策

接两个**真正强制主权**的调用点（均 fail-open，C7）。第三个候选咽喉点经实证后**主动放弃**（见下）。

### 1. §6.6 LLM egress 审计（`providers/__init__.py`）

在 `get_provider_profile` 解析 profile 时，调 `classify_url(profile.base_url)` 做 warn-once 审计：

- **公开非白名单主机**（`CAT_TOOL_FETCH`）→ **WARN 一次**：用户文本将经未审核第三方主机离机。fail-open（仅告警，不阻断，用户配置的 provider 仍可用）。
- **白名单主机**（`CAT_LLM_PROVIDER`，智谱/DeepSeek/OpenAI-compat 等）→ 静默。
- **loopback / 本地**（`CAT_BLOCKED_INTERNAL`，如 ollama @127.0.0.1）→ **静默**。本地模型服务器是**最主权**的选项（数据不离机），告警它反而是反主权的。

这让 §6.6 承诺**可审计**而非静默——一个未审核的公网 LLM 主机会在日志里响一次。

### 2. 外部内存 provider dormant guard（`plugins/memory/__init__.py`）

在 `load_memory_provider` 顶部加守卫：若 name ∈ `_V6_EXTERNAL_MEMORY_BLOCKLIST`（honcho/hindsight/mem0/openviking/holographic/retaindb/byterover）且未设 escape-hatch 环境变量 → **拒绝加载**（return None）+ 醒目 WARNING。调用方对 None 的既有回退（→ 本地 ptg / 内置精选记忆）生效，保证本地主权不破。

escape hatch：`HERMES_REALITYOS_ALLOW_EXTERNAL_MEMORY=1`（用户自担主权风险时可解锁，不硬堵）。

### 3. ❌ 主动放弃：`fetch_guard` 接 `open_credentialed_url`

**经实证放弃，非遗漏**。起初想把 `fetch_guard` 接进 `hermes_cli/urllib_security.open_credentialed_url`（凭证 stdlib fetch 的共享咽喉），但实测暴露两个不可调和的问题：

1. **会断本地 LLM provider**：`fetch_guard` 复用 `is_safe_url` 的 SSRF 地板，把 **loopback 全部挡掉**。但 `open_credentialed_url` 也服务于本地模型服务器（ollama/lm-studio @127.0.0.1）——本地是**最主权**选项，挡它反主权。`tests/providers/test_fetch_models_base_url.py` 的本地测试服务器即被挡（7 测试红）。
2. **咽喉策略二义**：`open_credentialed_url` 同时服务 LLM egress（应仅允许 loopback + 白名单公开）和 tool fetch（web/vision，任意公开主机）两类，单次 `fetch_guard` 调用无法表达两种策略。

而 tool-fetch 面**已由 `url_safety.is_safe_url` 直接守卫**（§0.6「fork 即得」继承，policy.py docstring 明示不重复造）。故 `fetch_guard` 在该咽喉的边际价值为**负**（重复 SSRF + 断本地）。

**`fetch_guard` 保留为已测原语**，留待未来 per-tool-site 审计接线（如要为每类 fetch 加独立 toggle/审计）。本 ADR 接的两个调用点才是**真正强制主权**的环节。

## 验证（C3）

`tests/hermes_cli/test_realityos_security_wiring.py`（15 测试全绿）：

| 类 | 覆盖 |
|---|---|
| `TestExternalMemoryDormantGuard` | 7 外部 provider 全拒（参数化）+ ptg 不受阻 + escape-hatch 谓词翻转 + sovereignty WARNING 落日志 |
| `TestLlmEgressAudit` | 公开非白名单 warn-once（确定性 mock classify_url，去 DNS 依赖）+ 白名单静默 + **loopback 静默**（最主权）+ 空 base_url 静默 + policy 缺失 fail-open |

回归：`ruff check .` 全量净；providers 全量 + realityos_security + urllib_security 共 **176 测试零破坏**（含此前被错误 fetch_guard 接线打红的 `test_fetch_models_base_url` 9 测试，revert 后全绿）。

## 后果

- **§6.6 可审计**：未审核公网 LLM 主机离机用户文本时，日志响一次（fail-open 不阻断）。
- **主权不破**：外部 SaaS 内存 provider 在 V6 dormant，激活即被拒 + 回退本地。escape hatch 保留用户自决权。
- **本地 provider 受保护**：ollama 等 loopback 模型服务器不被误挡/误警（最主权选项得到最宽待）。
- **诚实延迟**：`fetch_guard` per-site 接线记为后续审计增强，非主权缺口（tool-fetch 已有 SSRF 守卫）。

## 延迟项

- `fetch_guard` 接入各 tool-fetch 站点（web_extract/vision/browser）做统一分类审计 + 独立类别 toggle——边际价值低（is_safe_url 已守），优先级排后。
- `assert_llm_egress` 的 fail-closed 模式（配置开关 `realityos_security.enforce_llm_egress`）——Phase 2 视威胁模型再启，Phase 1 保持 fail-open warn。
