# ADR-V6-045 — `deletion_log` 审计表（C2 软删可追溯）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: C2（不可丢失）/ C7（无静默失败）/ R12 用户主权 / [ADR-V6-040](ADR-V6-040.md) P0-1 / [ADR-V6-044](ADR-V6-044.md)

## 背景（C2 的盲点）

V6 已有 `soft_delete()` 与 `cascade_soft_delete()`（§6.2 两模式遗忘）—— 两者都只
`UPDATE ... SET deleted_at = ?`。`deleted_at` 记录了**何时**退休，却**不**记录：

- **谁** 触发删除（`user` 显式 / `system` / `cascade` 级联 / `agent`）；
- **为何** 删除（R12 显式请求 / sovereignty §6.2 窗口 / 校正级联 …）；
- **删了什么**（退休前的行快照——无法事后取证）。

这是一处真实的可观测性缺口：一次静默级联可以软删数据而用户无从知晓——正是
ADR-V6-040 P0-1「反假绿基建」点名要补的 `deletion_log` 审计表。也是 R12 用户主权
「我有什么被退休了」问答的取数基底。

## 决策

### D1：新增 `deletion_log`（append-only WORM，C2 豁免）

```sql
CREATE TABLE deletion_log (
  id          TEXT PRIMARY KEY,
  created_at  TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  user_id     TEXT NOT NULL,
  table_name  TEXT NOT NULL,
  record_id   TEXT NOT NULL,
  actor       TEXT NOT NULL,    -- user | system | cascade | agent
  reason      TEXT NOT NULL DEFAULT '',
  snapshot    TEXT              -- 退休前行 JSON 快照（可空）
);
```

注册进 `APPEND_ONLY_TABLES`（与 `dlq_messages`/`llm_call_logs` 同列）→
`soft_delete()` 拒绝删它自身；`purge_soft_deleted()` 只迭代 `C2_USER_TABLES`，永不动它。
`export_user_data()` 自动把它纳入一键导出（PIPL §45——用户的审计轨迹属于用户）。
`SCHEMA_VERSION` 7→8。新增表走 `CREATE TABLE IF NOT EXISTS`，老库 reopen 即附加，
**无需** `_RECONCILE_COLUMNS` 条目。

### D2：`soft_delete()` 与审计 INSERT **同一事务**（atomicity is the point）

`soft_delete(table, row_id, *, actor='system', reason='', user_id=None)` 现在在
`store.transaction()`（ADR-V6-044 的 F4 原子上下文）里做三件事：

1. `SELECT *` 取退休前快照（forensic `snapshot`）；
2. `UPDATE deleted_at`；
3. `INSERT deletion_log`。

任一失败 → 整体回滚，`soft_delete` 返回 `False` 并 WARN（C7：**绝不**静默——一次没有
审计行的软删，正是 `deletion_log` 要堵的静默缺口）。**绝不**降级成「删成了但审计漏写」
那种 fail-open——那等于审计自诩完整性却给不了。

### D3：sovereignty 级联也按行审计

`_soft_delete_window()` 改为：在 `store.transaction()` 里先 `SELECT *` 快照窗口行 →
`UPDATE` → **每行一条** `deletion_log`（`actor='cascade'`，
`reason='sovereignty §6.2 mode {A|B} (…)'`）。窗口级 try/except 让单表审计失败不阻断
其它表的退休（C7： sovereignty 永不向调用方抛错，已返回 result dict）。

### D4：读取面 `list_deletion_log()`

`list_deletion_log(user_id, *, limit=200, table_name=None)`（newest-first，可按表过滤，
never raises C7）—— R12 主权 UI（Phase 2-A A3 显式通路）的取数面。

## 不做（诚实边界）

- **UI / CLI**：读取面已就绪，但「删除历史」桌面页是 Phase 2-A R12 显式通路（A3）的活，
  本 ADR 只交付基底与读取 API，**不**假装有 UI。
- **purge 一并清理 deletion_log**：故意不做。审计轨迹是 WORM——即便被审计行已
  `purge_soft_deleted` 物理删除，其退休记录仍须可追溯（否则 purge 本身变成不可观测的
  静默删除，违反 C7）。`deletion_log` 只随整个 PTG 库退役，永不按行删。

## 验证

- `tests/plugins/memory/test_adr_v6_045_deletion_log.py`：单行审计（actor/reason/快照）/
  幂等已删/原子回滚（日志写崩则不删）/级联按行审计/级联 reason/读取过滤/append-only
  自身拒删/导出纳入审计/export 含审计行。
- 本地全量 `tests/plugins/memory/` + `realityos_sovereignty/` + 双插件 + confidence/citation
  gate 绿；`ruff check .` 净。
- CI 三连（headSha + step 级非跳过 + 实际运行）核验随提交一起做。
