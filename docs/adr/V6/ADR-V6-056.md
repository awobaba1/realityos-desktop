# ADR-V6-056 — `hermes k` CLI 闭环 `compute_k_correlations`（治「做了没发」）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-044](ADR-V6-044.md)（K-domain compute 实现）/ [ADR-V6-053](ADR-V6-053.md) P1-T1 / [ADR-V6-037](ADR-V6-037.md)（「做了没发」= 最致命假绿）/ 五份审计 A1 P1

## 背景（A1 审计发现 K 域 compute 生产零可达）

ADR-V6-044 落地了 K-domain 相关性计算 `compute_k_correlations(store, user_id)`（纯统计：读 R9 `feeling_events` → `P(negative|entity)/P(negative)` lift → 样本门 F6 ≥10 → 写 `K_Correlation` 边到 `relations`，零 LLM，never raises C7）。但 A1 反假绿审计（2026-07-20）发现：

- **生产零可达**：`compute_k_correlations` 只被测试调用，**无 CLI、无调度、无 hook**。
- **「做了没发」**（ADR-V6-037 的最致命假绿类别）：实现完整、测试覆盖、CI 绿，但**没有任何生产路径触发它**——K 域 compute 永远不会跑，K_Correlation 边永远不会被写。
- **连带**：F3/F4 基建（`upsert_relation` delta / `mark_k_correlation_stale` / `store.transaction()`）也沦为「写基建无消费」。

这不是 ADR-V6-044 的实现 bug——实现是对的；缺的是**消费层闭环**（ADR-V6-040「先 A（消费层）后 B」哲学在 K 域的未完成部分）。

## 决策

### D1：`hermes k compute` — 显式 arm 入口

新增 `hermes_cli/k_cmd.py`（仿 [ADR-V6-051](ADR-V6-051.md) `theory_cmd.py` 的 I/O 适配器模式）：开 PTGStore → 解析 founder → 调 `compute_k_correlations(store, founder)` → 打印确认边数。

- 零参数即可跑（founder 自动解析）。
- 零边时诚实提示「R9 样本不足或无显著偏移」，**不**伪造成「无相关性」（反假绿）。
- 懒 import `compute_k_correlations`，CLI shell 不拉 K 域模块（与 theory_cmd 一致）。

### D2：`hermes k show` — 消费面（治 write-only-no-consumer）

`compute` 写的 K 边必须有读面，否则仍是「写了无消费」。`hermes k show` 渲染当前 `K_Correlation` 边（`stale_at IS NULL`）：每条 `实体：负偏/正偏（lift X，n=Y）` + 永久标注「相关性≠因果」（PRD 01:93）。空状态诚实提示「尚无边」，**不**编造。

### D3：`store.k_correlation_edges(user_id, *, include_stale=False)` — 专用读方法

`relations_for_user`（[ADR-V6-012](ADR-V6-012.md)）返回通用关系、不含 `delta`/`stale_at`。K 边的丰富信息（lift / sample_size / polarity / baselines）在 `relations.delta` JSON 里。新增专用读方法 join object entity 名 + parse `delta`，让 CLI 一次拿到渲染所需全部字段。纯 SQL，零 LLM。

### D4：main.py 注册（仿 theory）

`hermes_cli/subcommands/k.py::build_k_parser` + `main.py` 三处：import、`cmd_k` thin delegate（→ `k_cmd.cmd_k`）、`build_k_parser(subparsers, cmd_k=cmd_k)`。无子命令白名单需改。

## 不做（诚实边界）

- **不**自动调度（cron / startup hook）compute——手动触发符合 Phase 2-A 定位（K 域是「按需洞察」，不是实时）；自动调度是 ServiceOrchestrator（[ADR-V6-053](ADR-V6-053.md) P3 roadmap）的职责，本 ADR 不越界。
- **不**改 `compute_k_correlations` 的统计逻辑或 F6 门——实现本就正确（ADR-V6-044），本 ADR 只补消费层闭环。
- **不**声称「K 域现在自动运行」——`hermes k compute` 是手动 arm，用户须显式调。

## 验证

- `tests/hermes_cli/test_k_cmd.py`（6 例，C4）：
  - **`test_k_compute_writes_real_edges`**（**承重**）：seed 张三(10 neg+2 pos)+李四(2 neg+10 pos)，调 `hermes k compute`，断言输出「确认 2 条」**且** `store.k_correlation_edges` 真有 2 边（张三 negative / 李四 positive）+ delta 含 lift/sample_size。**这是 A1 怨点的直接反证**：compute 经 CLI 真写边。
  - `test_k_compute_cold_start_zero`：空 store → 0，诚实提示。
  - `test_k_compute_below_sample_gate`：王五 4 events <F6 ≥10 门 → 无边，诚实零。
  - `test_k_show_renders_edges`：compute 后 show 渲染「张三：负偏」「李四：正偏」+ lift + n + 「相关性≠因果」。
  - `test_k_show_empty_state`：无边 → 「尚无边」。
  - `test_k_no_action_prints_usage`：无子命令 → usage。
- 本地 `bash scripts/run_tests.sh tests/hermes_cli/test_k_cmd.py`：6 passed；ruff 净。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
| --- | --- | --- |
| compute 是否真经 CLI 可达 | ✅ | `hermes k compute` → `compute_k_correlations` 真调用（test_k_compute_writes_real_edges） |
| 是否真写边（非只 print） | ✅ | 测试断言 `store.k_correlation_edges` 返回 2 边，非仅 stdout |
| 是否声称自动调度 | ❌ 不声称 | D4 明示手动 arm；自动调度归 ServiceOrchestrator（P3） |

| 是否改了 compute 统计逻辑 | ❌ 未改 | 复用 ADR-V6-044 实现，只补 CLI + 读方法 |
