# ADR-V6-019：Phase 1b-4 — 洞察报告 startup-lazy 自动调度接线

- **状态**：Accepted（Phase 1b 洞察层，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（广度优先，INSIGHT 产品的「自动生成」收口）
- **关联**：ADR-V6-017（每周镜面）、ADR-V6-018（每日报告 + 基类）、ADR-V6-014（backup startup-lazy，本 ADR 镜面其哲学）、ADR-V6-008（PTGStore 共享单例 registry）、架构 §4.4（洞察缓存）、§11.4（创始人校准，后续）
- **铁律**：C1（本 ADR）、C2（生成走 upsert，绝不 hard-DELETE；判重读 `deleted_at IS NULL`）、C5/C6（生成本身已合规——本 ADR 只调度它，不绕过）、C7（调度全路径 fail-open，绝不向调用方抛异常；缺失/已存/失败三态皆返回结构化结果）

---

## 背景

Phase 1b-2（ADR-V6-017 每周镜面）与 1b-3（ADR-V6-018 每日报告）交付了两个 INSIGHT 产品：`WeeklyMirrorService` / `DailyReportService`，都是 `InsightReportService`（`_base.py`）的薄子类，**生产流程已全量落地**（解析周期 → 聚合 → 冷启动门 → 一次 LLM → C5 → C6 → `insight_aggregation` 缓存）。到 1b-3 为止，这两个服务**只能被手动/UI 显式调用**——`__init__.py` 的 `register()` 仅一句 debug log，明文标注「接线是文档化的下一步，此处未假绿」。

ADR-V6-018 收口段白纸黑字写了「后续 Phase 1b-4+：**调度接线**（日报每晚 + 周报每周一，同 backup startup-lazy）」。本 ADR 兑现这一承诺：**让洞察报告在没有人在场按按钮的情况下自动生成**。这是 INSIGHT 层从「能调」到「自己跑」的临门一脚。

### 为什么不能用 cron / 定时器

V6 是**桌面大脑**（零服务器、数据主权）。桌面进程**不一定在理想的墙钟时刻开着**：周报理想时刻是周一 09:00，但用户那时可能没开电脑；日报理想时刻是每晚收尾，但用户可能已经合盖。一个跑在进程内的 `threading.Timer`（cron 也一样）在进程不运行时**必然 miss**。这恰恰是 ADR-V6-014 backup 已经解决过的问题——它没有用定时器，而是**startup-lazy**：进程启动时检查「上次该做的做了没」，没做就补做。洞察报告是同一类问题，复用同一套哲学。

---

## 决策

### 1. startup-lazy，而非 cron/Timer

新增 `plugins/realityos_insights/scheduling.py`，提供 `run_due_reports(store, *, user_id, now=None, caller=None)`：

- 对每个报告类型（daily / weekly），用**该服务自己的周期解析器**算出「当前目标 period_key」（=最近一个**完整**周期——日报看昨天，周报看上一个完整的周一~周日）；
- 用 `PTGStore.get_insight(user, type, period_key)` 查缓存行；
- **已存在 → 跳过**（包括冷启动占位——占位也是一行，占位意味着「这周期判定过，不值得真跑 LLM」，不重跑直到周期翻页）；
- **不存在 → 调 `service.generate(generated_by="scheduled")` 生成**（生成本身已是 fail-open + 门控 + C5/C6 的完整流程）。

这个函数**纯函数式、可测、无定时器**——它就是「检查并补做一次」。真正决定「何时检查」的是下面的 trigger。

### 2. 触发器：进程启动时，insights 插件的 `register()` 起一个守护线程

`start_scheduler_if_due(*, enabled=True, founder_wait_seconds=60.0)`：

- **once-guard**（进程级 `_scheduler_started` + `_scheduler_guard`）：`register()` 可能在 cli / gateway / oneshot 多个入口各调一次，**只第一次真正起线程**，后续 no-op；
- 起一个 **daemon 线程** `name="realityos-insights-scheduler"`（daemon ⇒ 永不阻塞进程退出）；
- 线程体 `_run_startup_lazy`：开共享 store → 等 founder → `run_due_reports` → 关 store（`finally` 关，绝不泄漏）；
- **fail-open**：开 store 失败 / founder 没就绪 / 跑挂了，统统 `logger.warning` 后安静退出（C7——调度是观察者面，绝不向调用方抛）。

`register()` 用 `_scheduler_should_start()` 决定要不要起：默认起（生产桌面大脑）；**pytest 下不起**（`PYTEST_CURRENT_TEST` / `PYTEST_RUN_CONFIG` 在 env 里）；**`REALITYOS_INSIGHTS_AUTOSCHED=0/false/no/off` 显式关闭**（transient CLI 上下文 / 调试）。这保证测试绝不会真起一个会去开共享 store、打真实 LLM 的调度线程。

### 3. founder 等待（首次启动竞态）

首次启动时，`register()` 可能在 `PTGProvider.ensure_founder()` **之前**被调用——founder 行是几毫秒后才写进 `ptg_meta` 的。`_wait_for_founder(store, wait_seconds, poll_interval)` 用**有界轮询**（默认预算 60s、每 2s 探一次 `ptg_meta.founder_user_id`）等它出现；预算耗尽就安静跳过本次（下次启动重试）。**非首次启动**（常态）founder id 早就在库里，立即返回。

### 4. 分层铁律：「lower cannot call upper」——为什么调度由 insights 层 own，而非 memory 层

这是本 ADR 最关键的架构判断。铁律第 8 条：**「❌向上运行时依赖 — lower layers cannot call upper layers」**。分层是：memory（底）→ insights（上）。所以：

- ❌ **不能**在 `PTGProvider.initialize()` 里 spawn 调度线程去调 insights 服务——那是 lower 调 upper，违规；
- ✅ **正确做法**：insights 插件在自己的 `register()` 里 spawn 调度线程，线程只**向下**依赖 memory（`PTGStore`）。memory 层对 insights **零依赖**。

**对比 backup（ADR-V6-014）**：backup 也用 startup-lazy + 守护线程，但它的线程是从 `PTGProvider.initialize()` 起的——因为 **backup 本身就是 memory 层的事**（同层，没有向上边）。本 ADR 的洞察调度**不能照搬这个起点**，必须从 insights 层的 `register()` 起。同样哲学，不同分层落点。这条区别写进 `scheduling.py` 顶部 docstring，防止后人照 backup 抄错。

### 5. 共享 store，开新 handle 安全（ADR-V6-008 registry）

调度线程需要自己的 `PTGStore` handle（它不在 provider 的生命周期里）。ADR-V6-008 decision 3 的 **shared-store registry** 解决了「多个 handle 会不会互相把连接提前关掉」：`PTGStore(db_path=same)` 复用同一个连接 + `RLock`，按 `refs` 计数；`close()` 只减计数，到 0 才真关。所以调度线程 `PTGStore(db_path=resolve_db_path(load_ptg_config()))` 开 handle、用完 `close()`，**不会**把 provider 正在用的连接提前关掉。

## 关键实现陷阱：period_key 探测必须与生成对齐（`now_fn` 注入）

最隐蔽的一个 bug。`run_due_reports` 用注入的 `now` 算出 period_key 去探测缓存；但若随后 `service = svc_cls(store, ...)` **不传 `now_fn`**，则 `generate()` 内部会用默认的 `beijing_now()`（真实当前时刻）**重新**解析周期——探测的 key 和写入的 key 可能不一致（测试用固定 `now` 时必然不一致），导致**探测说「没有」、生成却写到了另一个 key**，下次启动又会探测到「没有」**无限重跑**。

**修复**：`_run_one` 构造服务时显式传 `now_fn=lambda: now`，保证生成用的周期**与刚刚探测的 key 同源**（都用同一个 `now`）。`scheduling.py` 的 `_run_one` docstring 钉死这一条。对应回归测试 `test_period_key_aligns_with_service`。

## 验证（反假绿，三连）

新增 `tests/plugins/realityos_insights/test_scheduling.py`（10 测试，全绿）：

1. **`test_skips_when_daily_row_exists`**——当前 period_key 有行 ⇒ 跳过，**零 LLM 调用**（`calls == []`）；
2. **`test_generates_daily_when_missing_and_gate_passes`**——缺行 + 8 原子过门 ⇒ 生成 sufficient，恰好 1 次 LLM 调用，入库可读；
3. **`test_cold_start_placeholder_blocks_rerun`**——冷启动占位也是一行 ⇒ 第二次 `run_due_reports` 跳过（幂等，不重跑 LLM）；
4. **`test_period_key_aligns_with_service`**——直接用 `DailyReportService` 生成昨天报告后，调度器探测到「已存在」（key 对齐）；
5. founder 读取（缺/在）、founder 等待（在则立即返回 / 缺则有界超时返回 None）4 测试；
6. once-guard（第二次起返回 False）+ enabled=False 不 arm guard 2 测试。

**回归护栏**：调度测试**只动 scheduling**，不碰周/日服务的 17+15 测试（它们是 ADR-V6-017/018 的已发布契约）。全量 `pytest tests/plugins/realityos_insights/ tests/plugins/memory/` + `ruff check .` 须全绿。

## 决策小结

| 维度 | 选择 | 理由 |
|------|------|------|
| 调度模型 | startup-lazy（非 cron/Timer） | 桌面进程不一定在理想时刻开着；复用 backup 已验证的哲学 |
| 触发落点 | insights 层 `register()` 起守护线程 | 铁律：lower 不能调 upper；memory 对 insights 零依赖 |
| store 共享 | `PTGStore(db_path=same)` registry | ADR-V6-008 refcount，调度 handle 不会提前关 provider 连接 |
| 幂等 | period_key 探测，含占位即跳过 | 不重跑 LLM；占位=「判过，不值得跑」 |
| period_key 对齐 | 服务构造传 `now_fn=lambda: now` | 探测 key 与生成 key 同源，防无限重跑 |
| 失败语义 | 全路径 fail-open，返回结构化三态 | C7：调度是观察者面，绝不抛 |
| 测试隔离 | pytest 下不起真实调度线程 | 测试绝不打真实 LLM / 开共享 store |
