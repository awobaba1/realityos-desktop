# ADR-V6-008: PTG 数据层——镜像 V5 实表 + 双插件 + 共享单例 + embeddings 可选

- **Status**: Accepted
- **Date**: 2026-07-17
- **Phase**: P0-4（个人时间线图数据层）
- **Supersedes / 修正**: 修正早期「V6 13 表」假设中 4 张在 V5 不存在的幻影表
- **Related**: ADR-V6-003（数据捕获边界）、ADR-V6-007（C1-C7 落地映射）、ADR-V6-006（computer-use 6 口子）

## 背景

P0-4 要给 V6 桌面端建「个人数据资产引擎」的存储底座（PTG = Personal Timeline Graph）。早期规划列了 13 张表（`realityos_users / memos / atoms / filtered_atoms / meaning_events / tasks / entity_events / people / emotion_records / feedback / insights / dlq_messages / llm_call_logs`）。逐行比对 V5 生产 SQLAlchemy 模型后发现，**其中 4 张在 V5 根本不存在**：

| 早期假设表 | V5 生产真相 |
|---|---|
| `atoms` | 内存 `Union` 类型（`extraction_schemas.py:73`），实际拆存 4 张事件表（identity_events R3人 / meaning_events R2任务+R7表达 / entity_events R0实体 / feeling_events R1自我） |
| `filtered_atoms` | 被拒 atom 直接进 `dlq_messages`（`extraction_service.py:228`），无独立表 |
| `people` | `entities` 表里 `entity_type='person'` 的子集 |
| `emotion_records` | 散在 `feeling_events.emotion_vad`（M2-F1）+ 声学层表 |

另两项纠正：`feedback.rating` 只有 2 值（`thumbs_up`/`thumbs_down`），ADR-089/414 加的是 19 个 `target_type` 而非 5 个 rating 类型；R8/R9/R12 不是独立 atom 类型（R12 靠 `meaning_events.completed_at/completion_note` 追踪）。

## 决策

### 决策 1：Schema 保真——镜像 V5 的 13 张真实表（REV-10 拍板：Option A）

V6 PTG schema 直接沿用 V5 生产真实表，不发明新表：

```
realityos_users          memos                   identity_events  (R3 Person)
meaning_events           entity_events  (R0)     feeling_events   (R1 SelfState + emotion_vad)
entities                 relations               task_suggestions (V-domain)
feedback                 insight_aggregation     dlq_messages     (吸收 filtered_atoms)
llm_call_logs
```

外加一个 sqlite-vec `vec0` 虚表（`memos_vec`，语义检索；H16 已验证 uv cpython-3.13 可 `enable_load_extension`）。

**理由**：
- **C3 生产现实即真理**——V5 的结构是 87 条 memo、44 用户、真实漏斗验证过的。
- **P0-5 迁移安全**——表对表近直接拷贝，无需 UNION 转换（Option B 统一 atoms 超表会让迁移变成一次性 transform，风险高）。
- `people`/`emotion` 作为 `entities`/`feeling_events` 的子集查询即可，不单列。

**代价（已知、可接受）**：继承 V5 的 4 表 atom 拆分——该拆分曾导致 ADR-088 的 R0 污染 bug。V6 写入层（后续提取 Phase）须自律：R0 实体只能进 `entity_events`，绝不进 `meaning_events`。这条纪律由 C5 schema 校验门把关，不靠人。

### 决策 2：双插件拆分

PTG 由**两个插件**承载，因为 memory 插件的 `register(ctx)` 收到的是 `_ProviderCollector`（`plugins/memory/__init__.py:330`），其 `register_hook` 是 **NO-OP**——memory 插件 loader 只收割 `provider`，注册不了生命周期 hook。

- **`plugins/memory/ptg/`**：`register(ctx)` → `ctx.register_memory_provider(PTGProvider)`。是 MemoryProvider，拿到 `sync_turn`（对话轮次捕获）+ `prefetch`（回忆注入）+ 工具 schema。
- **`plugins/observability/ptg_capture/`**：`register(ctx)` → `ctx.register_hook(...)`（用**真实** `PluginContext`，`hermes_cli/plugins.py:339`，其 `register_hook` 真生效）。注册 `post_tool_call`（工具执行捕获）+ `pre_gateway_dispatch`（出站消息捕获）+ `on_session_end`（会话边界）。

两者**共享 PTGStore 单例**（决策 3）。

### 决策 3：PTGStore 进程级共享连接单例

照搬 holographic `MemoryStore._shared` 注册表（`plugins/memory/holographic/store.py:113`）：同 db_path 的所有 PTGStore 实例共享**一条** sqlite 连接 + **一把** RLock，refcount 计数，关一个不撕裂兄弟。双插件（provider + observer）+ 主 agent + 每个 subagent 各自 `PTGStore(db_path)`，全部落到同一条连接，彻底消除「database is locked」多写竞争（holographic 已踩过此坑并解决）。autocommit（`isolation_level=None`）保证中途抛错不留悬挂事务。

### 决策 4：embeddings 入 `[embeddings]` extra，base 纯 FTS5 永远可用

sqlite-vec / numpy / fastembed（BGE-zh）是重 ML 依赖。参照 ADR-069「重 ML dep 不进基础镜像」教训：
- **base**：纯 sqlite + FTS5 全文检索（永远可用，无额外依赖）。
- **`[embeddings]` extra**：`sqlite-vec` + `numpy` + `fastembed`（本地 BGE-zh 语义检索）。
- PTGStore 启动时探测：能 `sqlite_vec.load(conn)` 就建 `memos_vec` 虚表；不能就降级为 BLOB+numpy（holographic 模式），再不行就纯 FTS5。三档降级，永不崩。

### 决策 5：捕获映射（修正 sync_turn 误判）

原「3 个采集 hook：post_tool_call / pre_gateway_dispatch / sync_turn」中的 **`sync_turn` 不是 plugin hook**，是 `MemoryProvider` 方法。修正后的捕获职责切分：

| 捕获面 | 落点 | 机制 |
|---|---|---|
| 对话轮次（user/assistant 文本） | `memos`（粗粒度）/ 后续提取 | `PTGProvider.sync_turn()`（MemoryProvider 方法） |
| 工具执行（tool_name/args/result） | `llm_call_logs` 关联 + 执行审计 | `post_tool_call` hook（黄金路径每个工具，model_tools.py:1300） |
| 出站消息（gateway event） | `memos`/事件 | `pre_gateway_dispatch` hook（用户消息 auth 前，gateway/run.py:9226） |
| 会话边界 | 触发批量提取 | `on_session_end` hook |

所有 observer 回调用 `**kwargs`（kwarg 集因 hook 而异 + 自动注入 `telemetry_schema_version`），返回 `None`（纯观察，绝不阻断；阻断只能靠 pre_tool_call/transform/middleware，PTG 不碰）。每个回调被 `invoke_hook` 的 try/except 隔离（C7 fail-isolated）。

## 铁律符合性

- **C1**：本 ADR。
- **C2**：13 表中 11 张用户数据表全有 `deleted_at` + `version`；`dlq_messages` + `llm_call_logs` 是 append-only 日志（V5 即如此，C2 明确豁免）。V6 绝无 hard DELETE。
- **C5**：Phase 0 的捕获是**观察式**（原样落库 user/assistant/tool 文本）；**提取 + schema 校验门**在后续 Phase 接入，届时 LLM 输出过校验才进事件表，被拒进 DLQ。
- **C6**：`llm_call_logs` 全字段（`prompt_input`/`response` JSON 全量、`prompt_template_version`、`schema_valid`、`cost_cny`、token/latency）——D1 对齐 V5 已确认。
- **C7**：捕获回调 fail-isolated（invoke_hook try/except + 各自 _safe 包装）；DLQ 记录所有失败。

## 后果

- P0-5 迁移 = V5 13 表近直接拷贝 + PG→SQLite 类型转换（UUID→TEXT / JSONB→TEXT(json) / TIMESTAMPTZ→TEXT(iso8601) / NUMERIC→REAL / BOOLEAN→INTEGER），`location_log` 不迁（D3，founder PG dump 永久备份）。
- `meaning_events` 同时承载 R2 任务 + R7 表达（按 `intent_class` 判别）——V6 查询层须带 intent_class 过滤。
- R0 写入纪律由后续提取 Phase 的 C5 门保证，本 ADR 仅声明约束。
- sqlite-vec 维度校验：首次插嵌入时校验 dim 与 `memos_vec` 定义一致，不符进 DLQ 不静默吞。

## 实现状态（P0-4b/c/d/e，2026-07-17）

数据层与双插件已落地，53 回归测试全绿、`ruff check .` 全量干净：

| 模块 | 文件 | 职责 |
|---|---|---|
| Schema | `plugins/memory/ptg/schema.py` | 13 表 DDL + FTS5 触发器 + `load_sqlite_vec`/`create_vec_table`/`validate_embedding_dim` + `_reconcile_columns`（声明式 additive 迁移）+ `C2_USER_TABLES`/`APPEND_ONLY_TABLES` 清单 |
| Store | `plugins/memory/ptg/store.py` | `PTGStore`：进程级共享连接单例（`_shared` 注册表，照搬 holographic `MemoryStore`）；`insert_memo`/`search_memos_fts`/`insert_llm_call_log`/`insert_dlq`/4 个事件插入/`soft_delete`/`count_rows`；共享 `load_ptg_config()` + `resolve_db_path()` |
| Provider | `plugins/memory/ptg/provider.py` | `PTGProvider(MemoryProvider)`：`sync_turn`→捕获 user 轮次为 memo（流经即捕获），`prefetch`→FTS5 回忆，`ptg_search` 工具，`system_prompt_block` 状态 |
| Memory 插件 | `plugins/memory/ptg/__init__.py` + `plugin.yaml` | `register(ctx)`→`register_memory_provider(PTGProvider)`；`name: ptg`，激活 = `memory.provider: ptg`，配置段 `plugins.ptg` |
| Capture 插件 | `plugins/observability/ptg_capture/__init__.py` + `plugin.yaml` | `register(ctx)`→`register_hook(post_tool_call/pre_gateway_dispatch/on_session_end)`；Phase 0 仅审计日志（语义落库随提取 Phase 接入）；`hooks:` 声明 3 hook |

**实现期确定的关键契约（超出原决策文本）：**

1. **共享单例靠同路径，不靠 import**：memory 与 capture 两插件各自 `resolve_db_path(load_ptg_config())` 解析到**同一** `<HERMES_HOME>/ptg.db`，共享连接注册表按 resolved path 去重 → 一条连接 + 一把 RLock。`db_path` 解析逻辑只此一份（store.py），两插件共用，杜绝漂移。
2. **`agent_context` 门控捕获**：`sync_turn` 仅在 `agent_context in ("primary", None)` 时落 memo——subagent/cron/flush 是 agent 内部流，不是真实用户数据，不进个人时间线（ABC 契约）。
3. **空/纯空白 user 轮次跳过**（无资产、无噪音）；assistant 回复存为 memo `summary`，轮次对可重建。
4. **Capture 插件 Phase 0 = 纯日志**：三个 hook 返回 `None`（allow，绝不阻断/改写），`_safe` 包装 + `invoke_hook` 双层 try/except（C7）。语义落库（工具执行事件 / 出站消息 / 会话批量提取）随提取 Phase 在 C5 门后接入，契约（kwarg 集）现已固定。
5. **失败优雅降级**：store 打不开 → provider 自禁用（`_store=None`，prompt/prefetch/sync_turn 全 no-op 不抛）；hook 触发前 founder 未建 → `_get_user_id()` 返 `None`（日志 `user=None`），非崩溃。

**测试**：`tests/plugins/memory/test_ptg_store.py`（19）+ `test_ptg_provider.py`（24）+ `tests/plugins/test_ptg_dual_plugins.py`（10）= 53 绿。覆盖 C2 软删+version 不变式、FTS5 召回（OR-join/排除软删）、append-only 日志、共享单例（同连接+同锁+交叉可见）、`agent_context` 门控、捕获永不抛、plugin.yaml 清单合法性。

## v3 修正（2026-07-18，真实数据验证触发）

P0-5 真实数据验证（见 ADR-V6-009「真实数据验证」节）暴露并修复了 2 处 schema/store 缺陷，SCHEMA_VERSION 2→3：

1. **`memos_fts` 分词器 `unicode61`→`trigram`**：unicode61 按空白切词，中文无空白 → CJK 召回全 0（合成英文测试一直绿，真实 founder 数据全 miss）。`search_memos_fts` 同步改为 **trigram 排序层 + 按 token LIKE OR-join 兜底层**（trigram 匹配不到 <3 字 CJK 如「北京/老婆」，LIKE 兜底）。旧 v2 库由 `_ensure_fts_trigram`（drop+recreate+rebuild）升级。回归测试 3 条（2字/3字 CJK、软删排除）。
2. **`ensure_founder` 改 UPSERT-promote**：V5 `is_founder` 生产全 false，忠实迁移带入 0；原实现仅 INSERT 置 1，已存在迁移行不修正。改为行已存在且 `is_founder=0` 时 UPDATE 置 1。回归测试 1 条。

`test_ptg_store.py` 19→28；整个 fork 1985 测试全绿、`ruff check .` 全净。

## v4 修正（2026-07-18，P0-6/7 触发——默认激活缺口）

本 ADR 一直说「用 `memory.provider: ptg` 激活」，但**从未种子这个默认值**——`hermes_cli/config.py` 的 DEFAULT_CONFIG 里 `memory.provider` 是 `""`（空 = 仅内置 curated memory）。后果：PTG 数据层（P0-4 实现 + P0-5 真实数据验证全过）在**真实 V6 启动时根本没加载**——`agent/agent_init.py:1438` 读 `mem_config.get("provider", "")`，空串直接跳过整个 provider 加载。每个单元测试都显式构造 PTGProvider，所以 1985 测试全绿掩盖了这个「数据大脑在运行时静默缺席」（C7 无静默失败的同类陷阱，正是合成测试掩盖真实行为的 ADR-088 教训再现）。

**修复（fork-specific 默认值变更）：**
1. `config.py`：DEFAULT_CONFIG `memory.provider` `""`→`"ptg"`。V6 全局默认 = PTG。内置 curated memory 仍并行（不互斥）；ptg 缺 `[embeddings]` extra 时优雅降级到纯 FTS5，故该默认在任何安装上都安全。
2. `apps/desktop/src/app/settings/constants.ts`：`memory.provider` 下拉枚举加入 `'ptg'`（原仅 `['', 'builtin', 'hindsight', 'honcho']`——UI 里连选都选不到 ptg），置首位以反映 V6 激活默认。
3. 回归测试 `test_v6_default_memory_provider_is_ptg`（test_ptg_provider.py 25→…）：锁默认值，防静默回退。

**验证**：memory 全量 558 测试绿（含 3 个曾断言旧 `""` 默认的 upstream 测试——核实均为「选 builtin→清空」的*动作*测试非默认测试，PUT 显式写空串，V6 默认下仍通过）；`ruff check .` 全净。
