# ADR-V6-059 — sequential 路径协作式 per-tool deadline（治挂死无观测）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-053](ADR-V6-053.md) P1-T8 / 五份审计 T6 安全 / 铁律 C7 无静默失败 / concurrent 路径 deadline 先例（`tool_executor.py:~673`）

## 背景（审计发现 sequential 零 deadline）

`agent/tool_executor.py` 有两条工具执行路径：

| 路径 | 触发 | deadline | 挂死处置 |
| --- | --- | --- | --- |
| `execute_tool_calls_concurrent`（:327） | 多工具批量 | ✅ `_resolve_concurrent_tool_timeout()`（420s 批级）+ ThreadPoolExecutor `wait(timeout)` + 到点 abandon | ✅ fabricate `tool_timeout` + per-thread `_set_interrupt` |
| `execute_tool_calls_sequential`（:1028） | 单工具 / 交互工具 | ❌ **无** | ❌ **无限悬挂，无 DLQ/日志/interrupt** |

T6 安全审计（2026-07-20）发现：sequential 路径**零 deadline**。单个工具挂死（网络工具无 timeout / MCP 工具无响应 / 子进程卡死）→ agent turn **无限阻塞**，无任何可观测信号（违反 C7 无静默失败）。concurrent 路径有完整 deadline 闭环，sequential 没有——同一执行器两路径不一致。

## 决策

### D1：协作式 per-tool deadline（daemon 看门狗）

新增 `_resolve_sequential_tool_timeout()`（env `HERMES_SEQUENTIAL_TOOL_TIMEOUT_S`，默认 420s，镜像 concurrent resolver）+ `_arm_sequential_tool_deadline(...)`。

`_arm_sequential_tool_deadline` 启一个 daemon `threading.Timer`，到点 `_fire()`：

1. `_ra()._set_interrupt(True, main_thread.ident)` —— **协作式中断信号**，与 concurrent 路径（`:587`）同一 API。轮询 interrupt 的工具（terminal / execute_code / MCP-with-cancellation）到点中止。
2. `agent._interrupt_requested = True` —— 级联停止：循环在每个工具前检查此 flag（`:~1042`），后续工具被 skip。
3. `_emit_terminal_post_tool_call(status="timeout", error_type="tool_timeout")` —— **可观测性**，与 concurrent 到点处置（`:~843`）同一持久错误日志路径。挂死不再静默。
4. `logger.warning` —— 明示哪个工具超时。

返回的 Timer 由调用方在工具返回后 `.cancel()`（`Timer.cancel()` 幂等）。

### D2：接线（arm 一次 / cancel 一次）

`execute_tool_calls_sequential`：
- 函数头 resolve 一次 `_seq_tool_timeout`；循环顶 init `_seq_deadline_timer = None`。
- `tool_start_time` 之后 arm（仅 `not _execution_blocked` 且 timeout 非空）。
- 迭代末 `_current_tool = None` 处 cancel（**单一汇聚 cancel 点**——所有正常/错误路径都到这；挂死则不到达，watchdog 正确触发）。

### D3：为何是「协作式」而非「强制 pre-emption」（诚实边界）

sequential 路径**在 agent 主线程**执行工具（它专为交互/单调用工具服务）。Python **无法安全终止同步主线程调用**——没有 `Thread.kill()`，`os.exit` 会杀整个进程。所以无法像 concurrent 那样把工具丢 worker 线程再 abandon。

强制 pre-emption 的唯一办法是把 sequential 工具也移到 worker 线程——但这会**破坏 sequential 路径存在的理由**（交互工具需要主线程：stdin / 线程绑定资源）。spinner-vs-else 两个分支是 UI 区分（是否显示 spinner），**不是**线程安全区分，所以没有任何子分支可安全 future-wrap。

因此本 ADR 走**协作式**：靠工具自检 interrupt flag 中止。这是 concurrent 路径**已经在用**的同一协作模型（concurrent 的 `_set_interrupt` 也只对轮询 interrupt 的工具有效——见 `:~812` 注释「the per-thread interrupt signal already causes individual tools (terminal, execute_code) to abort, but tools without interrupt checks (web_search, read_file) will run to completion」）。sequential 与 concurrent 在这点上**对齐**了：协作工具到点中止，非协作工具跑完。

## 不做（诚实边界）

- **不**强制 pre-empt 非协作主线程工具——Python 限制（D3）。concurrent 路径是批处理非交互工具的 remedy；sequential 服务交互工具，强制 pre-emption 会破坏交互语义。
- **不**把工具移 worker 线程——破坏交互工具主线程需求。
- **不**改 concurrent 路径——它本就有完整 deadline。
- **不**声称「sequential 挂死被根治」——协作工具被救（到点中止 + 可观测），非协作工具仍跑完但**现在可观测**（C7 gap 关闭：从「静默无限悬挂」变「可观测 + 协作中止 + 级联停止」）。

## 验证

- `tests/run_agent/test_adr_v6_059_sequential_tool_deadline.py`（C4，8 例）：
  - `TestResolveSequentialToolTimeout`（4 例）：默认 420 / env 覆盖 / 非法回退 / ≤0 禁用（None）。
  - `test_disabled_returns_none`：None/0/负 → 不 arm。
  - **`test_fires_sets_interrupt_and_emits`**（**承重**）：arm 0.2s → 到点 `_interrupt_requested=True` + `_set_interrupt(True,main_tid)` 记录 + `tool_timeout` emit（status/error_type/function_name/tool_call_id 全验）。直接证明 deadline 闭环。
  - `test_cancel_prevents_firing`：cancel 后不误触发（快工具不误杀）。
  - `test_watchdog_never_raises_on_patch_failure`：`_set_interrupt` 抛错时 watchdog 吞掉 + emit 仍发（看门狗自身 fail-safe，绝不抛）。
- 回归：`test_tool_call_incremental_persistence` / `test_tool_batch_segmentation` / `test_malformed_tool_arguments` / `test_tool_call_guardrail_runtime` / `test_concurrent_interrupt` 共 50 例零回归（默认 420s 在快测试永不触发，watchdog 被 cancel，零干扰）。
- 本地 ruff 净；tool_executor 导入 OK。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
| --- | --- | --- |
| sequential 挂死是否仍静默 | ❌ 不再 | 到点 `_interrupt_requested` + `tool_timeout` emit（C7 gap 关闭） |
| 是否真触发（非只声明） | ✅ | test_fires_sets_interrupt_and_emits 断言 interrupt + emit 真发生 |
| 是否声称强杀非协作工具 | ❌ 不声称 | D3：Python 限制，协作式对齐 concurrent 模型 |
| 是否破坏交互工具主线程 | ❌ 不破坏 | 不移 worker 线程；watchdog 只设 flag + emit |
| cancel 是否误杀快工具 | ❌ 不会 | test_cancel_prevents_firing 钉死 |
| 是否回归既有 sequential 测试 | ❌ 不回归 | 50 例零回归 |
| 看门狗自身是否 fail-safe | ✅ | `_fire` 全程 try/except，test_watchdog_never_raises 钉死 |
