# ADR-V6-048 — M 域人物画像（纯 SQL 聚合，A5）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-040](ADR-V6-040.md) Phase 2-A A5 / [ADR-V6-044](ADR-V6-044.md) F3 R9→entity /
[ADR-V6-011](ADR-V6-011.md) 决策6 图实体 / danao13 人物 Tab

## 背景（有人物节点，没有人物画像读取面）

V6 早就有完整的 M 域写入侧：`entities` 表存人物节点（`entity_type='person'`），
`identity_events` 存每次提及（含 sentiment/interaction_type），`relations` 存关系边，
ADR-V6-044 还让 R9 `feeling_events.trigger_source.entity` 能挂到具体人。但**读取侧缺位**：
`list_top_entities` 是给 LLM 词表用的扁平列表，`relations_for_user` 只给边——没有一个把
「这个人是谁、互动多少、最近聊啥、和谁有关系、引发我什么情绪」聚成一体的读取面。danao13 有
人物 Tab；V6 在 A5 之前等价物为零。

## 决策

### D1：纯 SQL 聚合，零 LLM（诚实边界）

`list_people` + `person_profile` 全部是 `SELECT … GROUP BY` 直读，**不经任何 LLM 合成**。
理由：M 域读取是「把已落地的事实摆出来」，不是「生成新判断」。混入 LLM 会违反 C5（agent 不能
无门地写现实）且让一个只读查询变得又慢又不可重放（C6）。CLI 末尾明示「未经 LLM 合成」，
不假装有「画像洞察」。

### D2：`list_people(user_id, *, limit=50)`

`SELECT … FROM entities WHERE entity_type='person' AND deleted_at IS NULL ORDER BY
mention_count DESC, last_seen_at DESC`。排除 self 节点（`properties.is_self`，复用
`list_top_entities` 的判定）。返回 `{entity_id, entity_name, mention_count,
first_seen_at, last_seen_at, aliases}`。提及频次高的排前——最常出现的人最可能是 ASR 错也最该被
校准。

### D3：`person_profile(user_id, entity_id, *, recent_limit=10)`

五面只读聚合：

| 面 | 来源 | 说明 |
|---|---|---|
| header | `entities` | name / mention_count / first·last_seen / aliases |
| interaction_breakdown | `identity_events` GROUP BY (interaction_type, sentiment) | by_type / by_sentiment / total |
| recent_contexts | `identity_events` ORDER BY timestamp DESC LIMIT N | 最近 N 条 mention_context |
| relations | 复用 `relations_for_user(entity_id)` | 图邻域（不重写） |
| emotions | `feeling_events` atom_kind='R9' WHERE trigger_source.entity 命中 | count + 近 N 条 trigger |

**实体↔事件按名匹配**：`identity_events` / `feeling_events` 没有外键到 `entities.id`（它们按
原始文本 `person_name` / `trigger_source.entity` 落库），故用 `LOWER(TRIM(person_name)) IN
(canonical + aliases)` 匹配。别名从 `entities.properties.aliases` 取——这就是别名字段的读取用途。

### D4：`hermes people list` / `hermes people show <名字|id>`

新 `hermes_cli/subcommands/people.py` + `hermes_cli/people_cmd.py`，按 `task`/`memo` 范式接线进
`main.py`。`show` 的 ref 解析：先试当 id 命中，再 `search_entities` 过滤 person，最后 `list_people`
精确名/别名兜底（2 字 CJK 可能被 token 拆分）。找不到→rc=1 + 提示。

## 不做（诚实边界）

- **不**做 LLM 人物总结 / 关系推断 / 「此人对我意味着什么」合成——那是 Phase 2-B 及以后的洞察
  项，且必须走 C5 门。A5 只摆事实。
- **不**新表、**不**新列（全复用 entities / identity_events / feeling_events / relations）。
- **不**做拼音/模糊人物归并（V5 ADR-052 pinyin 归并 defer，ADR-V6-011 决策6 已记）——按精确名 +
  显式别名匹配。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
|---|---|---|
| 读取面是否真落地 | ✅ | 2 store 方法 + CLI；smoke 实测 list 排序 + show 多面输出 |
| 是否纯 SQL 无 LLM | ✅ | 全 SELECT/GROUP BY；CLI 末尾标注「未经 LLM 合成」 |
| 实体↔事件匹配是否真通 | ✅ | 测试实测别名命中 + R9 entity 匹配 + R1 排除 |
| 是否假装洞察 | ❌ 不假装 | LLM 合成明示 defer Phase 2-B |
