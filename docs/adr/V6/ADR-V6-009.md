# ADR-V6-009：V5→V6 创始人数据迁移策略（JSONL 中间格式 + 双侧工具）

- **状态**：Accepted
- **日期**：2026-07-17
- **关联**：ADR-V6-008（PTG schema）、REV-10（镜像 V5 实际表）、D3（location_log 不迁）、铁律 C1/C2/C6/C7

## 背景

V6 桌面端是 greenfield 重建（fork hermes），数据层为本地 SQLite（`ptg.db`，13 表镜像 V5，见 ADR-V6-008）。创始人 4 年真实数据在 V5 PostgreSQL（阿里云 ECS 生产实例 / `realityos` 库，RLS 租户隔离）。P0-5 须把这批数据搬进 V6，作为桌面端首批个人资产。

约束：
- V6 桌面**不内嵌 Postgres 驱动**（零服务端原则）。
- 铁律 C2（不可丢失）、C6（可重放）、C7（无静默失败）。
- D3：`location_log` 不迁移（founder PG dump 永久备份）。
- 迁移须幂等、可复跑、可审计。

## 决策

### 决策 1：JSONL 作为中间格式（不是 pg_dump SQL，不是 V6 直连 PG）

- **V5 侧 exporter**（`scripts/migrate_v5/export_v5.py`，asyncpg）：每表 `SELECT * ORDER BY id` → 序列化 PG 原生类型为 JSON 原生值（UUID→str、TIMESTAMPTZ→iso8601、DATE→YYYY-MM-DD、NUMERIC→float、JSONB→dict/list、BOOLEAN→bool、bytes→base64）→ 写 `<table>.jsonl` + `manifest.json`。
- **V6 侧 importer**（`plugins/memory/ptg/migrate.py`，纯 Python）：读 JSONL，逐表 `convert_row` + 列映射 → 批量 `INSERT OR IGNORE` 进 `ptg.db`。

**为什么 JSONL**：① 桌面免装 PG 驱动；② 人可读、可 diff、可断点续传；③ JSONL dump 即 D3 所述「永久备份的产物」；④ 序列化值被 importer 转换器幂等处理（exporter 序列化 = importer 转换的逆方向，已 `test_export_then_import_roundtrip_idempotent` 实证）。

### 决策 2：幂等 = INSERT OR IGNORE（首胜）

PK 冲突 → 跳过新行、保留旧行。复跑只加新行，**绝不覆盖、绝不删除**（C2）。重导修正数据 = 删 `ptg.db` 重导（V6 文件重建，非删除既有用户数据；真理之源是 V5 + JSONL）。

### 决策 3：FK 安全导入顺序

`users → memos → identity/meaning/entity/feeling_events → entities → relations → task_suggestions → feedback → insight_aggregation → dlq_messages → llm_call_logs`。SQLite 批量导入期间 FK 强制关闭，顺序保证子表在父表之后，为将来开 FK 留一致基础。

### 决策 4：不迁 location_log（D3）与 memo_embeddings

- `location_log` 独立表（确认：`location.py:22`，lat/lng NUMERIC(10,7)），按 D3 不迁，founder PG dump 永久备份。
- `memo_embeddings`（Vector(512)，HNSW 余弦，BGE-small-zh）是**可重建派生缓存**（无 deleted_at/version），不在 13 表内；V6 在提取阶段重新嵌入。`memos` 表本身无向量列。

### 决策 5：失败 → DLQ，绝不静默（C7）

importer 逐行解析；非法 JSON 行计入 errors、写 `dlq_messages`（source=`v5_migrate:<table>`，original_data 含表名/行号/原文片段）、继续；不整表回滚。导入结束在 `ptg_meta` 写审计行 `v5_migration_<iso8601>`（全表 read/written/skipped/errors）。

### 决策 6（前置修正）：schema v2 fidelity 修正

精读 `danao13/backend/app/models` 后发现 P0-4b 的 V6 schema 与 V5 实际有 4 处偏离「镜像 V5」决策（详见 ADR-V6-008 v2 注 + 本 ADR「发现」节）。**先修 schema 再写 importer**，否则列映射无法 1:1、`feeling_events.ser_source` 旧 CHECK 会拒掉 V5 真值 `ser_audio`/`both`。SCHEMA_VERSION 1→2。

## 发现（schema 精读实证，修正了 P0-4b 的假设）

| 表 | P0-4b 假设 | V5 实际（models + alembic 实证） | 修正 |
|---|---|---|---|
| identity_events | 有 `person_attributes` | 无；实际有 `sentiment`(positive/neutral/negative) + `interaction_type`(meeting/communication/conflict/casual) | 去假列、加真列 |
| feeling_events | `ser_source` CHECK(llm_text/acoustic/fused)；state/direction/intensity 可空 | 无 CHECK（值 llm_text/ser_audio/both）；三字段 NOT NULL | 去 CHECK、改 NOT NULL |
| relations | source_id/target_id/updated_at | subject_id/object_id/last_updated；evidence_count 默认 1 | 改列名 |
| 全局嵌入维度 | 1024 | **512**（BGE-small-zh-v1.5，EMBEDDING_DIM=512） | DEFAULT_EMBEDDING_DIM 1024→512 |
| meaning_events | 无 updated_at | 有可空 updated_at（f8a9b0c1d2e3） | 加列 |

**确认无误**：`feedback.rating` 仅 thumbs_up/thumbs_down（无 1-5 星历史）；`intent_class` 8 值精确匹配；`location_log` 独立；13 表无 partial unique（`feedback.ix_feedback_user_target` 是 plain unique）；3 个 confidence 列是 Float 非 Numeric（task_suggestions/insight_aggregation/llm_call_logs.cost_cny）— 全映 REAL，无影响。

## 铁律符合性

- **C1**：本 ADR + ADR-V6-008 v2 修正。
- **C2**：importer 全程 INSERT OR IGNORE，无 DELETE；append-only 日志（dlq/llm_call_logs）原样拷贝；13 表软删列随数据迁入。
- **C6**：`llm_call_logs` 全字段（prompt_input/response JSON 全量、prompt_template_version、schema_valid、cost_cny、token/latency）随迁，V6 可重放。
- **C7**：非法行→DLQ + 日志，不静默；exporer 单表失败不中断其余表。

## 后果

- 迁移工具链已就绪并测试：importer 11 测试 + exporter 序列化 12 测试 + converters 27 测试全绿；含端到端（合成 V5 JSONL → ptg.db）、幂等、DLQ、批边界、users→realityos_users 改名、relations V5 列名、feeling ser_audio 通过、审计行。
- **真实数据迁移**（读 V5 生产 PG）是 founder 一次性动作：在有 V5 PG 访问的机器上 `python scripts/migrate_v5/export_v5.py --dsn "$DATABASE_URL_ADMIN" --out ./v5dump`，再在 V6 桌面侧导入。exporter 对 V5 只读、幂等，安全。
- 嵌入不迁 → V6 首次启动后语义检索（vec0）需提取阶段重建；Phase 0 关键词召回（trigram FTS + LIKE 兜底，见下「真实数据验证」）立即可用。
- 若将来 V6 要更好的嵌入模型（非 BGE-small-zh 512），另起 ADR，且须处理已迁 512 维数据。

## 真实数据验证（C3 / 2026-07-18）

P0-5a 的 72 个单元测试全用**合成**样本。C3（生产现实即真理）+ ADR-088 教训（合成样本曾掩盖 41% 污染 bug）要求对 founder 真实数据跑一遍。dev 机无 Postgres/Docker/asyncpg、V5 PG 端口不可达，故写**一次性验证工具** `scripts/migrate_v5/_parse_pgdump_copy.py` 解析 7/5 真实 pg_dump（COPY 块）→ JSONL（与 exporter 同格式），再用**真实 importer 不改一行**导入临时 ptg.db（`_validate_real_migration.py` 编排：解析 → 列存在性预检 → 导入 → 对账 → 抽检）。

**结果（founder 7/5 dump，13 表 1611 行）：全表 read==written，0 skipped，0 error。** 抽检：CJK 原文逐字节保真（founder 真实中文备忘原文），relation/llm_call_logs 完整。

**但真实数据暴露了 2 个合成测试漏掉的缺陷（正是 C3 的价值）——均已修 + 加回归测试（C4）：**

1. **CJK 关键词召回全失效（P0）**：`memos_fts` 默认 `unicode61` 分词器按空白切词，中文无空白 → 北京/<真实 3 字人名>/辞职 全 0 命中，而英文「budget」合成测试一直绿。**fix**：schema v3，`memos_fts` 改 `tokenize='trigram'`（≥3 字 CJK 子串召回）+ `search_memos_fts` 加**按 token LIKE OR-join 兜底层**（trigram 匹配不到 <3 字词如「北京/老婆」，LIKE 兜底；并保留多 token OR 语义）。v2 旧库由 `_ensure_fts_trigram` drop+rebuild 升级。修后真实数据：北京→1、<3 字人名>→3、辞职→1。回归测试 `test_search_recalls_2char_cjk` / `test_search_recalls_3char_cjk` / `test_search_cjk_excludes_soft_deleted`。
2. **`is_founder` 生产全 false（P1）**：V5 的 `is_founder` 列对**全部 19 用户都是 f**（含真 founder）——V5 从未置位。忠实迁移带入 is_founder=0，原 `ensure_founder` 仅 INSERT 时置 1，对已存在迁移行不修正 → 任何按 is_founder=1 查询都查不到人。**fix**：`ensure_founder` 改 UPSERT-promote——行已存在且 is_founder=0 时 UPDATE 置 1。回归测试 `test_ensure_founder_promotes_migrated_row`。

**工具链定性**：`_parse_pgdump_copy.py` / `_validate_real_migration.py` 是**一次性验证工具**，非 ADR-009 的规范迁移路径（规范路径仍是 V5 侧 live-PG exporter）。仅在「只有 pg_dump、无 live PG」时复用其思路。验证产出为临时库，已删；founder 真实 V6 资产在其本机部署时按规范路径构建。

**1985 测试全绿（整个 fork）+ ruff . 全净**，确认搜索/ensure_founder 改动零回归。

## 真实数据验证 II（C3+C7+C4 / 2026-07-18 · C7 分类计数 + convert_row DEFAULT 修复）

上一轮验证用旧 `_bulk_insert_ignore`，返回裸 `skipped = read-written-errors` 计数——把「幂等 PK 冲突」（合法）与「NOT NULL/CHECK 约束违例」（静默丢数据）**混算成一个 skipped**（C7 缺口）。本轮两件事：

**1. C7 修复：`_bulk_insert_classified` 区分 pk_conflict vs violation。** 对账批次先查已存在 PK（合法幂等冲突），bulk insert 后再查「非冲突却未落库」的 id——只可能是约束违例→逐行进 DLQ（`error_type=constraint_violation`，全行 `original_data`）。快乐路径零逐行开销（多一次 SELECT 对账）。原 `_bulk_insert_ignore` 已删除。统计 shape 从 `{read,written,skipped,errors}` → `{read,written,pk_conflict,violation,errors}`，不变量 `read == written+pk_conflict+violation+errors`（每行可追溯，无静默吞）。回归测试 `test_import_classifies_pk_conflict_vs_constraint_violation`（3 memo：A 写入/B 重复/PK 冲突/C CHECK 违例→DLQ；两轮）。

**2. convert_row DEFAULT 被击穿修复（真实数据暴露，合成测试掩盖——又是 ADR-088 教训）。** 用 C7 分类计数重跑 founder **6/14 dump**，立刻暴露 **179 行违例（feeling_events 14/14 + entities 165/165，100%）**——旧 skipped 计数会把这 179 行算成一个无意义的 skipped 数，看着像「迁完了」实则丢光。逐行 plain-INSERT 抓真实报错：`NOT NULL constraint failed: feeling_events.ser_source` / `entities.voiceprint_confidence`。**根因**：`ser_source`/`voiceprint_confidence` 是 V6 新增、带 schema DEFAULT 的列（`ser_source DEFAULT 'llm_text'`、`voiceprint_confidence DEFAULT 0.0`），V5 schema **根本没有这些列**；column map 仍把它们映射，`convert_row` 用 `v5_row.get("ser_source")` 拿到 None，V6 行显式带 `ser_source: None`，INSERT 显式写 NULL → **击穿 DEFAULT** → 触发 NOT NULL。schema 给 DEFAULT 本就是为「V5 没这列」兜底——但仅当 INSERT **省略**该列时生效，显式 NULL 反而破坏它。合成 fixture 永远带 `ser_source='ser_audio'`，所以测不出。**fix**：`convert_row` 当 V5 key **不存在**（V5 schema 无此列）时**省略该列**（INSERT 不列名 → SQLite 套 DEFAULT）；V5 key **存在但 NULL**（真可空列该行为空）仍发 NULL——两者区分清楚。同时把 DLQ `original_data` 从截断的 `list(r)[:12]` 改全行 `dict(r)`（C7 可追溯）。回归测试 `test_import_v5_row_lacking_v6_defaulted_columns`（feeling 无 ser_source/emotion_vad、entity 无 voiceprint_* → DEFAULT 生效：ser_source='llm_text'、voiceprint_confidence=0.0、emotion_vad/voiceprint_samples NULL）；并把 `test_convert_row_missing_v5_key_becomes_none`（原**把 bug 写进测试**）改正为 `_is_omitted_so_schema_default_applies` + 补 `_present_but_null_key_still_emits_none`。

**结果（C7 计数 + convert_row 修复后）**：
- **6/14 dump**：13 表 **read 984 = written 984，pk_conflict 0 / violation 0 / error 0**；运行时门绿（PTGProvider 在迁移库上 init/prefetch 召回/sync_turn 捕获/system_prompt_block 报状态全 ✓）；founder 用户 60 memo。
- **6/13 dump 交叉验证**（不同数据、同 schema，证修复**泛化**非单 dump 偶然）：13 表 **read 818 = written 818，0 violation**；运行时门绿；founder 53 memo。

**PII 卫生（ADR-403）**：`_validate_real_migration.py` 加 `--counts-only`——仅对账计数 + dump 无关运行时门（recall key 从真实 memo 前 2 字派生但**不打印**、capture key 自注入非真实数据），跳过打印真实 memo 文本的抽检。真实 founder 内容**零行进 stdout/会话**，可安全自动化复跑。

**552 测试全绿（plugins/memory）+ ruff 全净**，convert_row 调用方仅 migrate.py（运行时无影响）。
