# ADR-V6-010：桌面分发模型 + fork-aware bootstrap（推 fork 到 GitHub + 改指）

- **状态**：Accepted（分发架构定案；P0-6/7b 端到端打通 ✅）；执行进度见文末
- **日期**：2026-07-18
- **Phase**：P0-6/7（mac/win 打包 + 冒烟 Gate）
- **关联**：ADR-V6-008（PTG 数据层 + v4 默认激活）、ADR-V6-009（迁移）、D2（computer-use 默认关）、D4（内测期不签名）

## 背景

P0-5（迁移）+ P0-6/7a（运行时 Gate）已完成：V6 数据层在 founder 真实数据上验证通过（1611 行 0 跳过、运行时召回/捕获绿）。P0-6/7b 要把 V6 打成 mac/win 安装包并端到端冒烟。

**核心阻塞（thin-installer 分叉）**：桌面端是「瘦安装器」架构——Electron 包**不内嵌 Python**，首启时 `apps/desktop/electron/bootstrap-runner.ts` 解析 `scripts/install.sh`，由后者把 agent 克隆进 `$HERMES_HOME/hermes-agent` + 建 venv。问题：**三处上游硬编码 + 一个构建期 commit 戳，全部指向 upstream `NousResearch/hermes-agent`，不是 V6 fork**：

| # | 位置 | 作用 | 当前值 |
|---|---|---|---|
| 1 | `bootstrap-runner.ts:127` | 打包构建首启时从 GitHub raw 拉 install.sh | `raw.githubusercontent.com/NousResearch/hermes-agent/${commit}/scripts/install.sh` |
| 2 | `scripts/install.sh:46-47` | 克隆 agent 仓库 | `REPO_URL_SSH/HTTPS = NousResearch/hermes-agent` |
| 3 | `update-remote.ts:15` | 被动更新检查的官方仓 | `OFFICIAL_REPO_HTTPS_URL = NousResearch/hermes-agent` |
| 戳 | `write-build-stamp.mjs fromLocalGit` | 烤进包的 commit SHA（拉哪个版本） | 构建 HEAD |

后果：**即使打出 mac/win 包，首启也只会拉 upstream hermes，V6 永远到不了用户机器**。bootstrap-runner Priority 1（dev 本地 source）用 fork 自己的 install.sh，故 **dev 安装已正常**；分叉只在打包路径上。

**fork 现状**：仅 `upstream` remote，**无 `origin`**，1 个 commit（0f102fa），全部 V6 工作**未提交**（26 项改动/未跟踪）。

## 决策

### 决策 1：分发模型 = Option A（推 fork 到 GitHub + fork-aware bootstrap）—— 推荐

三选一：

| 选项 | 做法 | 代价 | 结论 |
|---|---|---|---|
| **A（推荐）** | fork 推到 GitHub 私有仓；三处硬编码 + 戳改指 fork；复用现有 thin-installer / venv / pip 基建 | 需用户建私有仓 + 授权 commit/push（**唯一用户侧动作**） | 最轻、零新机制、与 hermes 自身分发一致 |
| B | 把 Python agent + venv 打进 Electron 安装包（离线） | 跨平台 venv 复杂、包体大（torch/funasr 重）、内测过重；违反 ADR-069「重 ML dep 不进基础镜像」精神 | 否 |
| C | 不定分发，仅本地预装 agent 冒烟 | 不交付测试者，P0-6/7 仅完成冒烟 | 兜底，非目标 |

**选 A 的理由**：① 复用全部现有基建（bootstrap-runner / install.sh / venv / 自更新），零新机制；② D4 内测期不签名 + 小规模测试者，thin-installer 的「首启拉 agent」正是为这种场景设计；③ fork-aware 改造面小且可全量回归测试；④ 与 hermes 自身分发模型一致，未来同步 upstream 零阻力。

### 决策 2：fork-aware bootstrap——单一 fork-identity 常量驱动三处

**不**在三处分别散写 fork URL。引入单一来源 `apps/desktop/electron/fork-identity.ts`（或扩 update-remote.ts）导出：

```ts
export const FORK_REPO_HTTPS = 'https://github.com/<owner>/realityos-desktop.git'  // 用户建仓后填
export const FORK_REPO_CANONICAL = 'github.com/<owner>/realityos-desktop'
export const FORK_RAW_BASE = `https://raw.githubusercontent.com/<owner>/realityos-desktop`
```

三处改指（**待 fork 上 GitHub、URL 确定后落地**，非现在盲改安全敏感远程地址）：
1. `bootstrap-runner.ts:127` → `${FORK_RAW_BASE}/${commit}/scripts/${scriptName}`
2. `install.sh:46-47` → fork 的 SSH/HTTPS（或加 `--repo` 入参由 bootstrap 注入，install.sh 默认仍 upstream 以保向上兼容）
3. `update-remote.ts:15-16` → fork canonical/HTTPS（被动更新检查指向 fork release）

构建期 commit 戳（`write-build-stamp.mjs`）无需改逻辑——它 `fromLocalGit` 取当前 HEAD；fork 提交后 HEAD 即 fork commit，首启拉 fork 版本。**关键：戳的 commit 必须已 push 到 fork**（否则 raw 404，bootstrap 已有 fallback 到 installed-agent install.sh 兜底，但分发路径上应保证已 push）。

**落地顺序（用户授权后）**：① 用户建私有仓 `realityos-desktop`；② 授权我 commit + `git remote add origin` + push；③ 我填 fork-identity 常量 + 改指三处 + 跑 TS/集成测试；④ mac `pack`→`dist:mac` 出未签名 dmg；⑤ 端到端冒烟（首启拉 fork agent → PTG 默认激活 → 召回迁移数据）。

### 决策 3：win 构建 = CI runner（非本机 Wine）

本机 macOS 出 mac dmg 直接可行。win(nsis/msi) 在 mac 上需 Wine 或跨编译，脆且重。参照 ADR-084（APK 构建 CI 化）教训：win 包放 GitHub Actions windows runner 出。内测期若测试者仅 mac，可先只出 mac、win 随后。

### 决策 4：分发前打磨清单（不阻塞，打包阶段一并做）

P0-1 换皮后残留的分发可见项（Explore 审计 + 本 ADR 背景核实）：
- `apps/desktop/scripts/test-desktop.mjs` 硬编码 `Hermes.app` 等 binary 名 → 改 `RealityOS`（否则 validateBundle 失败）。
- 图标 `icon.icns` 实为 PNG 误标 → 重新生成合法 .icns。
- ~13 处用户可见 "Hermes" 字符串（about/release-notes URL、onboarding docs 链等）→ 按品牌偏好扫一遍（**用户可能有品牌偏好，扫前确认**）。
- `apps/desktop/build/` 目录当前不存在（图标/资源待打包阶段置入）。

## 铁律符合性

- **C1**：本 ADR。分发模型 + fork-aware bootstrap 是架构决策，必须先记录（已在 push 前落档）。
- **C3**：P0-5 真实数据 Gate 证明数据层可用；本 ADR 追问「打包后是否真活」直接暴露了 ADR-V6-008 v4 的默认激活缺口（已修）。
- **C7**：provider 默认静默缺席 = 无静默失败同类陷阱，已修 + 回归测试锁。
- **D2/D4**：computer-use 默认关 + 内测不签名，在分发模型中继承（打包出的首启流不含 computer-use 自动执行；不签名 = `notarize.mjs` env-absence 自然 off）。

## 后果

- **已解除阻塞**：V6 数据层运行时默认激活（ADR-V6-008 v4）——无论 dev 还是（未来）打包，PTG 在每次启动都加载。这是分发**独立**的 P0-6/7 交付。
- **仍阻塞（用户侧）**：① 建 GitHub 私有仓 `realityos-desktop`；② 授权 commit/push（harness 铁律：仅在用户要求时提交）；③ 确认 win 构建走 CI runner。
- **fork-aware 三处改指 + 端到端冒烟**：待上述①②后落地，预计一个工作段完成。
- **暂不盲改**：三处上游硬编码是安全敏感远程地址，目标 URL（fork 仓）由用户决定，URL 未定前盲改无法测试且可能写错。URL 确定后按决策 2 单一常量驱动，全量回归。
