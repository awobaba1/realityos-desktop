## 决策

### 1. quality_metrics 漂移治愈（`schema.py`，核心代码改动）

把 `_RECONCILE_COLUMNS["quality_metrics"]` 从空填上 legacy V5 库缺的 v4 列：

```python
"quality_metrics": {
    "user_id": "TEXT",                       # legacy 行无；insert 总显式传
    "metric_date": "TEXT",                   # v4：自 `date` 重命名
    "version": "INTEGER NOT NULL DEFAULT 1", # C2
    "deleted_at": "TEXT",                    # C2
    "created_at": "TEXT",                    # SQLite 禁 ADD COLUMN 用 NOT NULL+CURRENT_TIMESTAMP
},
```

`date`→`metric_date` 是**重命名**（非新增），ADD COLUMN 表达不了——加 `_backfill_legacy_data(conn)`：reconcile 加完 `metric_date` 后，`UPDATE quality_metrics SET metric_date=date WHERE metric_date IS NULL AND date IS NOT NULL`，把存活旧行前拷。**旧 `date` 列保留**（C2 nothing-lost），只前拷不销毁。幂等（只填 NULL）。

### 2. 🔥 翻转 apply_schema 顺序（顺手修一个潜伏 bug）

`executescript(_SCHEMA_SQL)` 含引用 reconcile 列的索引：`idx_qm_date_type(metric_date)`、`idx_qm_deleted(deleted_at)`，**还有 `idx_meaning_overdue(is_overdue)`**。原顺序 executescript→reconcile 在 legacy 库上 executescript 会在这些索引处**中途崩**（列还没加），后续表（含 ptg_meta）建不出来。

改为 **reconcile→executescript→backfill→fts**：reconcile 先给已存在的 legacy 表补列，executescript 再建缺失表+全部索引。同时让 `_reconcile_columns` 对不存在的表跳过（`if not existing: continue` + 逐列 try/except），fresh 库 reconcile 全 no-op、executescript 整建。这一翻转**同时治愈** quality_metrics 与 meaning_events(is_overdue) 两族「索引引用 reconcile 列」的潜伏漂移。

### 3. ptg_meta 无需代码改动

已在 `_SCHEMA_SQL`，apply_schema 跑一次即建。本 ADR 不动它。

### 4. run_eval bridge 无需代码改动，补 C4 测试

`_record_report_to_quality_metrics` 已正确写 overall(3) + per-type precision/recall 行（recall=None 跳过）。漂移治好后在 live 库即可写进。补 `tests/benchmark/test_run_eval_quality_metrics_bridge.py`（2 测试）：行数/类型/sample_size/note 契约 + C7 fail-open（坏库不抛）。

### 5. time_trend_daily 诚实降级（无代码改动）

母纲 #9 已定：Phase 1b 不建聚合表（R10/R11 是 Phase 2 特性，建了也无聚合源）。**亲核确认无需改 prompt**：周镜面 prompt 的趋势内容源本就是 R1 feeling_events，不引用 time_trend_daily；R10/R11 原子 v11 不抽。故本批**零代码**，仅本 ADR 如实记录延迟。表本身 Phase 2 起建（live 库的 V5 遗产表保留不动，C2）。

C3 真实库操作（本 ADR 落地后，2026-07-20）：对真实 `/Users/wugang/.realityos/ptg.db`（备份→`backups/ptg.pre-adr027-*.db`）实跑治愈：

- ✅ **reconcile 层在真实 legacy 数据上生效**：quality_metrics 补齐 5 个 v4 列（user_id/metric_date/version/deleted_at/created_at，旧 `date` 保留）；meaning_events 补 atom_kind/is_overdue 等 v6 列。DB `integrity_check=ok`，danao14 数据无损（meaning_events 26/identity_events 24/relations 7，memos 仍 `ts` 契约不破）。
- ⚠️ **full apply_schema 在该库被更深的分歧阻断**（见下「新发现」）：executescript 撞 `memos.ts`↔桌面 `timestamp`，ptg_meta 未能建。**非 quality_metrics 问题**——reconcile 已正确治愈本 ADR 目标列。
- ✅ **quality_metrics 诚实保持 0 行**：不向用户真实时序灌 mock 数据（那才是假绿）；真实行待用户/CI 跑 `run_eval --ptg-db`（批次 2 ADR-030）累积。治愈代码已就位，bridge C4 测试已锁。

### 🔥 新发现（超出 ADR-027 范围，如实标注）

`~/.realityos/ptg.db` 是 **danao14（V5 产品线）维护的库**，其 `memos` 表用 `ts`，桌面 fork `memos` 用 `timestamp`（+ text/summary/audio_clip_id 列集不同）——**重新设计级分歧**，非 additive 可治。后果：桌面 fork 的 `apply_schema` 用此库会 executescript 中途崩，**无法直接打开 danao14 V5 库**。

**裁定**：这是 V5→V6 跨产品迁移问题，**超 Phase 1b 反假绿范围**。桌面 fork 对外分发面向**新用户 fresh 库**（schema 自洽，本 ADR 治愈代码对其完全适用）。开发者本机若要用桌面 fork 读 danao14 历史，需独立迁移工具（Phase 2+，另立 ADR）。本批仅治愈 quality_metrics/ptg_meta/time_trend 三项，不碰 memos 分歧。

## 后果

- **quality_metrics 漂移死锁解开（桌面 fork 库）**：fresh/桌面-legacy 库的 quality_metrics C2 列由 reconcile 治愈，run_eval bridge 可写进，§8 Gate KR 有证据落点（真实行待 run_eval 累积）。
- **迁移健壮性提升**：apply_schema 顺序翻转顺修 meaning_events(is_overdue) 等所有「索引引用 reconcile 列」的潜伏漂移——桌面-legacy 库首启不再中途崩。
- **ptg_meta 自动就位（桌面 fork 库）**：桌面 fork 首启 apply_schema 即建（schema_version 簿记）。
- **time_trend_daily 如实延迟**：不假装有节律聚合；周镜面趋势内容诚实走 R1。Phase 2 起建表。
- **memos V5↔V6 分歧显式标注**：不掩盖桌面 fork 打不开 danao14 库的事实；留给独立迁移 ADR。
