# ADR-V6-066 — C7 漏 DLQ 补齐（5 处 log-yes/DLQ-no 收口）

**状态**: Accepted
**日期**: 2026-07-21
**关联**: [ADR-V6-053](ADR-V6-053.md) 第三轮审计 R4 / 铁律 C7 / [ADR-V6-065](ADR-V6-065.md)（DLQ 读面，本 ADR 的 DLQ 现有消费方）/ [ADR-V6-052](ADR-V6-052.md)（theory per-row DLQ 模式）

## 背景（第三轮审计 R4 发现 5 处 log-yes/DLQ-no）

ADR-V6-053 第三轮审计 R4（铁律 C7）发现 5 处异常「logger.warning yes / DLQ no」——按 CLAUDE.md C7「EVERY exception, error, timeout MUST produce a DLQ entry」口径，这些吞异常路径缺 DLQ 兜底：

| # | 位置 | 现状 | 性质 |
|---|---|---|---|
| 1 | `provider.py` sync_turn 捕获失败 | warn + return | memo §0.6 捕获面静默丢失（`_spawn_atomize` 同类已 DLQ，捕获路径反漏） |
| 2 | `correction.py` re_extract_memo atomize uncaught | warn + return | 与 `provider._spawn_atomize` 不一致（provider DLQ，correction 不） |
| 3 | `store.py` insert_feedback 失败 | warn + return `""` | 创始人校准信号（ADR-V6-028 §11.4）静默蒸发 |
| 4 | `store.py` insert_tool_event 失败 | warn + return | §9#4 操作捕获面 sink 失败无 DLQ |
| 5 | `store.py` `_resolve_task_ref` exact-id 查询失败 | bare `except: pass` | 连 DEBUG log 都无，完全静默 |

C2/C5 干净（0 违规），缺口集中在 C7 的「DLQ 兜底」层。

## 决策

### D1-D2：store 外层补 DLQ（provider.sync_turn + correction.re_extract_memo）

类比 `_spawn_atomize`（`provider.py:381-392`）已验证的 DLQ 模式：`store.insert_dlq` + fail-safe try（DLQ 写自身失败则 warn，observer 永不崩）。

- D1 sync_turn：捕获失败补 `source="ptg.sync_turn", error_type="sync_turn_capture_failed"`。
- D2 re_extract_memo：atomize uncaught 补 `source="correction.re_extract_memo", error_type="atomize_uncaught"`（与 `_spawn_atomize` 一致化，消行为不一致）。

### D3-D4：store 内部 sink 补 best-effort DLQ（insert_feedback + insert_tool_event）

这两处 except 在 `with self._lock` **外**（方法级 try）→ lock 可重 acquire，**无死锁**。补 best-effort `self.insert_dlq`（fail-safe try）。

**诚实边界（best-effort 自举限制）**：store 内部 sink 失败常因 `_conn` 异常状态，此时 `self.insert_dlq`（同 `_conn`）大概率也失败 → fail-safe warning 兜底（已有）。但**非连接级失败**（SQL/schema 逻辑错，`_conn` 仍 OK）时 DLQ 能写。这是诚实的 best-effort，非假绿——明确 fail-safe + 承认连接级失败时 DLQ 也失效。

### D5：`_resolve_task_ref` 补 DEBUG log（非 DLQ）

exact-id 查询失败回退子串匹配是**合法降级**（非数据丢失），补 DEBUG log 即可（非 DLQ——这是查询回退路径，非用户数据写入）。

### D6：C4 测试钉死

`tests/plugins/memory/test_adr_v6_066_dlq_gap_closure.py`：每处触发 except → 断言 DLQ 一行（D1-D4）/ DEBUG log（D5）+ fail-safe（DLQ 写失败不二次抛）。

## 不做（诚实边界）

- **不**强求 store 内部 DLQ 100% 写入（自举限制：`_conn` 坏时同连接 DLQ 也坏）——best-effort + warning 兜底是诚实地板，非假绿。
- **不**把 `_resolve_task_ref` 回退当 DLQ 事件（查询回退≠数据丢失，DEBUG log 足够）。
- **不**改这些方法的「never raises / observer 永不崩」契约——补 DLQ 全程 fail-safe。

## 反假绿

| 项 | 判定 | 依据 |
| --- | --- | --- |
| 5 处是否曾 log-yes/DLQ-no | ✅ | R4 行号实证（test 钉死补前无 DLQ） |
| D3/D4 best-effort 是否假绿 | ❌ 非 | fail-safe try + 明示自举限制；非连接级失败时真写入 |
| 是否破坏 observer 永不崩契约 | ❌ 不破坏 | 全程 fail-safe try |
| `_resolve_task_ref` 是否误升 DLQ | ❌ 否 | D5 DEBUG log（查询回退非数据写入） |
