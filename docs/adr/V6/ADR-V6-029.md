# ADR-V6-029：6 通道验收口径钉死 + 飞书端到端冒烟接缝（桶 D 边界诚实化）

- **状态**：Accepted（Phase 1b 收口·批次 2，2026-07-20）
- **日期**：2026-07-20
- **Phase**：1b
- **关联**：ADR-V6-022（母纲，镜头 10 / 桶 C-C3 / 决策 D2#11 / 验收标准 line 138）、ADR-V6-011（Phase 范围裁定）、ADR-V6-021（飞书作为原子捕获输入源）
- **铁律**：C1（本决策一 ADR）、C3（生产现实即真理——「可用」须区分代码可用 vs 生产可用）、C7（验收口径不静默、不假绿）

## 背景

ADR-V6-022 镜头 10 审计结论：6 个 messenger adapter 全在（亲核），但**零默认启用 + 无验收口径**，导致「6 通道可用」作为 Phase Gate 极易假绿——口头宣称「6 通道 done」无法被任何可运行检查证实或证伪。母纲据此把三件事推迟到本 ADR（line 60 / 98 / 138）：

1. 钉死 6 通道验收口径（推荐：**飞书 1 通道端到端冒烟 = 「可用」最低线）；
2. 桌面加「连接飞书」表单；
3. 飞书端到端冒烟（用户活环境，桶 D）。

本 ADR 把这三件事落到可核验的代码 + 诚实的边界。

## 现状核实（亲核，非自报）

| 维度 | 事实 | 证据锚点 |

| 桌面「连接飞书」表单 | **已存在（数据驱动 catalog）** | `_messaging_platform_catalog()`（`web_server.py:6891`）枚举 Platform + 插件注册；飞书 `_PLATFORM_OVERRIDES` 给 docs_url + required_env（`web_server.py:6624-6635`）；桌面 `apps/desktop/src/app/messaging/index.tsx` 渲染必填/可选字段 + Switch toggle + save/clear + intro 文案（`:564`）+ docs 按钮（`:407`） |

**关键发现**：母纲「桌面加连接飞书表单」的字面诉求**已被现有数据驱动 catalog 满足**——飞书自动出现在桌面 Channels 页，带必填 APP_ID/SECRET、可选 ENCRYPT_KEY/VERIFICATION_TOKEN/DOMAIN/ALLOWED_USERS 等、docs 链接、启用开关。本 ADR 不再造冗余表单。

## 决策

### D1：验收口径钉死为两层 Gate（反假绿核心）

把模糊的「6 通道可用」拆成两层**可核验**口径，杜绝口头断言：

| 层 | 口径 | 如何核验 | 状态 |
|---|---|---|---|
| **L1 代码可用（code-ready）** | 6 adapter 存在 + 接入 relay + 默认禁用 + 凭据 env 自动启用 + 进桌面 catalog + 各有必填字段 | **CI 可运行不变式**：`tests/gateway/test_six_channel_acceptance.py`（无需凭据，纯结构断言） | ✅ 本 ADR 落地 |
| **L2 生产可用（production-ready）** | ≥1 通道**真实**端到端：活 bot 连接 → 入站消息 → agent → 出站回复 round-trip | **桶 D**：需真凭据 + 注册 app + 活 bot；推荐飞书（中国相关性）= 「可用」最低线 | ⏳ 阻塞用户（桶 D） |

L1 测试把「6 通道 done」从口头宣称转成 **CI 强制不变式**——任何 adapter 被删/默认启用被改/catalog 漏通道，CI 立刻红。这正是母纲「口径易假绿」的对症。

### D2：桌面「连接飞书」表单 = 现有 catalog（不造冗余）

经亲核，数据驱动 catalog 已完整提供飞书连接表单（见上表）。本 ADR **不新建** QR 配对向导（whatsapp/telegram 有专门 `/api/messaging/{platform}/onboarding/*` 配对端点，飞书用通用凭据表单）。理由：

1. 飞书 CLI `interactive_setup` 的 QR 是「扫码建 app」（调飞书开放平台 API 注册新应用），**非设备配对**——其真实路径同样依赖活飞书 API（桶 D），桌面向导无法在无凭据下验证；
2. Phase 1b 启用路径 = 用户已有飞书 app → 手填 APP_ID/SECRET → 开关启用 → 重启 → 自动 websocket 连接。该路径**已端到端可用**（catalog + toggle + auto-enable 全在）；
3. 建一个无法真实验证的 QR 向导本身就是假绿风险（写了测绿但真飞书下没跑过）。诚实选择 = 文档化现有表单 + 把 QR 向导列入延迟项。

### D3：飞书端到端冒烟接缝（桶 D 接口预备）

新增 `tests/gateway/test_feishu_live_smoke.py`：
- `@pytest.mark.live_feishu`（marker 本 ADR 注册进 `pyproject.toml`）；
- **无 `FEISHU_APP_ID`/`FEISHU_APP_SECRET` 环境变量时 `pytest.skip`**（skip ≠ pass，诚实信号，CI 永不假绿）；
- 有凭据时跑最小活验：`probe_bot(app_id, app_secret, domain)` 真实调飞书 API 验证凭据 + bot 可达；
- 模块 docstring 写完整 e2e runbook（websocket 连接 → 入站 → agent → 出站 round-trip），供用户给凭据后一键 `-m live_feishu` 全量冒烟。

这是母纲 line 119「代码侧先备好 telemetry 入口」在通道维度的落地——桶 D 的代码侧接缝备好，等用户凭据即填。

### D4：6 通道叙事澄清（并入母纲 P2#18）

母纲 §2.1/§9 待修叙事「6 messenger 内置生产级」应澄清为：**6 个核心 messenger（telegram/discord/slack/whatsapp/feishu/dingtalk）均为内置 Platform enum 成员 + 内置 adapter**（非纯插件）；仓库另有 ~14 个额外平台（wecom/matrix/email/irc/teams/ntfy/…）以插件形式存在。即「6 built-in」对核心集成立，叙事修正落在「生产级」三字——生产级须过 L2（桶 D），非代码内置即生产级。文档对齐归 P2（母纲 #18）。

## 验证

- **L1 不变式**：`tests/gateway/test_six_channel_acceptance.py` 覆盖——6 通道在 Platform enum / 6 adapter 模块存在 / `PlatformConfig().enabled is False` / catalog 含全部 6 且各有必填字段 / 飞书 catalog 项 docs_url+required_env 齐全（证「连接飞书表单」存在）。
- **桶 D 接缝**：`tests/gateway/test_feishu_live_smoke.py` 无凭据 skip、有凭据 `probe_bot` 活验；marker 注册。
- **过时 docstring 修复**：`tests/gateway/test_setup_feishu.py` 3 处引用已搬走的 `_setup_feishu() in hermes_cli/gateway.py` → 改指 `plugins.platforms.feishu.adapter::interactive_setup`（函数在 #41111 已搬迁，docstring 未跟进）。
- **三连核验**：headSha == CI 绿（`gh run view --json conclusion`）+ 内容指纹 + 真实非 skip。

## 后果

- 「6 通道可用」Gate 从此有**可运行 L1 不变式**（CI 守门）+ **诚实 L2 桶 D 边界**（不伪装生产可用）。
- 反假绿：任何人再宣称「6 通道 done」须指 L1 测试绿；宣称「生产可用」须指 L2 真实冒烟记录。二者不可混。
- 不增冗余 UI 代码（D2），避免「为做而做」的假绿式堆砌。

## 延迟项（诚实）

- **L2 真飞书端到端冒烟**：桶 D，需用户提供真 `FEISHU_APP_ID`/`SECRET` + 注册 app + 活 bot。给到后跑 `pytest -m live_feishu`（probe_bot）+ 手工跑 docstring runbook（websocket round-trip）。
- **其余 5 通道真实冒烟**：每通道需各自真凭据，桶 D，按需逐通道补。
- **桌面飞书 QR 配对向导**：D2 已述，Phase 1b 手动表单足够；QR 向导（镜像 CLI `interactive_setup`/`qr_register`）若要做须与 L2 同批真实验证，否则不入。
