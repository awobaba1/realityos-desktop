# ADR-V6-026：精度诚实降级 + insight 缓存键 prompt_version 失效（反假绿桶 D + 缓存谎）

- **状态**：Accepted（批次 1，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（反假绿收口）
- **关联**：ADR-V6-022（母纲，镜头 5）、ADR-V6-012（⑦ precision 结构天花板，commit 7758542e0）、ADR-V6-017/018/019（insight 报告生成/调度/缓存）、ADR-V6-025（同 commit 落地）
- **铁律**：C1（本 ADR）、C3（生产现实——审计暴露「门禁横幅误导 + 缓存文档撒谎」）、C4（两缺陷各配回归测试）、C6（prompt 版本化——缓存须随版本失效）

---

## 背景

ADR-V6-022 镜头 5 揭露两条假绿：

### D. 评测门禁横幅误导（precision 假绿）

`tests/benchmark/run_eval.py` 的门禁横幅打印 `GATE: ALL GREEN ✅`，而 `all_green` **只检查 recall 各类型达标**——precision 是**故意不门禁**的（⑦ precision 天花板 ~70% 是结构性的，commit 7758542e0 / ADR-V6-012 实证，非回归）。后果：precision 实测 ~63% 时，横幅读起来像「提取已全面验证」，实际只有 recall 过了门。这是一条**安抚性的假绿标签**，掩盖了一个被延迟的指标。

### 缓存键 prompt_version 谎（insight 报告新旧打架）

`weekly_mirror_v1.md` / `daily_report_v1.md` 模板头注释自称「缓存键含 prompt_version → 改模板自动失效旧镜面」——但 `insight_aggregation` 的唯一键 `uq_insight_user_type_period (user_id, aggregation_type, period_key)` **不含 prompt_version**。而调度探针 `scheduling._run_one` 见任意已存在行就 skip（`reason: exists`）。后果：prompt 升版（v1→v2）后，**旧 v1 报告留在缓存里供到 TTL 过期**——新旧报告打架，C6「prompt 版本化」承诺在缓存层落空。模板文档的「缓存键含 prompt_version」是**直接撒谎**。

## 决策

### 1. 门禁横幅诚实降级（recall green ≠ 全面绿）

把横幅构造抽成纯函数 `_gate_banner(all_green, precision, gates, errors)`：

- **recall 达标 + 0 error** → `GATE: RECALL GREEN ✅ / PRECISION DEFERRED (⑦ ~{precision:.0%} structural ceiling — ADR-V6-012, ungated)`——明示 precision 被延迟、且把实测 precision 值打出来（不再用「ALL GREEN」安抚）。
- **recall 未达标或有 error** → `GATE: RED ❌ — do NOT ship (recall missed: [...]; errors=N)`——列出未达标的类型。

JSON 报告同步加 `gate_recall_green`（语义清晰别名）+ `gate_precision_status: "deferred_structural"` + `gate_precision_note`（解释 precision 不门禁的来龙去脉），让下游工具无法把「recall 过」误读成「全面过」。旧字段 `gate_all_green` 保留（向后兼容，语义=recall green）。

### 2. insight 缓存 prompt_version 失效（探针版本失配→重生成）

**真相先纠正**：`insight_aggregation.schema_version` 列**早就存了 prompt_version**（`_base.py:261 schema_version=self.PROMPT_VERSION`）——prompt_version 并非没存，只是**调度探针没校验它**。故无需加新列/迁移（SCHEMA_VERSION 不变=6），只需：

1. `store.get_insight` 的 SELECT + 返回 dict 补 `schema_version`（探针要读它）。
2. `scheduling._run_one` 探针：`existing` 存在 **且** `existing.schema_version == svc_cls.PROMPT_VERSION` 才 skip；**失配则视为 stale，fall through 重生成**（`reason: "stale_prompt"`）。重生成走 `upsert_insight` 的 ON CONFLICT 原地替换（version+1），刷新该行的 schema_version + 内容。
3. 纠正两模板头注释：唯一键**不含** prompt_version；刷新靠**探针版本失配重生成**，下次启动时生效（startup-lazy 调度哲学，ADR-V6-019）。

冷启动占位行同样受此校验：占位也带 schema_version，旧版占位会被重生成（仍走 insufficient 门→不调 LLM，零成本刷新）。

## 验证（C3 + C4）

| 测试 | 覆盖 |
|---|---|
| `tests/benchmark/test_gate_banner.py`（7） | recall-green 不说 ALL GREEN + 精度值必打 + recall 未达标列类型 + error 计数 + 精度参数化 |
| `tests/plugins/realityos_insights/test_scheduling.py` +3 | `get_insight` 返回 schema_version；stale(v0)行触发重生成(reason=stale_prompt, 1 LLM call, 内容刷新)；重生成后二次 skip(收敛不无限重跑) |

回归：insights 全量 45 绿（42 旧 + 3 新）；ruff 全净。代码与 ADR-V6-025 守卫纠正 #2 同 commit 落地（dc24e326a）。

## 后果

- **门禁不再假绿**：recall 过门如实标「RECALL GREEN / PRECISION DEFERRED」，并打出实测 precision——读者看到真数字，不被「ALL GREEN」安抚。precision 仍是结构延迟（⑦），治法是重标 samples / dedup（ADR-049），非门禁。
- **缓存随 prompt 版本失效**：升 prompt（v1→v2）后，下次启动调度探针发现失配即重生成，旧报告不再供到 TTL。C6「prompt 版本化」在缓存层兑现。
- **文档不再撒谎**：两模板头注释如实描述机制（探针版本失配重生成，非键失效）。

## 延迟项

- precision ⑦ 结构天花板的真治法（重标 samples / ADR-049 dedup / per-type 阈值）——Phase 1b+，本 ADR 只把门禁横幅改诚实，不改精度本身。
- insight 读 API（`/api/insights/...`）cache-first 路径不主动查版本失配（依赖下次启动调度刷新）——桌面日启场景可接受；如需瞬时刷新可在读 API 加版本检查（Phase 2 视需要）。
