# ADR-V6-054 — windows subprocess race 全量硬化（9 处裸断言 → `_spawns`，闭合 1b2065c11 残留）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-052](ADR-V6-052.md)（同批收口）/ commit `0ea318a7d`（首次引入 `_spawns` 硬化 6 处）/ commit `1b2065c11`（补硬化第 7 处 checkpoint_manager）/ 铁律 C4（缺陷→回归用例）/ 五份审计 A5 P1

## 背景（A5 审计发现的同源 flaky 残留）

`tests/test_windows_subprocess_no_window_flags.py` 共 16 个测试，断言「Windows 下每个 console spawn 都带 `creationflags=CREATE_NO_WINDOW`」。这些测试 patch `<module>.subprocess.run`，而 `subprocess` 是**进程级单例**——patch 是进程级全局的。

`hermes_cli/banner.py` 启动 update-check **daemon 线程**（5s 超时跑 `git ... origin`）；`tui_gateway/server.py` 在 import 时触发 `prefetch_update_check()`。任一测试 import `tui_gateway.server`（如 `test_tui_gateway_fuzzy_file_listing_hides_git_windows` line 73）后，daemon 在该 pytest 进程内长存，其 stray `git` spawn 随时可能落进 `captured`，把 `captured[0]` 替换成它，触发 `KeyError: 'creationflags'` 或 argv/计数不匹配。

**`_spawns(captured, *needles)` helper**（line 18）按 argv token 过滤，把断言限定在「本测试独有的调用」，免疫 daemon 串扰。`0ea318a7d` 用它硬化了 6 处，`1b2065c11` 补了第 7 处（checkpoint_manager，C4 收口 61a3d3543 的 slice 6 flake）。

**A5 审计（2026-07-20）发现：仍有 9 处兄弟测试带裸 `captured[0][...]` 断言未硬化**——同源竞态，只是 CI 还没踩到。这是已知的 flaky 源，迟早红。

| file:line | 测试 | 裸断言 | 本测试独有 token（daemon 不撞） |
| --- | --- | --- | --- |
| :111 | `test_coding_context_git_hides_git_windows` | `captured[0][1]["creationflags"]` | `status`, `--short` |
| :171-172 | `test_copilot_gh_cli_probe_hides_gh_windows` | `captured[0][0]` + `["creationflags"]` | `gh`, `auth`, `token` |
| :230 | `test_stale_dashboard_windows_scan_hides_wmic` | `captured[0][1]["creationflags"]` | `wmic` |
| :283 | `test_shell_hooks_hide_hook_command_windows` | `captured[0][1]["creationflags"]` | `hook-bin`, `--flag` |
| :300-301 | `test_inline_skill_shell_hides_bash_window` | `captured[0][0]` + `["creationflags"]` | `bash`, `-c`, `echo ok` |
| :319-320 | `test_tts_opus_conversion_hides_ffmpeg_window` | `captured[0][0][0]` + `["creationflags"]` | `ffmpeg` |

daemon 只 spawn `git`（含 `origin`），上表 token 均不含 `git`/`origin`，故 `_spawns` 过滤后 daemon 串扰被排除。两处 `ffmpeg`、两处 `gh auth token` 虽 token 相同，但分属不同测试函数、各自 `captured` 列表独立，无跨测试污染。

## 决策

### D1：9 处统一改 `_spawns(captured, *needles)` 模式

每处把 `captured[0][...]` 替换为「过滤 → 计数断言 → 取首项断言」三段：

```python
xxx_calls = _spawns(captured, <needle1>, <needle2>)
assert len(xxx_calls) == 1, captured          # daemon 串扰若混入 → 计数暴露
assert xxx_calls[0][...].get("creationflags") == _CREATE_NO_WINDOW  # .get 防 KeyError
```

保留原断言的额外契约（如 `gh_calls[0][0] == ["gh", "auth", "token"]`、`ffmpeg_calls[0][0][0] == "ffmpeg"`），不削弱覆盖。

### D2：`.get` 而非 `[]` 取 `creationflags`

daemon 的 stray `git` spawn **不带** `creationflags`（它走 `text=True, timeout=5` 的另一条路径）。即便过滤后意外残留，`["creationflags"]` 裸取会 `KeyError`（正是 61a3d3543 slice 6 红的根因），`.get` 返回 `None` → 断言失败信息更清晰（`None != 0x08000000`），便于诊断。

### D3：不引入新 helper / 不改测试结构

`_spawns` 已在 line 18-35（0ea318a7d），复用即可。不抽公共 fixture、不改 `fake_run` 签名、不动 daemon 本身（daemon 的治理是 ADR-V6-055 eval-nightly 之外的话题；本 ADR 只闭合测试侧竞态）。

## 不做（诚实边界）

- **不**改 `banner.py`/`server.py` 的 daemon 行为——daemon 是产品功能（更新检查），测试侧用 `_spawns` 隔离即可，治本不在此 ADR 范围。
- **不**把这 9 处合并成参数化测试——它们 import 的模块、patch 的对象、断言的契约各不同，参数化会掩盖差异、降低可读性。
- **不**声称「修了产品 bug」——这是**测试 flaky 的硬化**（C4 回归守卫），产品代码 `creationflags` 传递本就正确，是测试断言方式脆弱。

## 验证

- `ruff check tests/test_windows_subprocess_no_window_flags.py` 净。
