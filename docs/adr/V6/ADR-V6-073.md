# ADR-V6-073 — `hermes dlq --resolve-all`：dlq_resolve_all 的 CLI 消费方

**状态**: Accepted
**日期**: 2026-07-21
**关联**: 第五轮 P2 收口 / [ADR-V6-065](ADR-V6-065.md)（DLQ 读面）/ [ADR-V6-037](ADR-V6-037.md)（做了没发）/ [C7](../../../CLAUDE.md)（无静默失败）

## 背景（P2 backlog：dlq_resolve_all 有生产者无消费方）

ADR-V6-065 落地 C7 DLQ 读面时，给 `PTGStore` 增了两个 ack 原语：

- `dlq_resolve(dlq_id) -> bool` —— 单条 ack（已有 `--resolve <id>` CLI 消费方）
- `dlq_resolve_all(*, user_id=None, source=None) -> int` —— **批量 ack**（UPDATE resolved=1，返回翻纽数）

第四轮审计只关了单条的消费方；**批量那一个没有 CLI/UI/API 任一消费方** —— 是 ADR-V6-037「做了没发」最致命假绿类的半个尾巴：生产者在、测试在、但零调用方。第四轮 P2 backlog 明确列出「`dlq_resolve_all` CLI 未暴露 `--resolve-all`」。

## 决策：加 `--resolve-all [--source]` 作唯一消费方

### D1：CLI 接线（4 点，薄委托）

`hermes_cli/dlq_cmd.py::cmd_dlq` 在 `--resolve <id>` 分支之后插 `--resolve-all` 分支：

```python
if getattr(args, "resolve_all", False):
    # ADR-V6-073: bulk-resolve consumer for dlq_resolve_all
    count = store.dlq_resolve_all(source=getattr(args, "source", None))
    ...
```

`hermes_cli/subcommands/dlq.py::build_dlq_parser` 加：

```python
p.add_argument(
    "--resolve-all", dest="resolve_all", action="store_true",
    help="批量标记所有未解决 DLQ 为 resolved（ADR-V6-073；dlq_resolve_all "
         "的消费方）。可选 --source 限定来源。与 --resolve 互斥（--resolve 优先）。")
```

无需改 `main.py` 的 `_BUILTIN_SUBCOMMANDS` —— `dlq` 子命令早已注册，`--resolve-all` 是其内参数。

### D2：与 `--resolve` 的互斥语义

两 flag 同时给时，**单条 `--resolve` 优先**（先命中分支先返回）。这是文档化的确定性顺序，不抛错（CLI 风格：flag 互斥用先到先得，不强制 fail）。

### D3：诚实空状态

`count == 0` 时打印「无可解决记录」+ 原因（空表 / `--source` 不匹配），**不**打印「✅ 0 条已解决」（那会把「什么都没干」伪装成成功 —— wrap-lucky-green 同型假绿）。只有真正翻了 ≥1 行才打 ✅。

### D4：JSON 输出契约

`--resolve-all --json` 输出 `{"resolved_all": N, "source": "atomize"|null, "stats": {...}}`，机器可读，`stats` 复用 `dlq_stats()`（与单条 resolve / list 同口径）。

## 合规性

- **C7（无静默失败）**：批量 ack 把「积压清零」从手工逐条升级为一键，但仍是创始人显式操作（非自动静默消化）；失败行本身永远不删（append-only，C2）。
- **C2（不可丢失）**：`resolved` 是状态元数据翻转，**失败 payload 从不 mutate**（ADR-V6-065 D3 同律）。
- **架构 §4.7**：单向数据流，store 开 → ack → 关，薄委托同 `citation_cmd`/`k_cmd`。

## 取舍：为何不自动「resolve-all」

批量 ack 必须**显式触发**（CLI flag），不进调度器自动跑。原因：DLQ 行是失败现场证据，自动消化会让「曾经失败过」的事实从可观测面消失，违背 C7「无静默失败」的可观测意图。积压是信号，不是噪音；创始人看到信号后显式 ack，是正确闭环。
