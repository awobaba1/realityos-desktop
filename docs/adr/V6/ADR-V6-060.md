# ADR-V6-060 — ASCII 兜底清洗 in-band 可见化（`_strip_non_ascii` 内容用 `errors="replace"`）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-053](ADR-V6-053.md) P1-T9 / 五份审计 T6 安全 / 铁律 C2「不可丢失」精神 / 诚实降级铁律 / [ADR-V6-050](ADR-V6-050.md) honest-degradation 同源

## 背景（审计发现 in-band 静默吞字）

`agent/message_sanitization.py::_strip_non_ascii` 是 **ASCII-only 系统编码兜底清洗器**——仅当 `locale.getpreferredencoding()` 为 ASCII（`LANG=C` / Chromebook / 精简容器）且 API 调用抛 `UnicodeEncodeError` 时，`conversation_loop.py` 的恢复块才触发它，把消息/工具/系统提示/凭据里的非 ASCII 字符清洗成纯 ASCII 再重试。

审计发现：该函数用 `text.encode('ascii', errors='ignore').decode('ascii')`——**非 ASCII 字符静默蒸发**。对本项目（**中文优先**产品）这意味着：在 ASCII 系统上，消息内容里的中文被**原地无声删除**，agent 在被「打洞」的文本上推理，却看不到洞。

这是 **in-band 静默假绿**：`conversation_loop.py:~2542` 有 console 警告（`stripped non-ASCII ... Retrying`），但警告只给操作员；**agent 看到的内容本身**没有任何痕迹——它把残缺文本当完整事实。违反 C2「不可丢失」精神与诚实降级铁律（降级必须可见，不让消费者把降级数据当真实数据）。

注：`conversation_loop.py:~2542` 已有警告，本 ADR 不重复加，仅**强化措辞**说明内容已用 `?` 标记（见 D3）。

## 决策

### D1：`_strip_non_ascii` 加 `errors` 参数（默认 `ignore`，向后兼容）

```python
def _strip_non_ascii(text: str, *, errors: str = "ignore") -> str:
    return text.encode('ascii', errors=errors).decode('ascii')
```

默认 `ignore` **不变**——所有既有调用方零影响、既有测试零回归。

### D2：仅 agent 推理面用 `errors="replace"`（外科式，非全局）

| 调用点 | 字段 | `errors` | 理由 |
| --- | --- | --- | --- |
| `_sanitize_messages_non_ascii` content string | 消息正文 | **`replace`** | agent 推理核心载体 |
| `_sanitize_messages_non_ascii` content-list text | 多模态文本段 | **`replace`** | 同上 |
| `_sanitize_messages_non_ascii` tool_calls fn_args | 工具调用参数 | **`replace`** | agent 选择的行动数据 |
| `_sanitize_messages_non_ascii` name | 工具名 | `ignore`（默认） | 结构字段，非推理面 |
| `_sanitize_messages_non_ascii` extra fields | reasoning_content 等 | `ignore`（默认） | 辅助字段 |
| `conversation_loop` active_system_prompt | 系统提示 | `ignore`（默认） | 指令层，非用户内容（边界见下） |
| `conversation_loop` ephemeral_system_prompt | 临时系统提示 | `ignore`（默认） | 同上 |
| `conversation_loop` api_key | **凭据** | `ignore`（默认） | **凭据安全，见 D4** |
| `_sanitize_structure_non_ascii` | 工具 schema / headers | `ignore`（默认） | 结构，非推理面 |

`replace` 把每个非 ASCII 字符变成可见 `?`：`"你好"` → `"??"`，`"hello ⚕"` → `"hello ?"`。agent 看到 `?` 标记**就知道这里有降级**，不会把残缺当完整。

### D3：强化 `conversation_loop` 警告措辞（反映 `?` 标记）

原：`System encoding is ASCII — stripped non-ASCII characters from request payload. Retrying...`

新（双语 + 明示标记非静默）：`sanitized non-ASCII from request payload (message content replaced with visible '?' markers, NOT silently dropped — ADR-V6-060). 系统为 ASCII 编码，已用 '?' 标记替换非 ASCII 内容（中文等），非静默丢失。Retrying...`

### D4：凭据为何保持 `ignore`（不做全局 replace 的反假绿理由）

审计原文建议「`_strip_non_ascii` 用 `errors='replace'`」（line 314）。**全局改 `replace` 会引入凭据认证回归**：

- API key 误粘贴场景：用户从 dashboard 复制时混入一个形近字（如 `ʋ` U+028B 代替 `v`），或夹了一个多余字符。
- `ignore`（删掉坏字符）：`sk-abcʋdef` → `sk-abcdef`——**若坏字符是多余的，key 可被恢复，认证成功**。
- `replace`（换成 `?`）：`sk-abcʋdef` → `sk-abc?def`——**`?` 必坏，无法恢复**。

两者在「坏字符替代了正确字符」时都失败（需重新粘贴，警告已提示），但在「坏字符是多余的」时 `ignore` 能救场、`replace` 必杀。因此凭据路径**必须**保持 `ignore`。

本 ADR 的审计**目标**（让中文内容损失 in-band 可见）由 D2 的外科范围**精确达成**，且不引入 D4 的凭据回归。这是比「字面全局 replace」更正确的工程选择。

## 不做（诚实边界）

- **不**做 turn-end rollback（快照/还原 canonical `messages`）。审计 T9 提及「回滚」，但分析后判定**不做**，理由：
  1. 该恢复块**原地改写** canonical `messages` 是**有意**的——还原原文会在下一 turn 重新引入同样的非 ASCII，**再次触发 `UnicodeEncodeError`**，陷入死循环。
  2. 安全的 rollback（每 turn 快照→清洗 wire-copy `api_messages`→turn 末还原 canonical，且保留 turn 内新增消息）是对恢复循环的**非平凡重构**，有真实的 re-trigger / 持久化风险。
  3. **该路径仅 ASCII-only 系统触发**（`LANG=C`）；生产（UTF-8）永不触发。按 C3「生产现实即真理」，为生产永不触发的路径做高风险重构是**误配投入**。
  4. D2 的 `?` 标记**已达成**反假绿目标（降级 in-band 可见）。rollback 的边际收益（在 ASCII 系统的 session DB 里保留中文）只作用于非生产边缘场景。
- **不**把 system prompt 纳入 `replace`——指令层非用户内容，且改它超出本 ADR 的外科范围；若未来审计要求，是一行 follow-up。
- **不**声称「修了 ASCII 系统的中文支持」——ASCII 系统本就无法发送中文；本 ADR 只让**损失可见**，非让中文可用。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
| --- | --- | --- |
| 内容损失是否 in-band 可见 | ✅ | `replace` → `?` 标记，agent 能看到降级（D2） |
| 凭据路径是否回归 | ❌ 未回归 | 默认 `ignore` 不变；`test_default_ignore_is_credential_safe` 钉死 |
| 是否全局改 `replace` | ❌ 不全局 | D4：全局会引入凭据认证回归；外科范围更正确 |
| rollback 为何不做 | ✅ 诚实边界 | 3 条理由（re-trigger 死循环 / 重构风险 / C3 非生产路径） |
| 是否声称修了 ASCII 中文支持 | ❌ 不声称 | ASCII 系统本就发不出中文；本 ADR 只让损失可见 |
| 既有凭据/结构测试是否零回归 | ✅ | 全部不变通过 |
