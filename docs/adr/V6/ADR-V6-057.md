# ADR-V6-057 — theory engine PC/FR per-row 失败补 DLQ（C7 一致性 / 延伸 ADR-V6-052）

**状态**: Accepted
**日期**: 2026-07-20
**关联**: [ADR-V6-052](ADR-V6-052.md)（atomize 外层 wrapper DLQ，同主题）/ [ADR-V6-050](ADR-V6-050.md)（theory engine）/ [ADR-V6-053](ADR-V6-053.md) P1-T4 / 五份审计 A2 P1 / 铁律 C7 / [ADR-V6-011](ADR-V6-011.md) 决策 5「任何异常 → DLQ + log」

## 背景（A2 审计发现 per-row DLQ 不一致）

`plugins/realityos_theory/engine.py::_build_derivations` 把 LLM 输出的 7 PC + 5 FR 维度逐行构造成 `TheoryDerivation`（pydantic gate）。A2 铁律审计（2026-07-20）发现 C7 不一致：

| 失败层 | 既有处置 | 是否达标 |
| --- | --- | --- |
| LLM 调用失败（`engine.py:145-152`） | `_safe_dlq(theory_error)` + warn | ✅ 已闭环 |
| C5 schema 无效（`engine.py:168-171`） | `_safe_dlq(theory_schema_invalid)` + warn | ✅ 已闭环 |
| **PC per-row 构造失败**（`engine.py:217-218`） | **仅 `logger.warning`，无 DLQ** | ❌ 缺口 |
| **FR per-row 构造失败**（`engine.py:229-230`） | **仅 `logger.warning`，无 DLQ** | ❌ 缺口 |

上层（LLM/C5）失败落 DLQ，下层（per-row 构造）失败只 warn——同一个 engine、同一个 C7 契约，两条路径处置不一致。per-row 失败虽 per-row isolation（continue，不炸整批），但「一个维度构造失败」是真实异常（通常是 LLM 输出畸形或 pydantic gate 拒），**应有 DLQ 审计行**便于诊断 + 留痕，否则是 C7 的边界静默变体（warn 可观测但缺审计）。

这与 ADR-V6-052 的 atomize 外层 wrapper 缺 DLQ 同主题——「逃逸出主路径处理的异常」都该有 DLQ 兜底，而非只 warn。

## 决策

### D1：PC/FR per-row except 补 `_safe_dlq`

在 `_build_derivations` 的两个 per-row `except Exception` 分支，warn 后追加 `self._safe_dlq(...)`（engine 既有的 C6/C7 helper，`engine.py:235`）。复用既有 helper，零新方法、零新依赖。`_safe_dlq` 自身 fail-safe（`engine.py:241-242` 嵌套 try/except，DLQ 写失败只 debug-log），不会让 per-row 失败升级为整批崩溃。

### D2：error_type 命名（区分 per-row 与上层）

- PC per-row：`f"{ENGINE}_pc_build_failed"`（即 `theory_pc_build_failed`）
- FR per-row：`f"{ENGINE}_fr_build_failed"`（即 `theory_fr_build_failed`）

与上层 `theory_error` / `theory_schema_invalid` 并列，运维可一眼区分「LLM 整调用炸」「schema 整体拒」「单个维度构造炸」。`ENGINE` 常量（`engine.py:46`）复用，与现有 DLQ source 一致。

### D3：original_data 含 dim + entry（便于诊断）

per-row 失败的诊断价值在「哪个维度 + 什么 entry 触发」。`original_data={"kind", "dim", "entry"}`：`entry` 是该维度的 LLM 原始输出片段（`pc_raw.get(dim)` / `fr_raw.get(dim)`），单个维度、体量小。DLQ 是设计的 PII 汇（ADR-403），非 stdout/log，放 entry 合规。

## 不做（诚实边界）

- **不**改 per-row isolation 逻辑——失败维度仍 `continue` 跳过（不炸整批），只是补 DLQ 审计。isolation 是对的，本 ADR 只补审计。
- **不**对失败维度重试/重抽取——per-row 构造失败通常是 LLM 输出畸形，重试同一输出无意义；DLQ + warn 留痕，人工/后续诊断。
- **不**把 `_safe_dlq` 抽公共——它是 engine 的 C6/C7 helper，与 `_log_call` 同一组；atomize 的 DLQ 在 PTG provider 另有路径（ADR-V6-052）。两处独立，不强合并。

## 验证

- `tests/plugins/test_adr_v6_050_theory.py` 新增 `test_per_row_build_failure_writes_dlq`（C4）：
  - monkeypatch `engine.TheoryDerivation` 让 PC=Energy + FR=Career 构造抛 `ValueError`；
  - 断言：其余 10 维仍构建（per-row isolation），Energy/Career 被跳过；
  - 断言：`dlq_messages` 有 `theory_pc_build_failed` + `theory_fr_build_failed` 各 1 行。
- 本地 `bash scripts/run_tests.sh tests/plugins/test_adr_v6_050_theory.py`：全绿；ruff 净。

## 反假绿（本 ADR 自身的诚实性）

| 项 | 判定 | 依据 |
| --- | --- | --- |
| per-row 是否原 warn-only | ✅ 是缺口 | A2 审计 + Read engine.py:217-218/229-230 确认无 `_safe_dlq` |
| 补的是否 fail-safe | ✅ | 复用 `_safe_dlq`（自身嵌套 try，DLQ 写失败只 debug） |
| 是否削弱 per-row isolation | ❌ 不削弱 | 仍 continue；测试断言 10 维仍构建 |
| 是否扩大 API 面 | ❌ 不扩大 | 复用既有 `_safe_dlq` + `ENGINE`，零新方法 |
| 是否声称修了 LLM 输出 bug | ❌ 不声称 | per-row 失败是症状，DLQ 让它可观测，非「修了畸形输出」 |
