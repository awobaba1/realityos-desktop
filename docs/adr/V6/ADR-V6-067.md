# ADR-V6-067 — purge_soft_deleted 可达化 + nightly-cron 注释诚实化

**状态**: Accepted
**日期**: 2026-07-21
**关联**: [ADR-V6-053](ADR-V6-053.md) 第三轮审计 / 铁律 C2 / [ADR-V6-037](ADR-V6-037.md)（做了没发——最致命假绿）/ [ADR-V6-045](ADR-V6-045.md)（deletion_log / cascade_soft_delete）

## 背景（第三轮审计发现：孤儿函数 + 文档假绿）

`purge_soft_deleted`（sovereignty.py:185）是 V6 **唯一合法硬删面**（§6.2 阶段2，C2 例外）。但第三轮审计发现两个叠加问题：

### 问题 1：文档假绿——注释谎称 nightly cron 存在

| 位置 | 原文 | 事实 |
|---|---|---|
| `sovereignty.py:197` | "the §6.2 nightly cron (Phase 1+) calls this" | 无任何调度调用 cron |
| `web_server.py:12356` | "purge (purge_soft_deleted) is a separate opt-in nightly cron" | 无 scheduler 接线 |

全仓 grep `purge_soft_deleted`：**仅测试调用**（test_realityos_sovereignty.py / test_adr_v6_045_deletion_log.py），**零生产调用方**。web_server 主权接线块（12344-12357）明确把 cascade_soft_delete / export 接到 UI，却**显式排除 purge**（"never a UI one-click"）。结果是：软删行**永远**不会被物理清除（宽限窗口形同虚设），但注释告诉维护者「有 nightly cron 在跑」。

### 问题 2：ADR-037「做了没发」——孤儿函数

一个有测试、有文档、声称「opt-in nightly cron 调用」的函数，在生产中**零调用方**。这正是 ADR-V6-037 标定的最致命假绿类：producer 写了，no consumer 读。对 purge 而言更严重——它是 C7 Phase-Gate「软删行物理清除」KR 的唯一实现，却不可达。

## 决策

### D1：加 `hermes purge` CLI——purge 唯一生产调用方

新增 `hermes purge` 子命令（sole production caller），让「opt-in purge」从注释承诺变成**字面真实**：
- **dry-run 是默认**：`hermes purge` 预览符合宽限窗口的行数（per-table 计数），**不删除**。
- `--confirm` 才执行硬删（C2 唯一例外，安全闸）。
- `--older-than-days N`：宽限窗口（CLI 默认 30 天，保守；primitive 自带默认 1 天仅供测试便利）。
- `--tables`：限定表（逗号分隔），默认全 `C2_USER_TABLES`。
- `--json`：机器可读。

### D2：purge_soft_deleted 加 `dry_run` 参数

`dry_run=True` 走 `SELECT COUNT(*)`（per-table 符合计数，不删）；`dry_run=False` 走 `DELETE`。加性可选参数，向后兼容（现有测试不改）。让 dry-run 成为一等公民（可测）而非 CLI 内重复 WHERE 逻辑。

### D3：注释诚实化（消灭假绿叙事）

- `sovereignty.py:197`：删「nightly cron calls this」假叙事 → 明示「NO nightly cron wired today；sole 生产调用方是 `hermes purge` CLI；future scheduler may call this」。
- `sovereignty.py:27`（模块 docstring）：收紧——purge primitive **已可达**（via CLI），defer 的是 *automatic* scheduler，不是 purge 本身。
- `web_server.py:12356`：「is a separate opt-in nightly cron」→ 「opt-in via `hermes purge` CLI；NOT a UI one-click and NOT (yet) an automatic nightly cron — no scheduler is wired today」。

### D4：C4 测试钉死

- `test_adr_v6_067_purge_reachable.py`：dry_run 计数准确 / `--confirm` 真删 / 默认不删 / CLI 渲染 / `hermes purge` 在 `_BUILTIN_SUBCOMMANDS`（可达性契约）。
- `test_realityos_sovereignty.py`：补 dry_run=True 不删一行（向后兼容回归）。

## 不做（诚实边界）

- **不**接自动 nightly cron scheduler——这是 §6.2 阶段2 的 desktop-UI/ops 层工作（模块 docstring 已标 deferred）。本 ADR 只把 purge 从「声称自动实为零调用」诚实化为「手动可达 + 自动 deferred」。
- **不**把 purge 接成 UI one-click——硬删不应一键触发；CLI + `--confirm` 是正确的创始人-admin 面。
- **不**改 purge 的 C2 合规性（仅删 deleted_at 已设 + 宽限窗口已过的行）。

## 反假绿

| 项 | 判定 | 依据 |
| --- | --- | --- |
| 注释是否曾谎称 nightly cron | ✅ | sovereignty.py:197 + web_server.py:12356 原文 + grep 零调度 |
| purge 是否曾零生产调用方 | ✅ | grep 仅测试调用（ADR-037 做了没发） |
