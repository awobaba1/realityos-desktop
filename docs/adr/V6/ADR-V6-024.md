# ADR-V6-024：捕获面默认开启 — ptg_capture observer 默认启用（反假绿桶 C1）

- **状态**：Accepted（批次 1，2026-07-19）
- **日期**：2026-07-19
- **Phase**：1b（反假绿收口）
- **关联**：ADR-V6-022（母纲，镜头 4 揭露）、ADR-V6-008（ptg_capture observer 半部设计）、ADR-V6-011（Atomizer 捕获面契约）
- **铁律**：C1（本 ADR）、C3（生产现实即真理——审计暴露「捕获面生产真空」）、C7（捕获失败 fail-open，本 ADR 不改 fail-open 语义，仅修启用）

---

## 背景

ADR-V6-022 十镜头审计镜头 4 揭露一条头号假绿：

**V6 招牌的「操作电脑捕获面」（post_tool_call → tool_events）在生产上是死的。**

证据链：

1. 捕获钩子（`post_tool_call` / `pre_gateway_dispatch` / `on_session_end`）真实存在于 bundled 插件 `plugins/observability/ptg_capture/`（ADR-V6-008 decision 2：observer 半部不能住 `memory/ptg/`，因 memory-plugin loader 的 `register_hook` 是 no-op）。
2. `hermes_cli/plugins.py` 的插件门控是 **opt-in**：`is_enabled = enabled is not None and (lookup_key in enabled or manifest.name in enabled)`（`plugins.py:1454`）。`enabled` 来自 config.yaml 的 `plugins.enabled` 白名单。
3. config v20→v21 迁移（`config.py:6001-6064`）只 grandfather `$HERMES_HOME/plugins/` **用户装**插件，**显式排除 bundled**（`config.py:6008-6010` 明文："Bundled plugins … ship off for everyone"）。
4. `DEFAULT_CONFIG` 原本**没有 `plugins` 段** —— fresh 安装 `plugins.enabled` 缺失 → `_get_enabled_plugins()` 返回 `None` → opt-in 默认 = 什么都不加载。

**结论**：在**每一个**安装（fresh + 升级）上，`ptg_capture` 都不在 `plugins.enabled` → 三个 observer 钩子永不注册 → R6/R12/R4（工具执行类原子）永不捕获 → 「操作电脑捕获面」是纸老虎。schema/store 里 `extracted_via='post_tool_call'` 的列、capture_schemas 的契约，全是死代码。

## 决策

**config 为唯一真相源**（不改 loader 语义，更低风险）。两处修复覆盖 fresh + existing 全部安装状态：

### 1. `DEFAULT_CONFIG` 播种（fresh 安装）

`hermes_cli/config.py` 的 `DEFAULT_CONFIG` 新增：

```python
"plugins": {
    "enabled": ["observability/ptg_capture"],
    "disabled": [],
},
"_config_version": 34,   # 33 → 34
```

fresh 安装无 config.yaml → `load_config()` 深合并 DEFAULT → `plugins.enabled=[observability/ptg_capture]` → loader 门控通过 → 钩子注册。

### 2. v33→v34 迁移（existing 安装）

新增迁移块（`config.py`，v32→v33 块之后）：

- 读 `plugins` 段（缺则建空 dict）。
- **显式 disable 优先**：若 `observability/ptg_capture` 或裸叶名 `ptg_capture` 在 `plugins.disabled`，**跳过**（用户意图赢）。
- 否则若不在 `enabled`，追加 canonical key（path 派生 `observability/ptg_capture`，匹配 `hermes plugins enable observability/ptg_capture` 写入的形式）。
- 幂等：已存在则不重复加。

为什么 existing 也需要迁移（DEFAULT 播种不够）：v21 迁移给 existing 安装写入了**显式** `plugins.enabled`（用户插件列表）。`load_config()` 深合并对**已存在的 list 键**是 user-wins —— 一个不含 ptg_capture 的用户列表会盖掉 DEFAULT 的播种。迁移显式追加，且因结果与 DEFAULT 不同（含用户额外插件）而**不会被 `save_config` 默认剥离**持久化。

### 为什么不改 loader

考虑过在 `plugins.py` 门控加 `V6_CORE_OBSERVER_PLUGINS` 白名单默认启用，但放弃：

- `hermes plugins list` 读的是 **config**（`_get_enabled_set()` → `load_config()`），**非** loader 状态。loader 默认启用而 config 不列 → 「实际加载但 list 显示 not enabled」= **二次假绿**，正是本 ADR 要消灭的。
- config 为唯一真相源：迁移 + 播种让 loader、list、config.yaml 三处一致诚实。
- 显式 disable 优先已在 loader 门控上游（`plugins.py:1388-1395`，先于 enabled 检查）保证，无需重复实现。

## 验证（C3 生产现实即真理）

`tests/hermes_cli/test_config_v34_ptg_capture_migration.py`（10 测试全绿）覆盖：

| 场景 | 期望 | 状态 |
|---|---|---|
| fresh 安装（无 config.yaml） | `load_config` 含 ptg_capture | ✓ |
| DEFAULT 版本号 | `_config_version==34` | ✓ |
| DEFAULT 播种 | enabled 含 canonical key | ✓ |
| v33 空 enabled 列表（最常见） | 迁移后启用 | ✓ |
| v33 含用户插件列表 | 追加且保留用户插件 | ✓ |
| v33 无 plugins 段 | 启用（DEFAULT 合并） | ✓ |
| v33 disabled（canonical key） | **不**追加到 enabled | ✓ |
| v33 disabled（裸叶名） | **不**追加到 enabled | ✓ |
| 迁移幂等（跑两次） | 单条目 | ✓ |
| 已 v34 且 key 在 | 不动 | ✓ |

**承重门端到端核验**（直接调 loader 用的 `_get_enabled_plugins()`）：fresh 安装返回 `{'observability/ptg_capture'}`（非 `None`）→ loader `is_enabled=True` → ptg_capture **默认加载** → 钩子注册。这是最终反假绿铁证。

回归：`ruff check .` 全量净；config 既有 155 测试 + plugin 129 测试零破坏。

## 后果

- **捕获面活了**：fresh + existing 全部安装，post_tool_call/on_session_end 钩子默认注册，工具执行 → tool_events → R6/R12/R4 原子捕获链路闭环。
- **可关**：用户 `hermes plugins disable observability/ptg_capture` 即关（显式 disable 优先）。
- **诚实**：`hermes plugins list` 如实显示 enabled；config.yaml 自文档化。
- **零 loader 语义改动**：plugin 加载契约不变，降低回归面。

## 延迟项

- pre_gateway_dispatch 出站消息捕获（§D4 deferred to v2，PTGProvider.sync_turn 已覆盖出站回合）—— 本 ADR 不触及，保持现状。
- on_session_end 语义提取（batch extraction hook point）—— 随提取 phase 到位，本 ADR 只保证钩子注册。
