# RealityOS V6 个人信息保护影响评估（DPIA）

> 架构设计文档 §6.8 + §8 Phase 0 明确要求：「**DPIA 先行（采集前完成，PIPL §55）**」。
> 本评估在 V6 数据采集启动前完成（Phase 0），评估对象是 RealityOS V6 桌面客户端的全部
> 个人信息处理活动。V6 是单租户、自部署、数据物理不出设备的个人数据资产引擎。
>
> **依据**：《中华人民共和国个人信息保护法》(PIPL) §55（应当事前作 DPIA 的情形）+ §56
> （DPIA 内容）+ §28（敏感个人信息）+ §31（未成年人）+ §38（跨境提供）。
>
> **诚实原则**：控制措施按现状标注（✅ 已实现 / 🟡 规划或骨架在途 / ❌ 未开始）。DPIA 不
> 做粉饰——未闭环的风险明确列出，不靠「计划」充数。

---

## 0. 评估结论（先给结论）

**V6 的个人信息处理，在以下三项控制措施全部落地后，风险可接受（residual risk = acceptable）**：

1. ✅ 数据物理不出设备（单机 SQLite，已完成）
2. 🟡 `realityos_security` net-policy fetch-guard 硬阻断唯一出网口（**P1-2 在途，当前未闭环**）
3. 🟡 `realityos_sovereignty` 两模式删除 + 一键导出 + 未成年人模式（**P1-3 在途，当前未闭环**）

**残留高风险**（直到闭环前不得对外公测）：
- **R-EGRESS**：LLM API 出境是 V6 唯一的数据处理出网口，当前仅靠「LLM provider 调用集中」软约束，
  尚无 net-policy 硬阻断兜底。一旦 prompt injection 诱导 `web_fetch` 类工具外发，无网络层防御。
- **R-MINOR**：未成年人保护为骨架，未成年用户当前会按成人采集 L3（声纹/情绪）。
- **R-DELETE**：删除权（PIPL §47）当前只有软删，模式 A/B 物理删未实现，数据宪法行使面缺失。

**放行建议**：
- 内部测试（创始人 + 受邀白名单，知情同意）→ 现状即可放行。
- 对外公测/发布 → **必须**先闭环 P1-2（R-EGRESS）+ P1-3（R-DELETE，至少模式 B）。R-MINOR 在
  确认无未成年用户前可延后，但产品文案须标注「不面向 14 岁以下」。

## 1. 处理活动描述（PIPL §56-1）

### 1.1 处理目的

RealityOS 把用户在日常对话/语音中「流经 agent」的内容，清洗 → 原子化 → 结构化成「人生图谱
(PTG)」→ 组合成洞察/提醒/问答服务（架构设计文档 §1 北极星）。**唯一目的是为用户提供「越聊越懂
你、帮你记住」的本地数据资产服务**，非广告、非画像售卖、非二次利用。

### 1.2 处理方式

- **捕获**：用户主动发来的对话（流经即捕获，§0.6）。**不监控**屏幕/剪贴板/其他 App；computer-use
  读屏图像不沉淀进 PTG（§0.6/§5.5）。
- **原子化**：HL-12 原子抽取（Phase 1：R0/R1/R2/R3/R7），LLM 在线抽取 → 经 C5 schema 校验 → 落 PTG。
- **聚合**：每原子带 `consent_tag`（默认 `local_only=true`）+ `deleted_at`（C2 软删）。
- **存储**：单一本地 SQLite（`<HERMES_HOME>/ptg.db`），14 表，数据宪法 §6.1 永久资产默认。
- **出境**：仅 LLM API 调用（抽取/RAG/洞察生成时上送文本片段）。

### 1.3 处理种类（个人信息的类别）

详见 `data-classification.md`。摘要：

- **L3 敏感个人信息**（PIPL §28）：声纹（`entities.voiceprint_samples`、`memos.audio_clip_id`）、
  位置（`memos.location_context` GPS 行踪）、健康/心理状态（`feeling_events` 全字段）、
  自由文本（`memos.source_text`/`meaning_events.task_description`/`llm_call_logs.prompt_input` 等按
  最坏内容判 L3）、未成年人信息（触发后）。
- **L2 一般个人信息**：姓名（本人及第三方）、邮箱、关系链（`relations`）等。
- **L1 凭证**：`password_hash`。
- **L0 派生/审计**：质量指标、计数、schema_version、时间戳。

### 1.4 必要性（PIPL §6 最小必要）

- 捕获范围**严格限定**为「用户主动、流经 agent 的执行」（§0.6 数据边界），是底线必要性。
- L3 字段中：声纹是 Phase 2.5 SER 的唯一源（§6.3 永久保留理由），位置是 PRD 明确的 V6 唯一
  多模态（§不做清单），情绪是 R1 核心原子——均直接服务北极星，无冗余采集。
- **不做**（§不做清单）：屏幕/充电/活动/蓝牙/App 使用、通知监听、通讯录、日历、社交、相机、
  除 GPS 外多模态。这些是必要性最直接的证据——竞品会采的，V6 明确不采。

### 1.5 合法性基础（PIPL §13）

- 主线：§13(1) 取得同意（安装勾选 + `consent_tag` 逐 atom）+ §13(2) 履行服务必需。
- 日志：§13(6) 工程必需（C6 可重放 + §6.9 灾备）。
- 单租户自部署形态强化了 §13(1) 的「充分告知 + 自愿同意」——用户即数据控制者本身。

## 2. 触发 DPIA 的 §55 情形

| §55 情形 | 是否触发 | V6 对应 |
|---|---|---|
| §55-1 处理敏感个人信息 | **是** | 声纹/位置/情绪/L3 自由文本 |
| §55-2 自动化决策有重大影响 | **部分** | V 域任务建议、洞察、每周镜面——Phase 2 才强；Phase 1 为建议非决定（§5 详评） |
| §55-3 委托/提供/公开给第三方 | **是** | LLM provider（DeepSeek/智谱）为受托处理者 |
| §55-4 跨境提供 | **是** | LLM API 出境（§6.6 唯一出网口） |
| §55-5 其他高风险 | 是 | 单点数据资产（硬盘坏即全没，§6.9） |

**结论：四项触发，DPIA 强制必需。** 这正是 §6.8 把 DPIA 列为 Phase 0 的原因。

## 3. 对个人权益的影响与安全风险（PIPL §56-2）

### 3.1 风险登记表

| ID | 风险 | 影响 | 可能性 | 现状控制 | 残留等级 |
|---|---|---|---|---|---|
| **R-EGRESS** | prompt injection 诱导工具外发用户文本 | L3 泄露 | 中 | 🟡 net-policy 未闭环，当前软约束 | **高（未闭环）** |
| **R-LLM-EXFIL** | LLM provider 收到完整 prompt（含敏感文本）并留存 | L3 跨境留存 | 高（必然） | 🟡 provider 数据政策依赖 + §6.6 Phase 4 端侧迁移为终极解 | **中-高** |
| **R-DEVICE-LOSS** | 单机数据，硬盘坏/失窃即全没 | 全部资产丢失 | 中 | ✅ §6.9 备份 + 月度演练（P0-2 已补调度） | **低（已闭环）** |
| **R-DELETE** | 删除权只有软删，模式 A/B 物理删未实现 | 删除权未满足（PIPL §47） | — | 🟡 P1-3 sovereignty 在途 | **中（未闭环）** |
| **R-MINOR** | 未成年人按成人采集 L3（声纹/情绪） | 未成年人 §31 违规 | 低（确认无未成年用户时） | 🟡 P1-3 年龄门骨架在途 | **中（条件性）** |
| **R-VOICEPRINT** | 声纹生物识别泄露不可撤销 | 不可逆身份泄露 | 低 | 🟡 本地存储 + consent_tag，未成年人降级 | **中** |
| **R-CASCADE** | 关系链泄露牵连第三方（被提及的他人） | 第三方 L2 泄露 | 中 | ✅ 本地存储 + soft_delete | **低-中** |
| **R-SILENT-FAIL** | 备份假绿/捕获静默丢数据 | 数据资产失真 | 中 | ✅ C7 verify_backup 反假绿契约（test_ptg_backup 锁定） | **低（已闭环）** |

### 3.2 最坏情形（worst-case）专评

**R-LLM-EXFIL 是 V6 最深层张力**（§0.6 预告、§6.6 主权缝隙）：用户文本必须过 LLM 云才能抽取原子，
这是「数据不出设备」不变式的**唯一例外**，且无法在 Phase 0/1 完全归零。架构设计文档已诚实预告
（§0.6「LLM 瞥过敏感屏不可完全归零，非 blocker」）并给解法：
- 短期：provider 数据政策审查（受托处理者不得用于训练）+ prompt 最小化（只送上抽取所需片段）。
- 中期：§6.6 回滚条款（泄露或投诉>5% 启动端侧迁移）。
- 终极：Phase 4 端侧 LLM（数据完全不出设备）。

DPIA 立场：**当前阶段该风险接受（靠透明告知 + 用户即控制者），但必须写入隐私声明并由用户明示
同意出境（PIPL §39 单独同意）。**

## 4. 保护措施是否合法、有效、与风险相适应（PIPL §56-3）

### 4.1 已实现控制（✅）

| 控制 | 证据 | 适应风险 |
|---|---|---|
| C2 不可丢失（软删+version 全表） | `schema.py C2_USER_TABLES` + test | R-DELETE 的软删半步 |
| C5 schema 校验门（LLM 输出先验证再入库） | atomizer → DLQ 拒绝路径 | R 数据污染 |
| C6 可重放（llm_call_logs 全量） | schema + atomizer 记录 | 审计可追溯 |
| C7 无静默失败（DLQ + 反假绿 verify_backup） | `backup.py` + test | R-SILENT-FAIL |
| 本地存储不出设备 | 单机 SQLite | R-DEVICE-LOSS 的前提 |
| §6.9 灾备（备份+演练） | `backup.py run_scheduled_protection`（P0-2） | R-DEVICE-LOSS |
| 不做清单（不采屏幕/通讯录/多模态等） | §不做清单 + 代码无相关采集 | 最小必要（§6） |

### 4.2 在途控制（🟡，闭环前为残留高风险）

| 控制 | 计划位置 | 闭环条件 |
|---|---|---|
| net-policy fetch-guard 硬阻断出网 | `realityos_security`（P1-2） | R-EGRESS 降为低 |
| 两模式删除（A/B 物理删） | `realityos_sovereignty`（P1-3） | R-DELETE 闭环 |
| 一键导出 JSON（<5s） | `realityos_sovereignty`（P1-3） | 可携带权（PIPL §45）闭环 |
| consent_tag 行使面（UI 切换 local_only/shareable） | `realityos_sovereignty`（P1-3） | 同意管理闭环 |
| 未成年人年龄门 + 降级采集 | `realityos_sovereignty`（P1-3） | R-MINOR 闭环 |

### 4.3 未开始（❌）

- 端侧 LLM 迁移（Phase 4，R-LLM-EXFIL 终极解，路线图已排）。

### 4.4 适应比例性判定

控制强度与风险层级匹配：L3（声纹/位置/情绪/出境文本）配套了「本地存储 + 软删 + consent_tag +
C5 校验 + 灾备 + 反假绿」，覆盖了机密性/完整性/可用性三性。**唯一比例性缺口**是出境文本在
net-policy 闭环前缺硬防御（R-EGRESS），这正是放行建议里「对外发布前必先闭环 P1-2」的依据。

## 5. 自动化决策专项（PIPL §24，对应 §55-2）

V6 的自动化输出有三类：
1. **R2 任务建议**（`task_suggestions`）：从用户对话抽任务 → 到点提醒跟进。**影响**：仅提示，
   用户可 `dismissed`，不做强制决定 → **不构成 §24「重大影响」**。
2. **洞察/每周镜面**（`insight_aggregation`）：LLM 生成行为画像总结。**影响**：仅呈现给本人，
   不对外、不定价、不拒服务 → **不构成重大影响**。
3. **未来 V 域 Priority Engine**（Phase 2+）：任务优先级。当前未实现。

**结论**：V6 Phase 0/1 的自动化输出均为「对个人的自我呈现」，不对外产生法律/合同/权益重大影响，
**不触发 §24 强制人工干预/解释权要求**。Phase 2 若扩展到影响第三方的自动化决策，须重评。

## 6. 未成年人保护专项（PIPL §31，对应 §6.7）

- **现状**：年龄门 + 未成年人降级模式为 `realityos_sovereignty` 骨架（P1-3 在途）。
- **降级措施**（设计）：未成年人不抽 R9/R1 生物标记（声纹）、不入聚合画像、删除权一键默认、
  关键操作监护人确认。
- **过渡口径**：未成年模式闭环前，产品文案须标注「本产品不面向 14 岁以下未成年人」；内部测试
  白名单确认无未成年用户。

---

## 7. 跨境提供专项（PIPL §38/§39，对应 §55-4）

- **出境数据**：见 `data-classification.md` §5（LLM API 上送的文本片段）。
- **当前合法化路径**：依赖受托处理者（LLM provider）的数据政策 + 用户单独同意（PIPL §39）。
  严格意义上，V6 尚未完成 §38 要求的标准合同/认证/安全评估任一路径——**这是未完全闭环的合规
  张力，诚实标注**。
- **缓解**：§6.6 回滚条款（泄露或投诉>5% 启动端侧迁移）+ Phase 4 终极消除。
- **DPIA 立场**：单租户自部署 + 用户即控制者 + 数据政策审查 + 可回滚，风险**有条件接受**；
  公测/发布前必须在隐私声明中取得用户对 LLM 出境的单独明示同意。

---

## 8. 通报与响应（PIPL §57，对应 §6.10）

发生或可能发生个人信息泄露/篡改/丢失时：
1. 立即切断唯一出网口（LLM provider 暂停，靠 `realityos_security`，P1-2 闭环后可一键）。
2. 导出事件日志（`llm_call_logs` 可重放）。
3. 按 §6.6 回滚条款评估端侧迁移。
4. **72h 内通报用户**（PIPL §57；架构设计文档 §6.10 已固化）。
